# LangChain、LangGraph 与简单 Agent 搭建

## 🎭 面试官这样问

> 面试官：LangChain 和 LangGraph 有什么区别？你如果要搭一个简单的 Agent，用哪个？

> 你：LangChain 是做 Agent 的框架，定义好工具和 prompt，它能自动调用工具完成多步任务。LangGraph 是做复杂流程编排的，用图来描述节点和跳转。

> 面试官：那如果我做的 Agent 只有 3 步——查订单、判断状态、决定是否退款，这算简单还是复杂？用什么？

> 你：3 步应该算简单吧，用 LangChain 的 agent executor 应该就够了。

> 面试官：那如果第二步「判断状态」里要调两个接口，一个查支付状态一个查物流状态，两个都成功才能继续；如果支付接口超时了要重试，重试 2 次还不行就转人工——你还觉得 LangChain 够吗？

> 你：这……确实涉及条件分支和重试逻辑了，LangGraph 更好。

> 面试官：对，但不止这些。真正的区别不是「步数多还是少」，而是「流程控制权在谁手里」。LangChain 把决策权交给模型——模型读了工具返回，自己决定下一步；LangGraph 把流程控制权交给你的图结构——你显式定义了每个节点执行什么、什么条件下跳转到哪。一个靠模型推理，一个靠工程编码。能讲清楚这个区别，才是面试官想听的。

这就是今天要讲的核心：框架帮你串组件，但有状态编排和流程控制权，才是 Agent 工程化的关键分水岭。

## 💡 简要回答（面试话术）

LangChain 和 LangGraph 解决的不是一个问题。LangChain 更像一个 LLM 应用的「乐高积木箱」——它提供了模型封装、prompt 模板、retriever、tool、output parser 这些标准组件，让你快速把它们拼在一起。它的 agent executor 能跑一个基本循环：模型选工具 → 执行工具 → 看结果 → 继续选工具，适合流程相对确定、分支不复杂的场景。但它的核心设计是用模型做流程决策——模型根据历史输出和工具返回，自己判断下一步做什么。

LangGraph 解决的是另一个问题：当你需要显式控制流程时。比如「先查支付状态和物流状态，两个都成功才能继续，任何一个失败需要重试，重试 3 次后转人工，转人工后 30 分钟内自动恢复继续」。这种复杂的分支、重试、中断恢复逻辑，靠模型自己决策太不可靠了——模型可能少重试一次，可能在错误分支继续走。LangGraph 把流程建模成有向图：节点执行具体动作，边定义转移规则，共享 state 保存所有中间变量，条件边根据 state 的内容决定下一跳。

一句话总结：LangChain 解决「怎么把 LLM 能力串起来」，LangGraph 解决「复杂流程每一步谁来拍板」。

## 📝 详细解析

### 核心矛盾：组件拼装和状态控制不是一回事

你有没有遇到过这种情况：你写了一个 Agent，本地跑得挺好，但一上线就开始翻车——用户问的问题稍微复杂两轮，模型就开始乱调工具，调了一个不该调的接口，花了钱，还没完成任务。你把 prompt 改来改去，加了「注意不要重复调用」「失败就停止」，但还是一样。

问题的根在哪？你指望模型读懂你的 prompt 里的每一条规则，在多轮交互中严格遵守——但模型本质上是一个「下一个 token 预测器」，不是规则执行器。它有时候记得规则，有时候在上下文膨胀后忘了；有时候你写的「失败就停止」和「遇到问题先重试」冲突了，它就随便选一个。

这就是 Agent 工程化的核心矛盾：**流程的复杂度超过某个临界点后，你不能靠 prompt 来控制流程——你必须用图结构、状态机和工程编码来保证行为。** LangChain 的设计假设是「模型能做合理的流程决策」，LangGraph 的设计假设是「工程代码应该接管流程决策」。

就像你开公司：初期 3 个人，什么事都商量着来，很灵活；但到了 300 人，你必须写制度、画流程图、定审批节点——不是人不聪明，而是靠「口头商量」的复杂度已经失控了。

### 底层机制：一个简单 Agent 的最小闭环

不管用 LangChain 还是 LangGraph，Agent 的底层运行闭环是一样的：

```text
loop:
  ① 模型读取当前上下文（用户目标 + 历史步骤 + 工具返回）
  ② 模型输出决策：直接回答 OR 调用工具
  ③ 如果是工具调用：
     - 解析工具名和参数
     - 校验参数合法性
     - 执行工具，获取结果
     - 把结果放入上下文
     - 回到 ①
  ④ 如果是直接回答：
     - 输出最终答案
     - 结束循环
```

这个循环最关键的工程点不是「它能跑」，而是「它什么时候该停」和「什么时候该换路」。你用 LangChain 写这个循环时，停不停止基本靠模型自己判断（通过 agent executor 的 `AgentFinish` 信号）。你用 LangGraph 写时，停不停止由你的条件边判断——比如「state.retry_count > 2 → 转人工结束」或「state.all_tools_success → 生成答案」。

### LangChain 的价值：快速拼装，原型验证

LangChain 最大的价值是**加速从 0 到 1**。你想验证一个 Agent 想法——比如「用 LLM 查数据库，自动生成周报」——你不需要从头写模型调用的 boilerplate、不需要自己写 tool schema 的解析、不需要自己管理 prompt 拼接。LangChain 把常见能力封装好了：

- **ChatModel 封装**：统一 OpenAI / Claude / 本地模型的调用方式
- **Prompt 模板**：用变量动态填充 system prompt 和 user prompt
- **Retriever 接口**：统一向量检索、关键词检索的调用
- **Tool 定义**：用 `@tool` 装饰器或 Pydantic schema 声明工具
- **Agent Executor**：管理「思考→工具调用→观察→再思考」的循环
- **Memory**：对话历史管理，支持 buffer / summary / token 限制

对简单场景——标准 RAG 问答、单工具查询、3 步以内线性流程——LangChain 能让你半天搭出来、第二天就能 demo。但当需求升级到「工具失败重试」「条件分支」「人工确认」「中断恢复」时，LangChain 的抽象层反而变成阻碍——你搞不清是模型输出错了、parser 解析错了、retriever 返回错了、还是 executor 状态管理有问题。

### LangGraph 的价值：显式流程，工程可控

LangGraph 把 Agent 流程建模成有向图，核心概念就三个：

| 概念 | 作用 | 类比 |
|------|------|------|
| **Node（节点）** | 执行具体动作：调 LLM、调工具、做判断 | 流水线工位 |
| **Edge（边）** | 定义转移规则：无条件转移 或 条件转移 | 传送带/分拣口 |
| **State（共享状态）** | 持久化所有中间变量，每个节点读写 | 随工单流转的文件夹 |

![简单 Agent 执行循环](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/09-agent-frameworks/04-langchain-langgraph/02-运行机制.png)

这意味着你可以在图上做这些 LangChain 做不到的事：

- **分支和条件跳转**：`if state.confidence < 0.7 → 追加检索`、`if state.action == "delete" → 进入人工确认节点`
- **中断和恢复**：跑到人工确认节点时，图暂停，等人点「同意」，图从暂停点继续
- **重试逻辑**：某个工具调用失败，条件边判断 `retry_count < 3`，回到工具节点重试
- **并行执行**：多个独立的工具调用可以在不同节点并行跑，汇总到下一个节点
- **回溯和回滚**：发现前面步骤有错误，可以回到之前的节点重新执行

这些控制逻辑如果靠 prompt 让模型实现，模型可能在 80% 的情况下做对，但剩下 20% 就是线上事故。LangGraph 的价值就是把这 20% 的不确定性消除。

### 工程例子：IT 助手从「问答工具」到「工单系统」

假设你在做一个 IT 助手 Agent，帮员工解决电脑问题。

**原型阶段（LangChain）**：用户说「VPN 连不上」，Agent 从知识库检索 VPN 排障文档，生成排查步骤。用户说「试了不行」，Agent 调用网络诊断工具收集更多信息再回答。这阶段 LangChain 足够——流程是线性的，没有复杂分支，模型自己能搞定。

**上线后需求升级**：产品经理说——
1. 不同操作系统走不同诊断路径（Windows 查防火墙，Mac 查系统代理）
2. 高权限操作（重启服务、修改注册表）必须先弹确认框
3. 诊断工具调用失败，自动重试 2 次，还不行就创建工单
4. 创建工单要填分类、优先级、影响范围，缺信息要追问
5. 用户长时间不回复，自动把上下文存为草稿，下次继续

这时候 LangChain 就撑不住了——你的 prompt 得像法律条文一样长，模型在 10 轮对话后大概率会漏规则。切换到 LangGraph 后，流程变成这样：

```text
[用户输入] → [识别操作系统] 
    ├─ Windows → [网络诊断节点] 
    │              ├─ 成功 → [生成排障建议]
    │              └─ 失败 → retry_count < 2? → 重试
    │                       └─ retry_count >= 2 → [创建工单节点]
    │                           ├─ 信息不完整 → [追问节点] → 回到等待用户输入
    │                           └─ 信息完整 → [确认节点] → 等待人工确认
    └─ Mac     → [系统代理诊断节点] → (同上)
```

这个图里每一步的跳转条件都是你写的 Python 代码，不是模型推理出来的。好处是：
- 任何情况下行为可预测
- 出问题时你能定位到具体节点
- 你能单独对每个节点写单元测试
- 审计日志能精确记录每步决策

### 框架和 Harness 的边界：能跑通 ≠ 能上线

这里有一个很多候选人会忽略的关键区分：LangChain 和 LangGraph 是**编排框架**——它们管的是「任务怎么做、步骤怎么走、状态怎么传」。但它们不负责**执行安全**——工具调用的权限校验、沙箱隔离、超时控制、审计日志，这些是 Harness 层的事。

![框架选型边界](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/09-agent-frameworks/04-langchain-langgraph/03-面试答题框架.png)

打个比方：框架是装修公司的项目经理，他规划「先砸墙、再走水电、再铺地板」，但他不检查工人有没有资质、有没有戴安全帽、砸的墙是不是承重墙——这些是 Harness 的事。

面试里如果说「LangGraph 搭的 Agent 可以直接上线」，有经验的面试官一定会追问：权限怎么控制？敏感操作用户确认了吗？重试逻辑会不会导致雪崩？能回答出框架和 Harness 的边界，说明你真正在生产环境踩过坑。

### 边界和风险：不是所有场景都需要图

LangGraph 虽好，但不是每个 Agent 都需要它。有几个判断标准：

1. **流程确定性**：如果你的 Agent 步骤固定（查→判断→回答），用 LangChain 简单可控，上 LangGraph 反而增加了图定义的维护成本。

2. **团队能力**：LangGraph 的图定义、状态管理、条件边都需要一定的工程能力。小团队做原型验证，先用 LangChain 快速跑通，确认方向后再重构为 LangGraph 是常见路径。

3. **调试成本**：LangGraph 的图虽然可测，但一旦节点多、边复杂，图本身的正确性验证就是一项工程任务。你需要确认每种可能的路径都被测试覆盖了。

4. **性能开销**：LangGraph 的 state 在每次节点间传递时会做深拷贝（保证不可变性），对大 state 会带来内存和延迟开销。

## 🎯 面试总结

- **必答 3 点**：
  1. LangChain 是 LLM 应用组件库 + 简单 Agent Executor，流程决策主要靠模型推理；LangGraph 是有状态图编排框架，流程决策由你写的条件边和节点逻辑控制。
  2. 简单 Agent 的最小闭环是「模型决策 → 工具执行 → 结果回传 → 继续判断」，关键是最大步数、异常处理和停止条件。
  3. 流程确定性低、步骤少、分支简单用 LangChain；分支多、有重试、有人工介入、需要中断恢复用 LangGraph。

- **加分项**：能讲清楚「流程控制权」的转移——LangChain 是把控制权交给模型（模型选下一步），LangGraph 是把控制权交给你（你写代码选下一步）。还能区分「框架管编排」和「Harness 管安全」的边界。

- **一句话总结**：LangChain 让你快速拼出能跑的 Agent，LangGraph 让你拼出能上线的 Agent——分界线不是代码行数，是流程不确定性谁在兜底。

## 🔍 排查和实践建议

排查 Agent 问题时，LangChain 和 LangGraph 的定位不一样。LangChain 出问题，先看 agent executor 的 trace——模型选了什么工具、传了什么参数、工具返回了什么、模型下一步判断是什么。LangGraph 出问题，看节点级日志——哪个节点执行了、state 在这个节点的输入输出是什么、条件边判断结果是什么。

实践建议：原型阶段用 LangChain 快速验证想法，确定方向后用 LangGraph 重构。重构时先画图——把所有可能的分支、异常路径、恢复策略在纸上画清楚，再写代码。图越复杂，越不应该让模型做流程决策。
