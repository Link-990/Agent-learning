# Encoder-Only 和 Decoder-Only 模型区别

## 🎭 面试官这样问

> 面试官：BERT 和 GPT 有什么区别？

> 你：BERT 是 Encoder-Only 做理解的，GPT 是 Decoder-Only 做生成的。BERT 用双向注意力，GPT 用单向注意力。

> 面试官：这就完了？那我问你，BERT 能不能做生成？GPT 能不能做分类？什么叫"能做"什么叫"擅长"？

> 你：BERT 理论上可以通过 MLM 预测 token，但它不是为从左到右生成长文本设计的。GPT 也可以做分类，把最后 token 的 hidden state 接一个分类头就行……

> 面试官：对。那如果公司现在要做 RAG，检索阶段用什么？生成阶段用什么？为什么不能反过来？

> 你：检索通常用 Encoder 类 embedding 模型，因为需要把 query 和文档编码成向量算相似度。生成用 Decoder-Only，因为它天然适合根据前文续写答案。

> 面试官：好。那我再挖深一点——为什么现在通用大模型几乎全是 Decoder-Only？BERT 那一套为什么不主流了？

> 你：嗯……因为 next token prediction 的训练目标和推理目标完全一致？而且 Decoder-Only 堆大了之后理解和生成都能做？

> 面试官：方向对，但还有一个关键点——Decoder-Only 可以通过 prompt 工程做任意任务，而 BERT 每换一个任务就要换一个任务头，灵活性差很多。你再想想，Encoder-Decoder 的 T5 为什么也没成为主流？

【从"BERT 理解、GPT 生成"这种简单二元论，深入到注意力范围、训练目标、工程成本和任务灵活性四个维度的完整对比。】

## 💡 简要回答（面试话术）

三类结构的差异不是"能不能做"，而是"天然适合什么"——这取决于两个核心变量：**注意力能看谁**和**训练目标是什么**。

Encoder-Only（BERT、RoBERTa）使用双向注意力，每个 token 可以同时看左右两侧上下文，配合 Masked Language Modeling 训练，天然适合理解类任务——分类、匹配、抽取、向量表示。就像一个侦探可以翻遍整本案卷再下判断。

Decoder-Only（GPT、LLaMA、Qwen）使用因果注意力，只能看已出现的 token，通过 next token prediction 训练。训练和推理目标完全一致，天然适合生成类任务——对话、续写、代码补全、工具调用。就像一个作家，只能根据前面写过的内容写下一句，不能偷看后面的草稿。

Encoder-Decoder（T5、BART）是两者的组合：Encoder 先双向编码输入，Decoder 再基于 Encoder 输出和已生成前缀逐 token 生成。适合翻译、摘要、改写这类"输入到输出"的转换任务。

面试时一定要补一句：**现在的强 Decoder-Only 模型通过大规模预训练和指令微调，理解和分类能力也很强，只是它的底层生成方式仍是自回归。**

## 📝 详细解析

### 核心矛盾：完整理解 vs 自回归生成

这两类任务对"看到多少信息"的要求截然相反。

**理解类任务**天然要求看到全局。判断"这家店服务不好吗？"和"这家店服务不好"的区别，差一个字加一个问号，左右上下文同等重要。做实体识别时，"苹果发布了新手机"——"苹果"的前面没词、后面是动词，双向上下文才能锁定它是公司而非水果。**这就像医生看化验单，不能只从第一行往下读，要同时看所有指标才能下诊断。**

**生成类任务**必须保证因果性。"今天天气真___"——填"好"还是"差"？模型只能基于前面的"今天天气真"来判断，不能偷看后面的内容。如果训练时让模型看到后面写的"好"，那它根本不需要学语言规律，直接抄答案就行。但推理时没有"后面"可看，训练推理不一致，性能直接崩盘。**这就像考试，平时练习时翻书抄答案，真正考试时书被收走了，你一个字都写不出来。**

### 底层机制：三类结构怎么工作

用一张表把核心差异钉死：

| 维度 | Encoder-Only | Decoder-Only | Encoder-Decoder |
|---|---|---|---|
| 注意力范围 | 双向，看完整输入 | 因果 mask，只看历史 | Encoder 双向，Decoder 因果 |
| 训练目标 | MLM（遮词预测）、分类微调 | Next Token Prediction | Seq2Seq（如 denoising） |
| 代表模型 | BERT、RoBERTa、DeBERTa | GPT、LLaMA、Qwen、DeepSeek | T5、BART |
| 擅长任务 | 分类、匹配、抽取、向量表示 | 对话、续写、代码、工具调用 | 翻译、摘要、改写 |
| 推理特点 | 一次前向出结果 | 逐 token 自回归 | 先编码再逐 token 解码 |
| 灵活度 | 每个任务需要特定任务头 | 通过 prompt 统一做任意任务 | 介于两者之间 |

**Encoder-Only 的机理**：BERT 每一层都是双向自注意力。预训练时随机遮住 15% 的 token，让模型根据左右上下文预测被遮住的内容——这强迫模型学会深层次的语言理解。微调时在 `[CLS]` token 的输出上加一个分类器，就可以做情感分析、文本匹配等任务。问题是每换一个任务就得换一个任务头，不够统一。

**Decoder-Only 的机理**：GPT 每一层都是因果自注意力。训练时给一段文本，让模型预测每一个位置的下一个 token——"今天天气真"→"好"，"今天天气真好"→"，"，"今天天气真好，"→"适合"……训练形式天然就是生成形式，不需要微调就能续写。后来通过 instruction tuning 和 RLHF，把各种任务都"翻译"成续写任务，实现了统一接口。

**Encoder-Decoder 的机理**：Encoder 读完整个输入，生成一个上下文表示；Decoder 在生成每一步时，既看 Encoder 的输出（cross-attention），又看自己前面生成的 token（self-attention with causal mask）。跨注意力是它独有的，让生成的每一步都能回头参考完整的输入编码。适合翻译这类"输入和输出不是简单续写关系"的任务。

### 工程例子：为什么 RAG 几乎都是"Encoder 检索 + Decoder 生成"

这是面试高频题，我见过太多人只会说"因为 BERT 理解好、GPT 生成好"，但说不出工程上的真实原因。

**检索阶段**的核心需求是：给你一个 query 和几百万个文档片段，快速找出最相关的 Top-K。这本质上是"语义匹配"——把 query 和文档都编码成向量，算余弦相似度。Encoder 类 embedding 模型擅长这个：双向注意力让每个 token 都能利用完整上下文，编码质量高；一次前向就能得到向量，延迟低。而且 embedding 可以离线预计算存到向量库里，检索时只算 query 向量。

如果你硬要用 Decoder-Only 做检索：每对比一个文档都得跑一次自回归生成，几百万个文档你要跑到猴年马月去。就算只取最后一个 hidden state 做向量，因果注意力的编码质量通常也不如双向注意力——因为每个 token 只能看左边，信息捕获不完整。

**生成阶段**的核心需求是：根据检索到的资料，用自然语言回答问题，可能还要引用来源、给出限制条件。这是 Decoder-Only 的主场：next token prediction 的训练目标让它天然能"根据前面的内容，生成后面的内容"。把检索片段、用户问题、系统指令拼在 prompt 里，模型逐 token 生成答案——整个过程就是一次续写。

**实战案例**：用户问"耳机买了 10 天还能退吗？"

- 检索模型（Encoder）：找到政策片段"15 天无理由退货 + 包装完好 + 配件齐全"
- 生成模型（Decoder）：输出"您在 15 天时效内，但仍需满足包装完好、配件齐全等条件……"

如果反过来：让 Decoder 直接扫完整知识库——肉眼可见地慢且贵。让 Encoder 输出答案——它只能给个标签或分数，没法组织成自然语言。

### 边界和风险：不要用"能不能"一棒子打死

**边界一：Decoder-Only 不是不能理解。** 经过万亿 token 的预训练和高质量的指令微调后，强 Decoder 模型的分类、抽取、推理能力经常超过同期的 Encoder 模型。它不是"不能"，而是"贵"——做一次情感分类也要走完完整的自回归流程，延迟和成本都高于 Encoder 的一次前向。

**边界二：Encoder-Only 不是不能生成。** BERT 可以通过 MLM 预测遮住的 token，也可以通过迭代式掩码生成做文本纠错。但它不是为开放式长文本从左到右生成设计的，强行自回归生成需要复杂改造且效果一般。

**边界三：为什么 Decoder-Only 成为主流？** 三个原因：（1）next token prediction 的训练方式可以利用互联网上所有文本，不需要标注；（2）通过 prompt，所有任务都统一为"根据前文续写"，一个模型、一套接口解决一切；（3）scaling law 表明堆数据和参数时，Decoder-Only 的能力上限高。相比之下，BERT 每个下游任务需要专门微调，工程上碎片化严重。

**边界四：attention mask 是红线。** 如果在原本该用双向注意力的任务上硬套 causal mask，模型看不到后文，关键信息丢失。如果在生成训练时漏掉 causal mask，模型偷看答案，训练 loss 好看但推理时一塌糊涂。这不是调参问题，是结构级别的错误。

## 🎯 面试总结

- **必答 3 点**：① 双向 vs 因果注意力：Encoder 能看全局上下文，Decoder 只能看历史 token；② 训练目标：MLM（遮词预测）vs Next Token Prediction，决定了各自天然适合理解还是生成；③ 推理方式：Encoder 一次前向，Decoder 逐 token 自回归，Encoder-Decoder 先编码再逐 token 解码。
- **加分项**：能讲清楚 RAG 工程中"Encoder 检索 + Decoder 生成"的选型理由（不只是"一个理解一个生成"，而是延迟、成本、编码质量的综合权衡）；能解释 Decoder-Only 成为主流的原因（统一接口、scaling law、数据效率）；能说清"能做"和"擅长"的区别。
- **一句话总结**：**BERT 和 GPT 的核心分歧不在模型名字，而在于"能不能看未来"和"训练的答案是什么"——一个为全局判断而生，一个为逐字续写而来，工程选型看的是你的输出是标签还是文本。**

## 🔍 排查和实践建议

**模型选型决策树**：
- 输出是**标签、分数、向量、实体边界** → Encoder-Only 或 embedding 模型（延迟低、成本低、编码质量高）
- 输出是**自然语言、代码、JSON 参数、多轮对话** → Decoder-Only（生成能力强、接口统一）
- 输出是**输入到输出的转换**（翻译、摘要）→ Encoder-Decoder 或强 Decoder-Only 均可

**线上排查分层**：
- 分类/匹配效果差：检查标注质量、类别定义、阈值、样本分布、embedding 模型版本
- 生成效果差：检查 prompt 是否完整、上下文是否截断、采样参数（temperature/top-p）是否合适、停止条件是否误触发
- RAG 效果差：拆成四层排查——召回（检索模型是否匹配）、重排（reranker 是否排序正确）、上下文（拼接是否合理）、生成（模型是否充分利用了检索资料）

**面试话术加分**：当你被问到"为什么选 BERT 而不是 GPT"时，不要只说"一个理解一个生成"，而是落到具体指标上说——"这个场景要求 99 分位延迟低于 50ms，GPT 的自回归方式达不到，而 BERT 的一次前向在 10ms 以内"。把架构差异落实到性能数据上，比背定义有力十倍。
