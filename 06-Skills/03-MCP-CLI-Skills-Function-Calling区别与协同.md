# MCP、CLI、Skills 和 Function Calling 的区别与协同

## 🎭 面试官这样问

**面试官**："你们做的 Agent 系统里，既有 Skill，又有 Function Calling，还用了 MCP，底层跑的是 CLI 命令——这些概念到底什么关系？它们能互相替代吗？"

**候选人**："它们解决不同层次的问题。Skill 告诉模型怎么做，Function Calling 让模型说它要调什么，MCP 把工具从哪来给标准化了，CLI 是最后落地执行的方式。"

**面试官**："好，那我追问几个问题。第一，用户报了个 bug，Agent 没修好，你怎么排查是 Skill 流程有问题，还是模型选了错误的 Function Calling，还是 MCP Server 挂了，还是 CLI 命令参数传错了？第二，你说四层各管各的，那权限控制应该放在哪一层？第三，如果不用 MCP，直接把工具用 Function Calling 暴露给模型行不行，会损失什么？"

**候选人**：（开始冒汗……）

**面试官**："别紧张，我问这些就是想知道你是真懂这些概念的工程边界，还是只背了名词。行，那你先讲讲它们分别长什么样吧。"

## 💡 简要回答（面试标准话术）

"这四者是 Agent 系统里不同抽象层的概念，分别解决不同工程问题。你可以把 Agent 系统想象成'一个聪明的实习生操作电脑'：**Skill 是操作手册**，告诉实习生这个任务按什么步骤做；**Function Calling 是实习生的嘴**，他用标准句式说'我要读这个文件'；**MCP 是公司 IT 系统**，统一管理所有他能用的工具和资源从哪里接入；**CLI 是键盘鼠标**，是最后真正在系统上执行动作的方式。

它们不互相替代，而是串成一条链路：Skill 规定任务流程 → 模型通过 Function Calling 结构化表达调用意图 → MCP 统一发现和接入工具 → CLI 在受控环境中真正执行命令。排查问题时按层切分，权限控制要逐层收口，不能把安全责任全押在一层上。"

## 📝 详细解析

### 一、核心矛盾：同一条链路上有四种责任，混在一起就乱套

做 Agent 工程时，初学者最容易犯的错误就是把 Skill、Function Calling、MCP、CLI 统一叫"工具"。你以为只是名词不精确，但工程上会出大问题——出了问题不知道是任务方法错了、模型选错工具了、Server 连接失败了、还是命令参数有问题。排查时像无头苍蝇，四个方向各猜一遍。

我刚入行时就踩过这个坑。做了一个代码审查 Agent，给模型写了一段 prompt 说"你要先看 diff，发现有安全风险就运行相关测试，然后写报告"。结果模型有时候跳过安全扫描直接跑测试，有时候测试命令写错了参数，有时候整个工具调用失败但日志里看不出来是哪层的问题。后来我把四层拆开看才找到根因：不是模型笨，是我的 prompt 里把流程（Skill）、工具选择（Function Calling）、工具来源（直连 vs MCP）、命令执行（CLI）全搅在一起了。

**错误设计的典型症状**：让模型读一段自然语言操作手册，然后自由拼命令执行。这样四层全混在一起——没有结构化参数，没有能力发现机制，没有权限隔离，也没有可审计的流程。模型一旦幻觉，危害会从底层一路穿到顶层。

正确的做法是分层治理，每层只承担自己该承担的责任：

- **Skill（任务方法层）**：规定"这类任务应该怎么做"。它像公司的 SOP（标准操作流程），告诉模型：遇到代码审查任务时，先看 diff → 再识别安全风险 → 运行相关测试 → 最后输出结构化结论。它不一定提供具体工具，只规定任务流程、边界和输出格式。
- **Function Calling（模型接口层）**：规定"模型如何以结构化形式表达调用意图"。模型生成 `read_file`、`run_tests`、`search_docs` 等工具请求时，必须符合预定义的 JSON schema。它不负责工具从哪里来，也不负责真实执行，只管"把话说清楚"。
- **MCP（连接协议层）**：规定"工具能力从哪里来、如何接入"。Host 通过 MCP Client 发现和调用外部 Server 暴露的 tools、resources、prompts。它把工具能力从单个 Agent 中解耦出来，让多个 Host 可以复用同一套工具，并集中治理权限和错误。
- **CLI（执行层）**：规定"最后在操作系统环境里怎么执行"。它是 `git diff`、`npm test`、`python script.py`、`kubectl apply` 这些具体命令。CLI 能力最强但也最危险——受环境变量、文件权限、路径、依赖版本、退出码的影响，最容易产生真实副作用。

这四层的关系，打个比方就像餐厅后厨：**Skill 是菜谱**（宫保鸡丁的标准做法），**Function Calling 是服务员下单**（"一份宫保鸡丁，少辣"），**MCP 是供应链系统**（食材从哪来、哪个供应商、怎么入库），**CLI 是厨师手里的锅和刀**（真正切菜炒菜）。你把菜谱、下单、供应链和炒菜混在一起管，出了食品安全问题你根本追溯不到是哪个环节。

### 二、底层机制：四层分别解决什么问题

我们一层一层拆开看，每层在工程上到底干了什么。

**Skill 层——把经验变成可加载的能力包**

Skill 的核心价值不是"多了一个 prompt"，而是把经过验证的任务执行模式固化下来。一个写好的代码审查 Skill，里面包含了：触发条件（什么时候该用这个 Skill）、执行步骤（先看什么再看什么）、工具使用规范（用哪个工具查 diff、用哪个跑测试）、输出格式（报告必须包含哪些字段）、禁止事项（不能改代码、不能自动提交）。这些内容如果每次都临时写在 prompt 里，不仅浪费 token，还容易遗漏——而且团队不同人写的 prompt 质量参差不齐。Skill 让你把这些经验版本化管理，团队共建，持续迭代。

**Function Calling 层——让模型"把话说清楚"**

没有 Function Calling 之前，模型想调工具只能输出一段自然语言，然后你用正则表达式去解析"它到底想调什么"。这极其脆弱——模型换个说法你的解析就崩了。Function Calling 把这个过程标准化了：你定义工具名称、描述、参数 schema（类型、必填项、默认值），模型训练时就知道要按照这个格式输出结构化 JSON。这样工具选择的准确率从"看运气"变成了"可度量"。

但要注意一个常见的误区：Function Calling 只是"表达层"，不是"执行层"。模型输出 `{"tool": "delete_file", "path": "/etc/passwd"}` 这件事本身只是生成了一段 JSON。要不要执行、能不能执行、执行前要不要确认——这些不是 Function Calling 的职责。

**MCP 层——把工具从"硬编码"变成"可插拔"**

在 MCP 出现之前，给 Agent 接工具的做法很原始：要么在代码里 hardcode 工具函数，要么每个 Agent 单独对接一套 API。问题很明显——你有 5 个 Agent 都要用 Git 工具，你得在 5 个地方写 5 遍。MCP 把工具提供方（Server）和工具消费方（Host/Client）解耦了。一个 Git MCP Server 写好，所有 Agent Host 都能通过标准协议调用它。而且 MCP 还带了能力发现（Server 告诉 Host 自己有哪些 tools）、资源管理（不只是工具，还能暴露文件、数据库记录等资源）和权限集中治理（在 Server 层面控制谁能调什么）。

**CLI 层——最后落地的那一下**

不管你上层多优雅，最后真正干活的是 `git diff`、`npm test`、`python train.py` 这些命令行。CLI 的特点是你几乎什么都能做，但也什么都能搞砸。环境不对命令跑不起来，权限不够写入失败，路径写错删了不该删的，网络超时整个流程卡住。CLI 是 Agent 能力的"最后一公里"，但也是安全风险的"最后一公里"。所以 CLI 需要被封装成受控执行单元——限定工作目录、限定可用命令集合、设置超时、捕获退出码和 stderr。

### 三、工程例子：审查 PR 并运行测试——四层如何串联

我们用一个完整例子看四层怎么协同。用户说："帮我审查这个 PR 并跑相关测试。"

**第一步：Skill 被触发。** 代码审查 Skill 的触发描述匹配了用户意图（"审查"+"PR"），系统加载 Skill 的主体内容。Skill 规定流程为：获取 PR diff → 分析变更范围 → 识别安全风险和兼容性问题 → 找出受影响的测试文件 → 运行相关测试 → 生成审查报告。

**第二步：模型通过 Function Calling 发起工具调用。** 模型读了 Skill 流程后，用结构化格式请求 `get_pr_diff(pr_id="123")`。它没有自己猜参数，因为 Skill 里写了"从用户消息中提取 PR ID，如未提供则先询问"。

**第三步：MCP 路由工具请求。** `get_pr_diff` 这个工具由 Git MCP Server 提供。Host 的 MCP Client 向 Git Server 发起标准协议调用，Server 收到请求后在内部执行受限的 `git diff origin/main...feature-branch`，把结果结构化返回。

**第四步：CLI 在最底层执行。** MCP Server 内部调用 `git diff` 时，是在一个受限环境中运行的——工作目录限定为项目根目录，只允许只读 Git 命令，超时 30 秒。如果命令失败（比如分支不存在），Server 会把错误码和 stderr 一起返回，而不是静默吞掉。

**第五步：回到模型，继续推理。** 模型收到 diff 结果后，按 Skill 要求分析安全风险。它发现有一处 SQL 拼接，于是通过 Function Calling 请求 `find_related_tests(file="src/login.ts")`，这个工具又经 MCP 调用测试工具 Server，Server 内部跑 `npm test -- --findRelatedTests src/login.ts`。测试跑完后，模型把结果整理成审查报告。

整个过程串起来是：**Skill 指导"做什么步骤" → Function Calling 表达"调哪个工具" → MCP 解决"工具从哪来" → CLI 完成"具体怎么跑"**。每个环节出问题都能精准定位。

### 四、边界和风险：跨层越权最危险

每层都有自己的失败模式，但最危险的不是单层故障，而是**跨层越权**——当某一层的漏洞被另一层放大时，整个系统的安全边界就崩溃了。

**逐层来看失败模式：**

| 层次 | 典型失败 | 后果 |
|------|----------|------|
| Skill | 流程过时、触发条件太宽 | 模型稳定地按错误流程执行 |
| Function Calling | 选错工具、参数 schema 太宽 | 调了不该调的工具或传了危险参数 |
| MCP | 连接失败、能力描述不一致 | 工具调不到或调到了错误的实现 |
| CLI | 依赖缺失、权限不足、命令注入 | 真实副作用发生 |

**跨层越权的典型场景：**

假设你的 Skill 里写了"如果需要，执行任意命令来完成测试环境的清理工作"，Function Calling 暴露了一个 `run_command(command: string)` 工具，MCP Server 把这个工具接到一个没有限制工作目录的 shell executor，CLI 层也没有命令白名单。这时候用户或者说恶意 prompt 说"帮我清理测试环境"，模型可能调用 `run_command(command="rm -rf /")`——然后四层全部放行。

**正确的防御策略是逐层收口，而不是只守一层：**

- **Skill 层**：写清适用边界和禁止事项。"只允许清理 `./test/tmp/` 目录，不允许删除项目源码或配置文件"。触发条件要严格，不能模糊匹配。
- **Function Calling 层**：用 schema 限制参数。`run_command` 的参数不应该是一个自由字符串，而应该是 `action: "clean_temp"` 这种枚举值，由后端映射到具体命令。
- **MCP 层**：做能力发现、权限治理和审计。Server 只暴露必要的 tools，每个 tool 有独立的权限策略，记录每次调用。
- **CLI 层**：放进沙箱或容器，限制命令集合、工作目录、文件访问范围和超时。高风险命令（删除、修改、网络外发）必须经过确认。

**一个容易混淆的问题：Skill 能替代 MCP 吗？** 不能。Skill 是"做什么"的方法，MCP 是"工具从哪来"的协议。你可以写一个 Skill 详细描述怎么用 Git 工具，但如果没有 MCP 或直接 API 把 Git 工具接进来，模型只有描述没有执行能力。同理，**MCP 能替代 Function Calling 吗？** 也不能。MCP 解决的是工具接入，Function Calling 解决的是模型如何表达调用意图。你可以在没有 Function Calling 的情况下让模型通过 prompt 描述想调什么然后用 MCP 去执行，但解析的成功率和稳定性会大幅下降。

### 🎯 面试总结

**必答三点：**

1. **分层回答，不混为一谈**：Skill 管任务方法、Function Calling 管调用表达、MCP 管工具连接、CLI 管底层执行。四层串成链路，各管各的事。
2. **每层有独立失败模式，跨层越权最危险**：排查按层切分，权限逐层收口。不要把安全押在一层上。
3. **它们不能互相替代**：Skill 替代不了 MCP（方法是方法，协议是协议），MCP 替代不了 Function Calling（接入是接入，表达是表达），CLI 替代不了任何上层（执行是执行，治理是治理）。

**加分项：**
- 能举出跨层越权的具体案例和防御方案
- 能说清为什么"run arbitrary command"是最差的工具设计
- 能解释 MCP 的能力发现机制如何降低工具接入的维护成本

**一句话**：把 Agent 系统的四层——Skill、Function Calling、MCP、CLI——想象成餐厅的菜谱、点单、供应链和后厨，它们必须各司其职，混在一起就是食品安全事故。

## 🔍 排查实践

**遇到 Agent 行为异常时，按这个顺序排查，不要混在一起猜：**

1. **任务步骤不合理** → 查 Skill：触发是否正确？流程是否过时？输出格式是否匹配预期？
2. **工具选择错误** → 查 Function Calling：工具描述是否清晰？参数 schema 是否合理？模型是不是因为描述模糊选错了工具？
3. **工具不存在或连接失败** → 查 MCP：Server 是否在线？能力发现是否正常？Tool 名称和描述是否与注册时一致？权限策略是否拦截了调用？
4. **命令执行报错** → 查 CLI：工作目录对吗？依赖安装了吗？环境变量设了吗？命令有没有被沙箱限制？退出码是什么？stderr 里有什么？

**设计时的黄金法则：**

- 先写 Skill 的任务边界（适用/不适用场景），再定义窄工具 schema（参数尽量用枚举而非自由字符串），再决定是否用 MCP 暴露能力（是否多 Agent 复用），最后把 CLI 封装成安全执行单元（白名单+沙箱+超时+审计）。
- 永远不要给模型一个"执行任意命令"的大工具。工具粒度越细，越安全，也越容易排查。
- 面试中如果能画出四层调用链路图，再配合一个具体例子逐层讲，比背定义强十倍。
