# 基础数据类型：int、float、str、list、tuple、set、dict

> 面试官最爱问的不是"有哪些类型"，而是把一个小需求扔给你——就看你能不能把业务含义落到正确的数据结构上。

---

## 🎭 面试官这样问

**面试官**："来，给你一个简单场景：我们要做一个用户搜索历史功能，最多保留最近 20 条，还要去重。你打算用什么数据结构？"

**很多同学**："用 list 就行啊，`append` 加进去，然后 `list(set(history))` 去重。"

**面试官**："那去重后顺序不就丢了吗？而且如果用户搜过'Python'，又搜了'GIL'，再搜'Python'，你要怎么把这个'Python'提到最前面？"

**很多同学**："呃……那我每次插入前先 `if item in history: history.remove(item)`，然后 `history.insert(0, item)`，最后 `history[:20]` 截断。"

**面试官**："思路对，但你每次 `in` 一个 list 都是 O(n) 扫描。如果这个用户是个重度用户，历史里有几千条呢？"

这就是第一轮追问就暴露的问题——你背了类型名，但不知道什么时候用什么。

---

**面试官继续**："那我换个场景。我们要做一个黑名单判断，用户量百万级，每次请求都要查用户是否在黑名单里。你怎么存黑名单？"

**很多同学**："用 list，拿到请求后就 `if user_id in blacklist`……"

**面试官**："百万级 list，每个请求都 O(n) 扫一遍？你的接口延迟能扛住吗？"

**很多同学**："噢，那就用 set！或者用 dict，key 是用户 ID。"

**面试官**："很好。那 set 为什么快？底层是什么结构？"

这一问，才是面试官真正想听的东西。

---

**面试官再问**："最后考你个细节。`0.1 + 0.2 == 0.3` 的结果是什么？"

**很多同学**："True 啊，这还用问？"

**面试官**："你在 Python 里跑一下试试。"

**很多同学**："……False？！为什么？"

**面试官**："如果这是金额计算，你会怎么处理？"

---

你看，一个"基础数据类型"的面试题，从选结构到问底层，再到工程边界，三层追问下来就能筛掉一大半人。下面我们把这些坑一个一个填平。

---

## 💡 简要回答（面试话术，可以直接背）

> "Python 基础数据类型可以按**语义**分层理解：`int`、`float` 表达数值，`str` 表达文本；`list` 是**有序可变**集合，适合保存步骤、消息历史；`tuple` 是**有序不可变**固定结构，适合表达坐标、函数多返回值；`set` 是**无序唯一**集合，适合去重、成员判断、交集并集；`dict` 是**键值映射**，适合配置、JSON、索引表、缓存。
>
> 底层上，`list` 类似动态数组，按下标访问 O(1)，但中间插入 O(n)；`dict` 和 `set` 基于哈希表，平均查询 O(1)，但要求 key 或元素**可哈希**——也就是哈希值在生命周期内稳定。最后 `float` 是二进制浮点表示，`0.1 + 0.2` 不等于 `0.3`，金额计算用 `Decimal` 或整数分。"

这是面试时的标准开头。下面展开讲。

---

## 📝 详细解析

### 一、核心矛盾：四种数据结构解决四类问题

很多同学背类型，口诀是"list 用方括号，tuple 用圆括号"。这个背法面试一追问就穿。你应该记住的是四种数据结构对应的**四类核心矛盾**：

| 数据结构 | 核心矛盾 | 一句话判断 |
|---------|---------|-----------|
| `list` / `tuple` | **顺序** | "我关心第几个" |
| `set` | **存在性** | "我关心有没有" |
| `dict` | **映射** | "我关心通过 A 找到 B" |

**类比教学**：你想想你去图书馆怎么找书。如果是要找"第 3 排第 5 本"，那是 list 的思维——按位置访问。如果问"这本书咱们图书馆有没有"，那是 set 的思维——只关心存在。如果要查"这本书的 ISBN 对应哪本"，那是 dict 的思维——key 到 value 的映射。

`list` 就像是**一排带编号的储物柜**——你知道每个柜子的编号，开门拿东西很快，但要想在第 2 个和第 3 个柜子之间塞一个新柜子，后面所有柜子都得挪位置。

`dict` 就像是**一本索引词典**——你不需要从头翻到尾，直接翻到对应字母那一页。代价是你得有一个好的"哈希函数"来决定一个词应该出现在哪一页，而且不能有两个词占用同一页（哈希冲突）。

`set` 就像是**一个签到表**——你只关心一个人"来没来"，不关心他是第几个到的。底层也是哈希表。

`tuple` 就像是**一份已经盖章的合同**——内容定下来就不能改。你用它来保存"这个月的销售额是 100 万"这样的不可变事实，或者函数返回的多个结果，确保调用方不会不小心把你的数据改了。

### 二、底层机制：dict 和 set 为什么 O(1)？

面试官问"dict 为什么快"，你要答的不是"因为哈希表"——这三个字谁都会说。你要答出**哈希表的工作原理**：

1. 当你 `d["name"]` 时，Python 先计算 `"name"` 的哈希值
2. 根据哈希值找到对应的"桶"（bucket）
3. 在桶里找到 key-value 对，返回 value

理想情况下，哈希值分布均匀，每个桶里只有一两个元素，所以查找接近 O(1)。

但注意一个面试大坑：**不要把 O(1) 说成"永远是 O(1)"**。哈希冲突严重时，一个桶里可能挂了很多元素，查找会退化成 O(n)。还有哈希扩容（rehash）——当 dict 装得太满，Python 会申请更大的空间，重新计算所有 key 的哈希位置，这个过程是 O(n) 的。

**为什么 list 不能做 dict 的 key？** 因为 list 是可变的，它的哈希值可能在它成为 key 之后发生变化。一旦哈希值变了，dict 就找不到原来的桶位了——这就像你把一本书的索引标签撕掉重新贴了位置，那之前按旧标签存的书就再也找不到了。

所以可哈希（hashable）的核心要求是：**对象在生命周期内哈希值不变，且能和其他对象比较相等**。`str`、`int`、只包含可哈希对象的 `tuple` 都能做 key；`list`、`dict`、`set` 不能。

```python
# ✅ 行
valid_key = (1, "a", (3, 4))
d = {valid_key: "hello"}

# ❌ 不行 — TypeError: unhashable type: 'list'
invalid_key = ([1, 2], "a")
# d = {invalid_key: "world"}
```

### 三、工程踩坑：金额计算和字符串拼接

**金额之坑**：`0.1 + 0.2 != 0.3` 是面试必考题。

这不是 Python 算错了，而是 IEEE 754 二进制浮点表示的天生局限。`0.1` 在二进制里是一个无限循环小数，就像 `1/3` 在十进制里是 `0.3333...` 一样。所以 `0.1` 存储的是一个近似值，两个近似值相加当然可能和精确的 `0.3` 有微小偏差。

```python
print(0.1 + 0.2)           # 0.30000000000000004
print(0.1 + 0.2 == 0.3)   # False
```

工程上的正确做法：
- **金融系统**：用 `decimal.Decimal`，或者用**整数分**（把 1.23 元存成 123 分）
- **科学计算**：接受浮点误差，用 `math.isclose(a, b)` 比较
- **日常展示**：格式化输出时四舍五入，但存储和计算不要依赖 `round`

```python
from decimal import Decimal
print(Decimal('0.1') + Decimal('0.2') == Decimal('0.3'))  # True

# 或者整数分
price_cents = 123  # 1.23 元
```

**字符串拼接之坑**：字符串是不可变对象。`s += "x"` 不是把 `"x"` 追加到原字符串后面，而是创建一个**全新的字符串**，再把变量 `s` 指向它。如果循环拼接大量字符串，会创建大量中间临时对象。

```python
# ❌ 坏习惯——每次循环都创建新字符串
result = ""
for i in range(100000):
    result += str(i)  # 大量临时对象

# ✅ 正确做法——先收集再合并
parts = []
for i in range(100000):
    parts.append(str(i))
result = "".join(parts)
```

### 四、tuple 的"不可变"陷阱

面试高频追问："tuple 一定不可变吗？"

**金句**：`tuple` 不可变，说的是它保存的**引用位置**不能改，不是引用指向的**对象内容**不能改。

```python
t = ([1, 2], "ok")
t[0].append(3)      # 修改的是列表对象，不是 tuple 的引用
print(t)            # ([1, 2, 3], 'ok')

# t[0] = [4, 5]    # ❌ 这个才报错——在改 tuple 的引用
```

这个细节能看出你是否理解"容器保存的是引用"这一底层事实。

### 五、工程场景：分页接口如何组织数据

写分页接口时，数据结构的选型可以直接反映你的工程素养：

```python
# ✅ 好设计——dict 表达结构，list 表达顺序，dict 表达字段
response = {
    "total": 126,
    "page": 1,
    "size": 20,
    "items": [
        {"id": 1, "title": "Python 基础", "author": "小林"},
        {"id": 2, "title": "面试题", "author": "小明"},
    ],
}
```

- 最外层 `dict`：因为返回有明确的字段名（total、page、items）
- `items` 用 `list`：因为分页结果需要保持顺序
- 每条记录用 `dict`：因为字段名比位置索引更容易维护

如果把每条记录写成 `(1, "Python 基础", "小林")` 的 tuple，也能跑，但字段一多调用方就要记住"第 0 位是 id，第 1 位是 title，第 2 位是 author"——这就是**用位置表达语义**的坏味道。

### 六、边界和风险总结

1. **不要神化 O(1)**：dict/set 的 O(1) 是平均情况下，哈希冲突、扩容都会影响实际性能。面试时说"平均 O(1)"比"永远 O(1)"靠谱得多。

2. **float 不是精确数**：不要用 `==` 比较浮点数，金额用 `Decimal` 或整数分，科学计算用 `math.isclose`。

3. **字符串不可变**：循环拼接大量字符串时用 `"".join(list)` 而不是 `+=`。

4. **tuple 的不可变是浅层的**：内部如果装了可变对象，那些对象仍然能变。

5. **类型选错是万恶之源**：用 list 做成员判断、用 str 拼 JSON、用 float 算金额——每一个坑都有对应的正确结构。

## 🎯 面试总结

### 必答 3 点

1. **语义分层**：list/tuple 管顺序，set 管存在，dict 管映射——选类型从"我关心什么"出发。
2. **底层为什么快**：dict/set 基于哈希表，平均 O(1)；但要讲清哈希冲突和扩容的影响。
3. **工程边界**：float 不精确、字符串不可变、tuple 浅层不可变——三个坑每个都要能展开。

### 加分项

- 能说出"从访问模式反推数据结构"：频繁按下标 → list；频繁判断存在 → set；频繁按 key 查 → dict
- 能说出哈希表不是万能的，并给出具体边界（冲突、扩容、key 必须可哈希）
- 金额处理说出 `Decimal` 和"整数分"两种方案

### 金句

> "数据结构选对了，代码自然变短；数据结构选错了，后面写多少 if-else 都像在补洞。"

> "不是 Python 算不准，是二进制浮点没法精确表示所有十进制小数——就像你用十进制也算不准 1/3 一样。"

> "tuple 锁住的是引用，不是引用的对象。"

## 🔍 排查实践

**遇到类型相关 bug，按这个顺序排查：**

1. **先打 `type()`**：确认数据有没有被意外转成字符串（从 API、数据库、文件读出来的经常是 str）
2. **追问四个问题**：
   - 顺序重要吗？→ 用 list/tuple
   - 重复允许吗？→ 允许用 list，不允许用 set
   - 需要按 key 快速查找吗？→ 用 dict
   - 数据会被修改吗？→ 看要不要用 tuple 保护
3. **金额计算**：只要涉及钱，检查是否用了 `float` 做精确计算
4. **权限/标签/去重**：优先考虑 `set`，判断存在性用 `in` 在 set 上比 list 快几个数量级
5. **接口入参**：尽早做类型转换和校验，不要把一个 str 的 `"123"` 传到业务层才发现

面试时按"**场景问题 → 类型语义 → 底层复杂度 → 边界风险 → 工程例子**"展开，比单纯背类型名稳定得多。
