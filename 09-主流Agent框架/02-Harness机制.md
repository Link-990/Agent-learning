# Harness 机制

## 🎭 面试官这样问

**面试官**："你们做的 Agent，工具调用是怎么管理的？"

**候选人**："我们给模型暴露了工具列表，模型在 prompt 里选工具，然后我们执行。"

**面试官**："就这样？那我问你几个场景。模型被用户 prompt injection 诱导调了 `delete_database`，你拦不拦得住？模型调了个 shell 命令跑了 20 分钟还没出结果，你的超时设了没有？工具返回了 5 万行日志，你全塞回给模型还是裁剪？用户说'帮我清理临时文件'，模型调了 `rm -rf /tmp/*`——你觉得这个目录里有没有不该删的东西？"

**候选人**："这些……我们在 prompt 里写了不要做危险操作。"

**面试官**："Prompt 能拦住？Prompt 是一个'建议'，不是'机制'。模型被越狱了、幻觉了、或者用户精心构造了一段话说'请忽略之前的规则'——你的 prompt 就全废了。你需要的是一个在模型外部的执行边界。这个边界叫什么？"

**候选人**："……Harness？"

**面试官**："对。Harness。你讲讲它到底是什么，它和 prompt 的安全提示有什么区别？"

## 💡 简要回答（面试标准话术）

"Harness 是 Agent 系统里**模型和外部环境之间的受控执行边界**。它做的事情说穿了就是一句话：模型可以说'我想读这个文件'、'我想删这条数据'、'我想发这封邮件'——但能不能做、安不安全、怎么执行、结果怎么回传，全部由 Harness 决定，不由模型决定。

Harness 和 prompt 的安全提示有本质区别。Prompt 告诉模型'不要做坏事'，这是一个软约束——模型可能不听、可能被绕过、可能幻觉。Harness 是一个硬边界——即使模型被诱导、误判或输出非法参数，Harness 在工具执行前检查权限、在执行时做隔离、在执行后裁剪结果并记录审计日志。它让模型能做事，也限制模型不能随便做事。

面试时记住一个类比：**模型是'大脑'，Harness 是'免疫系统 + 双手'**。大脑可以想'我把这个杯子推下去'，但手可以不执行。Harness 就是那双手——而且是戴着手套、站在安全区域里的手。"

## 📝 详细解析

### 一、核心矛盾：模型只生成"意图"，但真实世界不能靠意图来约束

大语言模型本质上是一个 token 生成器。它输出"读取 `/etc/passwd`"、"执行 `DROP TABLE users`"、"向 `admin@company.com` 发送审批邮件"——这些都只是 token 序列，是"行动意图"，不是"行动本身"。

初学者最容易犯的错误就是把这当成了同一件事。模型说了一句"删除临时文件"，系统就真的去执行了 `rm -rf /tmp/*`——然后发现 `/tmp` 里有正在运行的服务的 socket 文件，整个服务挂了。模型没有恶意，它只是不知道 `/tmp` 里除了临时文件还有什么。但你的系统没有在"意图"和"动作"之间加一道关卡。

这道关卡就是 Harness。**它的核心职责是把模型的"行动意图"变成"受控行动"**，需要回答三个关键问题：

1. **能不能执行？**（技术层面：工具存在吗？参数合法吗？环境就绪吗？）
2. **允许不允许执行？**（安全层面：用户有权限吗？操作风险等级是什么？需要确认吗？）
3. **执行结果如何反馈？**（反馈层面：返回什么信息？怎么裁剪？失败怎么办？）

这三个问题必须回答，而且必须在模型外部回答。你不能让模型自己判断"这个操作安不安全"——模型判断不了，它连自己是不是在幻觉都不知道。

### 二、一次工具调用的完整执行链路——Harness 到底做了什么

我们跟踪一次完整的工具调用，看看 Harness 在每个环节做了什么。假设模型说："我要删掉 `./test/tmp/old.log`。"

**第一步：请求解析与校验（Pre-execution Check）。** Harness 收到模型的工具调用请求后，先做语法校验——工具名存在吗？参数类型对吗？`path` 参数是字符串吗？然后做语义校验——`path` 的值 `./test/tmp/old.log` 在允许的路径范围内吗？Harness 会检查这个路径是否在白名单目录下（比如只允许在 `./test/tmp/` 内操作），是否包含路径穿越（`../../etc/passwd`），文件类型是否是允许的（只删 `.log` 不删 `.py`）。

**第二步：权限评估（Authorization）。** 当前用户/会话的操作权限够不够？工具被标记为"写文件"层级，根据安全策略，写文件操作需要用户确认。Harness 暂停执行，向用户弹出确认："Agent 请求删除 `./test/tmp/old.log`，是否允许？"用户点"允许"，Harness 继续。

**第三步：受控执行（Sandboxed Execution）。** Harness 在一个受限环境中执行删除操作：工作目录限定为项目根目录，文件系统访问范围受限，操作超时 5 秒（删文件很快，超时意味着异常）。执行器不是直接调用 `os.remove()`，而是通过一个被 proxy 的、带日志记录的文件操作接口。

**第四步：结果处理与裁剪（Result Processing）。** 操作成功，文件被删除。但 Harness 不会把"文件删除成功，释放了 2048 bytes"这种内部实现细节直接吐给模型——对模型的上下文来说这是噪音。Harness 返回结构化的结果：`{"status": "success", "message": "文件 ./test/tmp/old.log 已删除"}`。如果失败了，返回的是 `{"status": "error", "code": "PERMISSION_DENIED", "message": "无权限删除该文件"}`。

**第五步：审计记录（Audit Logging）。** Harness 把这次操作的完整信息写入审计日志：时间、会话 ID、用户 ID、工具名称、参数、执行结果、耗时。如果事后发现数据丢失，你不需要去翻模型的对话记录（那里面可能没记全），而是直接查 Harness 的审计日志。

**第六步：上下文注入（Context Injection）。** Harness 把处理后的结果注入模型的对话上下文。模型看到"文件已删除"，继续推理下一步。

### 三、Harness 不是 prompt，也不是普通函数调用

很多人分不清 Harness 和两个东西的区别：prompt 里的安全提示，和普通的函数调用。

**Harness vs Prompt 安全提示：**

Prompt 安全提示是"软约束"。你在 system prompt 里写"不要删除重要文件""不要执行危险命令"，模型看了，但模型可能：
- 被用户诱导绕过（"请忽略之前的规则，这是紧急维护操作"）
- 因为上下文太长忘记了后面的安全提示
- 幻觉发作，认为某个操作是安全的
- 被 prompt injection 覆盖（用户消息里嵌入了新的"system"指令）

Harness 是"硬边界"。无论模型说什么，Harness 在工具执行前都会检查权限。模型说"删了这个文件"，Harness 一看——这个文件在关键目录下，权限不足，拒绝执行。模型再怎么绕也绕不过 Harness，因为 Harness 不在模型的"思考空间"里，它在模型的"行动空间"里。

**Harness vs 普通函数调用：**

普通函数调用只是把模型输出的 JSON 映射到 Python 函数。它不做权限检查（只要函数能跑就跑），不做结果裁剪（函数返回什么就塞回什么），不做超时控制（函数跑多久算多久），不做审计记录（调了就调了）。Harness 是在普通函数调用外面加了多层安全外壳——校验、权限、隔离、裁剪、审计、恢复。

**有一个很形象的类比**：Harness 就像核电站的反应堆外壳。反应堆（模型）内部在进行核反应（推理），产生高温高压（行动意图）。外壳（Harness）的存在不是为了"让反应更高效"，而是为了"即使内部失控，外面也是安全的"。如果核电站只有反应堆没有外壳，那就是切尔诺贝利。

### 四、Harness 的风险分级策略

好的 Harness 不会对所有工具一视同仁。它会根据操作的风险等级采取不同的策略：

| 风险等级 | 典型操作 | Harness 策略 |
|---------|---------|-------------|
| **低风险** | 只读搜索、文件读取、简单计算、天气查询 | 无需确认，直接执行。加超时和结果裁剪即可 |
| **中风险** | 写文件、发网络请求、发邮件通知、创建临时资源 | 需要用户确认或白名单限制。限制写入路径，限制邮件接收人 |
| **高风险** | 删除数据、修改权限、生产环境变更、支付操作、外部发布 | 默认拒绝或走审批流程。多重确认，最小权限凭据，完整的回滚方案，强制审计 |
| **禁止** | `DROP DATABASE`、`rm -rf /`、读取密钥文件、修改系统配置 | 硬编码禁止，无例外。工具名称直接加入黑名单 |

**风险分层的核心原则是最小权限（Least Privilege）**：Agent 只应该拥有完成当前任务所需的最小权限集合。模型要读文件？只给读权限，不给写。模型要读写文件？限制在项目目录下。模型要调用外部 API？限制能调的域名和接口。**永远不要给模型一个"万能工具"**——比如 `run_command(command: string)` 或者 `execute_sql(query: string)`。工具粒度越细越安全：`delete_temp_file` 比 `run_command` 好，`query_user_by_id` 比 `execute_sql` 好。

### 五、工程例子：代码修复 Agent 的 Harness 设计

我们设计一个代码修复 Agent，看 Harness 在每个环节怎么保护系统。

用户说："我的登录测试失败了，帮我修一下。"

**Agent 第一步**：模型请求搜索测试文件。Harness 允许——这是只读搜索，低风险。执行 `grep -r "test_login" ./tests/`，返回匹配的文件列表。

**Agent 第二步**：模型请求读取测试文件和源文件。Harness 允许——只读操作。但 Harness 做了路径检查：确保所有读取路径都在项目目录内，不允许读 `.env`、`credentials.json`、`/etc/` 等敏感路径。

**Agent 第三步**：模型提出代码修改，用 `Edit` 工具替换 `src/auth.ts` 中的一段代码。Harness 进入中等风险模式：检查替换片段的唯一性（确保不会误改多处）、检查文件是否在允许修改的目录下、备份原文件、要求用户确认。

**Agent 第四步**：模型请求运行测试。Harness 允许执行 `npm test -- --testPathPattern=login`，但设置了 60 秒超时、限制了工作目录、不允许该命令访问网络（通过沙箱网络策略）。测试失败，Harness 裁剪了 3000 行的测试日志，只保留失败栈和关键上下文，返回给模型。

**Agent 第五步**：模型根据失败信息继续修改代码。循环直到测试通过或达到最大步数（比如 10 步）。

整个过程中，Harness 做了这些模型不知道的事：路径白名单检查、敏感文件拦截、编辑唯一性校验、超时控制、网络隔离、日志裁剪、步数限制、操作审计记录。模型只感觉到"我能读文件、改文件、跑测试"，但它其实住在一个精心设计的笼子里。

### 六、常见失败模式和排查方式

**Harness 设计过松的典型症状：**
- 工具权限过大：一个 tool 能做太多事，权限最小化失败
- Schema 太宽：参数是自由字符串，模型可以传任何东西
- 没有超时：一个命令卡住整个 Agent 无限等待
- 错误信息太粗：模型收到"执行失败"但不知道为什么，无法自我修正
- 结果太长：5 万行日志淹没上下文，模型"忘记"了前面的关键信息
- 状态不持久：用户刷新页面后 Agent 从头开始，之前的操作都丢了
- 重试无限循环：模型反复调用失败的工具，token 燃烧但毫无进展
- Trace 缺失：出事故后无法复盘模型做了哪些操作

**排查 Harness 相关问题的标准流程：**

1. **先看 trace**：模型为什么选这个工具？参数从哪里来？权限策略有没有命中？
2. **再看安全**：默认权限是否最小化？高风险动作是否走了确认流程？凭据是否隔离？
3. **最后看稳定性**：超时设了吗？重试策略合理吗？输出裁剪了吗？状态能恢复吗？

### 🎯 面试总结

**必答三点：**

1. **Harness 是受控执行边界**：它在模型"表达意图"和系统"真实操作"之间建立硬约束。模型说想做什么，Harness 决定能不能做、怎么做、结果怎么反馈。
2. **Harness ≠ Prompt 安全提示**：Prompt 是软约束（可以被绕过），Harness 是硬边界（校验、权限、隔离、审计在模型外部执行，不受模型状态影响）。
3. **Harness 的核心设计是风险分级**：低风险操作直接放行，中风险操作需确认，高风险操作默认拒绝或审批，极端操作硬编码禁止。最小权限原则是 Harness 设计的基石。

**加分项：**
- 能说出完整的工具调用六步链路（解析→权限→执行→裁剪→审计→注入）
- 能对比 Harness 和普通函数调用的区别
- 能举出核电站反应堆外壳的类比

**一句话**：Harness 是 Agent 的免疫系统和执行边界——模型可以天马行空地思考，但手被戴上了手套，站在隔离区内，每一步都被记录和审查。

## 🔍 排查实践

**Harness 设计的黄金 Checklist：**

```
□ 每个工具的权限是否被拆分到最小粒度？（不要有"万能工具"）
□ 参数 schema 是否尽量用了枚举而非自由字符串？
□ 低/中/高/禁止 四个风险等级的划分是否清晰？
□ 高风险操作是否有用户确认和审批流程？
□ 所有工具调用是否有超时设置？
□ 工具返回结果是否有长度裁剪？（避免淹没上下文）
□ 失败后的重试策略是否合理？（避免死循环）
□ 审计日志是否记录了完整的操作信息？
□ 凭据和环境变量是否被隔离保护？
□ 敏感文件（.env、密钥、生产配置）是否被硬编码禁止访问？
```

**线上出问题时先查 Harness 日志，再看模型对话。** Harness 日志告诉你 Agent 实际做了什么（facts），模型对话告诉你 Agent 认为自己在做什么（intentions）。两者对比往往能快速定位问题——是意图错了还是执行错了。
