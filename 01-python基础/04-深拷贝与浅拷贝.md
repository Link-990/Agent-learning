# 深拷贝与浅拷贝

> 线上最难查的 bug，有一类叫"我明明复制了，为什么原来的也变了"。这道题看似只考 `copy.copy` 和 `copy.deepcopy` 两个函数，实际上考的是你对 Python 对象图、引用共享和状态隔离的理解。

---

## 🎭 面试官这样问

**面试官**："赋值、浅拷贝、深拷贝有什么区别？"

**很多同学**："赋值完全不复制，浅拷贝只复制外层，深拷贝全部复制。"

**面试官**："那你看这段："

```python
import copy

base = {"timeout": 3, "headers": ["token"]}
req = base.copy()           # 浅拷贝
req["headers"].append("trace-id")

print(base["headers"])
```

**很多同学**："`['token']` —— 因为 `copy()` 是新字典，两个字典不共享……"

**面试官**："你跑一下。"

**很多同学**："啊？`['token', 'trace-id']`？为什么外层的 `base` 也变了？不是已经 copy 了吗？"

这就是浅拷贝最迷惑人的地方——你复制了"壳"，但没复制壳里的东西。

---

**面试官继续**："那深拷贝一定安全吗？是不是所有情况下都应该用 `deepcopy`？"

**很多同学**："对，深拷贝完全隔离，安全。"

**面试官**："那你试试用 `deepcopy` 拷贝一个数据库连接。"

**很多同学**："……这不行吧？"

**面试官**："为什么不行？`deepcopy` 不是递归复制吗？"

---

**面试官再追问**："如果对象 A 和对象 B 都引用同一个对象 C，你深拷贝 A，那 B 里的 C 会被一起复制吗？"

**很多同学**："呃……不会吧，毕竟只是拷贝 A……"

**面试官**："那 `deepcopy` 的 `memo` 字典是干什么用的？循环引用怎么办？"

---

三层追问下来，你发现了吗——"深拷贝"和"安全"之间不是一个简单的等号。下面我们把整张对象图讲透。

---

## 💡 简要回答（面试话术，可以直接背）

> "Python 中赋值、浅拷贝、深拷贝的区别在于复制到**对象图的哪一层**。赋值不复制，只让新变量指向同一个对象，`a is b` 为 True。浅拷贝（`copy.copy`）创建新的外层容器，但**内部元素引用不变**——适合只隔离外层增删的场景。深拷贝（`copy.deepcopy`）递归复制内部对象，并用 `memo` 字典记录已复制对象，处理**循环引用**和**共享引用**——适合隔离嵌套可变状态。
>
> 但深拷贝不是万能的：文件句柄、数据库连接、线程锁、网络 socket 这类**外部资源对象**不能按业务语义复制；而且深拷贝大型对象有**显著的 CPU 和内存成本**。工程上更好的做法是明确哪些字段要共享、哪些要隔离，而不是无脑 `deepcopy`。"

---

## 📝 详细解析

### 一、核心矛盾：对象图，不是对象

**类比教学**：Python 中的对象不是孤岛，它们通过引用连接成一张**对象图**。赋值、浅拷贝、深拷贝的区别，就像三种不同的"复印"方式：

- **赋值**：你只是把一张写着地址的便签复印了一份。两张便签指向同一个房子（对象），所以谁进去动了家具，另一个人看到的也是动过的。
- **浅拷贝**：你盖了一栋**一样外观的新房子**（新外层容器），但里面的家具（内部元素）还是从旧房子搬过来的——两栋房子的沙发是同一个。你在客厅墙上挂一幅画（外层增删），不影响旧房子；但你在沙发上划一刀（修改内部可变对象），两个房子的沙发都有划痕。
- **深拷贝**：你不仅盖了新房子，还把里面的家具**全部复制了一套新的**放进去。现在两栋房子完全独立。

![深拷贝与浅拷贝概念总览](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/01-python-basics/04-deep-shallow-copy/01-概念总览.png)

```python
import copy

a = [[1, 2], [3, 4]]

b = a                 # 赋值：连房子都是同一个
c = copy.copy(a)      # 浅拷贝：新房子，旧家具
d = copy.deepcopy(a)  # 深拷贝：新房子，新家具

# 验证外层独立性
c.append([5, 6])
print(len(a))  # 2 — 外层 a 没变

# 验证内层共享
c[0].append(99)
print(a[0])    # [1, 2, 99] — 内层 list 还是共享的！

# 验证深拷贝隔离
d[0].append(88)
print(a[0])    # [1, 2, 99] — a 不受 d 影响
```

### 二、底层机制：浅拷贝做了什么

`list.copy()`、`dict.copy()`、切片 `[:]`、`copy.copy()` 本质都是浅拷贝。

浅拷贝的流程：创建一个新的外层容器对象，然后**把原容器里每个位置的元素引用，逐个复制到新容器的对应位置**。关键是"复制引用"——新老容器里第 i 个位置都指向同一个对象。

```python
users = [{"name": "Ada"}]
shadow = users.copy()

# 外层增删隔离
shadow.append({"name": "Bob"})
print(len(users))   # 1 — users 不受影响

# 内层修改共享
shadow[0]["name"] = "Changed"
print(users[0])     # {'name': 'Changed'} — 同一个字典！
```

**所以判断浅拷贝是否够用的规则很简单**：如果你的数据结构只有一层，或者内部对象都是不可变的（int、str、tuple 等），浅拷贝通常足够。因为不可变对象"修改"会产生新对象，不会发生原地"传染"。

### 三、底层机制：深拷贝和 memo 字典

深拷贝递归遍历整个对象图，为每个遇到的嵌套对象创建副本。但这带来两个问题：

**问题一：循环引用。** 对象 A 引用 B，B 又引用回 A。如果深拷贝无脑递归，会陷入死循环。

**问题二：共享引用。** 对象图里，A 和 B 都引用同一个人 C。如果深拷贝把 A 和 B 分别复制，C 会被复制两次——复制后 A' 和 B' 各自拿着一个不同的 C'，原本的共享关系被破坏了。

`memo` 字典就是解决这两个问题的。它是一个**"已复制对象登记表"**，key 是原对象的 id，value 是它的副本。

```python
# memo 的作用类似这样（简化）
memo = {}  # {id(原始对象): 副本对象}

def deep_copy(obj):
    if id(obj) in memo:
        return memo[id(obj)]        # 已经复制过了，直接返回副本
    if isinstance(obj, list):
        new_list = []
        memo[id(obj)] = new_list     # 先登记，再递归——防止循环引用
        for item in obj:
            new_list.append(deep_copy(item))
        return new_list
    # ... 其他类型 ...
    return obj  # 不可变对象直接返回引用
```

遇到循环引用时，第二次遇到同一个对象会直接从 `memo` 取到已创建的副本，不会无限递归。遇到共享引用时，两个地方引用的同一个对象会被复制一次，然后两个副本位置都指向这个同一个副本——保持了原始对象图的拓扑结构。

### 四、深拷贝不是万能按钮

**类比教学**：深拷贝就像复印机——能精准复制纸张上的文字图案。但如果你试图复印一张**信用卡**（数据库连接）、一把**钥匙**（文件句柄）、一个**正在计时的秒表**（线程锁），复印出来的"副本"要么没用，要么会造成灾难。

![深拷贝与浅拷贝运行机制](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/01-python-basics/04-deep-shallow-copy/02-运行机制.png)

```python
import copy

f = open("data.txt")
# f2 = copy.deepcopy(f)  # 即使能执行，复制出的"文件句柄"也是没意义的
f.close()
```

深拷贝真正适合的是**纯数据对象**：嵌套的列表、字典、基本类型、以及你明确知道所有字段都是可独立复制的自定义对象。自定义类还可以通过 `__copy__` 和 `__deepcopy__` 控制复制行为：

```python
class Config:
    def __deepcopy__(self, memo):
        # 明确指定：复制纯数据字段，但共享连接池
        new = Config()
        new.data = copy.deepcopy(self.data, memo)
        new.pool = self.pool   # 连接池不复制，共享！
        return new
```

### 五、工程场景

**场景一：请求上下文隔离**

```python
base_context = {
    "user": "u1",
    "steps": [],
    "meta": {"retry": 0},
}

# ❌ 错误：浅拷贝导致 steps 和 meta 共享
ctx1 = base_context.copy()
ctx1["steps"].append("fetch_profile")

# ✅ 推荐：选择性隔离
ctx2 = {
    **base_context,
    "steps": list(base_context["steps"]),  # 显式复制
    "meta": dict(base_context["meta"]),
}
```

**场景二：配置模板**

```python
template = {
    "timeout": 5,
    "retries": 3,
    "headers": {"Content-Type": "application/json"},
    "pool": get_connection_pool(),  # 不能复制的资源
}

# ❌ 无脑 deepcopy：pool 复制了可能出错
# config = copy.deepcopy(template)

# ✅ 手动控制
config = {
    **template,
    "headers": dict(template["headers"]),  # 复制可变字段
    "pool": template["pool"],              # 共享连接池
}
```

### 六、边界和风险总结

1. **浅拷贝够不够**：数据结构只有一层？或内部都是不可变对象？→ 浅拷贝够。否则需要深拷贝或手动复制。
2. **深拷贝不能复制的东西**：文件句柄、数据库连接、线程锁、网络 socket、生成器、C 扩展对象等。
3. **深拷贝的性能代价**：大型嵌套对象深拷贝可能消耗大量 CPU 和内存。在请求路径中频繁深拷贝是性能杀手。
4. **深拷贝 + 循环引用**：Python 的 `deepcopy` 通过 `memo` 处理了循环引用，但你自定义的 `__deepcopy__` 可能打破这个机制。
5. **共享引用的语义**：深拷贝会尽量保持原始对象图中的共享关系。如果你不想保持共享，你需要手动控制。

## 🎯 面试总结

### 必答 3 点

1. **三种操作的区别**：赋值（不复制、id 相同）、浅拷贝（新外壳、内部引用相同）、深拷贝（递归复制、内部也隔离）
2. **浅拷贝的适用场景**：单层结构 / 内部是不可变对象 —— 外层增删隔离即可
3. **深拷贝的 `memo` 作用**：处理循环引用（防死循环）和共享引用（保持对象图拓扑）

### 加分项

- 能说出 `deepcopy` 不适用于文件句柄、连接、锁等外部资源对象
- 能讲清工程上"选择性隔离"比无脑 `deepcopy` 更好——表达意图、控制成本
- 知道自定义类可以覆写 `__copy__` / `__deepcopy__`
- 能讲清浅拷贝的性能优势和对"单层结构足够"的判断标准

### 金句

> "赋值是贴便签，浅拷贝是换房子不换家具，深拷贝是连人带房子全复制一份。"

> "深拷贝不是万能按钮——你能复印一张纸，但不能复印一把钥匙。"

> "工程中最重要的不是 copy 的函数，而是你清楚哪些字段要共享，哪些字段必须隔离。"

## 🔍 排查实践

**遇到"数据被意外修改"的 bug，按这个顺序排查：**

1. **用 `id()` 定位共享**：`print(id(original_nested), id(copied_nested))` —— 如果 id 相同，就是共享引用。
2. **追踪原地修改**：`append`、`update`、`dict[key] = value`、`list[i] = value` —— 看看这些操作作用在哪个对象上。
3. **判断复制深度**：数据结构是单层（`[1, 2, 3]`）还是嵌套（`[{"k": [1]}]`）？单层浅拷贝足够，嵌套要深入。
4. **所有权原则**：设计数据结构时，让只读配置和请求临时状态分开。跨请求复用的对象不要塞可变临时字段。
5. **不要为了省事到处 `deepcopy`**：优先手动复制必要层级，精确表达意图。

面试回答按"**赋值 → 浅拷贝 → 深拷贝 → memo 的作用 → 不适用场景 → 工程做法**"展开，逻辑链会很完整。
