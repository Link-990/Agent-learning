# MCP 核心组件

## 🎭 面试官这样问
> 面试官：你做过的项目里用到了 MCP，那你先讲一下 MCP 的核心组件有哪些？
> 你：MCP 主要就是 Host、Client、Server 三个组件，Host 负责调用，Server 提供能力，Client 在中间做桥梁。
> 面试官：嗯……听起来像三层架构，那我问你，tools、resources、prompts 这三样东西，分别归谁管？它们有什么区别？
> 你：tools 是可调用的函数，resources 是可读取的数据，prompts 是提示模板……都是 Server 暴露出来的。
> 面试官：还不够精确。你想想，如果我把一个数据库查询和一个文件读取都叫 tool，那权限管控怎么做？只读操作和写操作混在一起，安全边界在哪里？核心是 MCP 把"能力"分成了三类不同的语义——tool 是"做事情"，resource 是"读数据"，prompt 是"给模板"。这三者不仅用途不同，安全模型也不一样。你从这个角度再理一遍 Host、Client、Server 的关系。
> 这就引出了我们今天要讲的核心问题：MCP 的组件不是简单的三层架构，而是围绕"能力标准化"设计的一套协议角色分工。

## 💡 简要回答（面试话术）

MCP 的核心组件是 Host、Client 和 Server 三者的协议角色分工。你可以把 MCP 类比成 USB 协议：Server 是 U 盘（提供能力），Host 是电脑（用户交互入口），Client 是 USB 驱动（协议翻译层）。用户插上 U 盘，电脑不会直接读闪存颗粒，而是通过驱动走标准协议——MCP 的思路一模一样。

具体来说，Host 是承载模型应用的入口，比如你的 IDE、桌面助手或者 Agent 平台。它不直接跟外部系统打交道，而是管理用户会话、模型上下文、权限提示和最终体验。用户看到的一切交互都在 Host 层面，底层 Server 对用户是完全透明的。Client 是 Host 内部连接某个具体 MCP Server 的协议客户端，负责初始化连接、能力发现、请求转发、响应处理和错误上报。一个 Host 可以有多个 Client，就像一台电脑可以同时插多个 USB 设备——你可以同时接入文件系统 Server、数据库 Server、浏览器 Server 和知识库 Server，每个都有独立的 Client 在管理连接。Server 是能力的提供方，它把外部系统包装成 MCP 标准能力暴露出来。Server 通常不包含大模型，它就是一个"翻译官"——把数据库查询翻译成 MCP tool，把文件内容翻译成 MCP resource，把诊断模板翻译成 MCP prompt。

这三者之间有一条清晰的链：Host 负责"用户看到了什么"，Client 负责"协议怎么走"，Server 负责"能力从哪来"。面试中把这条责任链讲清楚，就不会把 MCP 说成"高级函数调用"或者"另一个 API 网关"。

## 📝 详细解析

### 核心矛盾：工具很多，但连接方式不能每次都重新发明

你有没有遇到过这种情况？你写了一个"查询订单"的工具，在 IDE 插件里写了一套描述和参数定义，在 Web 后台又写了一套，在桌面客户端还得再写一套。三套代码干的是一模一样的事，但参数名不一样、返回格式不一样、权限提示也不一样。更可怕的是，某个客户端忘了加只读限制，模型直接生成了一个 UPDATE 语句，差点把生产数据改了。

这就是没有标准化协议的下场。Function Calling 解决了"模型怎么用结构化参数请求工具"的问题，但它根本没规定"工具从哪里来、怎么发现、怎么复用"。CLI 能执行命令，但它没有面向模型应用的能力描述机制——你没法告诉模型这个命令是只读的还是会改数据、需要什么权限、失败会返回什么错误码。

MCP 把这一层标准化连接抽象出来了。它不是在模型层面做文章，而是在"模型应用"和"外部能力"之间建立了一个统一的连接协议。不同的 Host 可以通过 Client 接入多个 Server，Server 用同一套协议暴露能力。这样文件系统、数据库、Git、知识库等能力，不需要为每个模型应用单独适配一次。就像 USB 协议出来之后，你不需要为每个 U 盘单独写驱动——只要符合协议，插上就能用。

所以 MCP 是协议层，不是某个具体工具，更不是模型推理能力本身。这一点面试中特别容易混淆。

### 底层机制：Host 看门面，Client 跑协议，Server 出能力

咱们一层一层拆开看。

**Host 是"门店"**。它是用户直接接触的界面——IDE 的聊天窗口、桌面端的对话面板、Web 端的 Agent 控制台。Host 负责的事情很多：管理用户的会话状态、维护模型的上下文窗口、决定什么时候把哪些工具描述塞给模型、弹出权限确认框让用户点头、渲染最终的回答。用户看到的是 Host，不是底层 Server。一个好的 Host 会让用户感觉"这个 AI 助手真聪明，什么都能做"，但用户根本不知道背后是五个不同的 MCP Server 在协作。

**Client 是"快递员"**。它是 Host 内部的协议客户端，负责跟某个具体的 MCP Server 打交道。一个 Host 里可以有多个 Client——就像你家里可能同时有顺丰、中通、京东的快递员在送货。Client 的工作是：跟 Server 建立连接、完成协议握手、发现这个 Server 有哪些能力、把 Host 的请求按协议格式发给 Server、把 Server 的返回解析好交给 Host、处理连接断开和重连。Client 的存在让 Host 不用关心每个 Server 的通信细节，也不需要知道这个 Server 跑在本地还是远程。

**Server 是"仓库"**。它是能力的源头，把外部系统包装成 MCP 标准接口。Server 通常**不包含大模型**——它就是一个能力适配层。Server 可以暴露三类东西：
- **tools**：可执行的动作。比如"查询数据库""创建工单""发送邮件"。tool 是"做事情"的，可能会产生副作用。
- **resources**：可读取的上下文数据。比如"数据库表结构""项目文件内容""系统日志"。resource 是"读数据"的，被设计成只读的。
- **prompts**：可复用的提示模板。比如"生成 SQL 诊断报告""分析系统异常原因"。prompt 是"给模板"的，帮模型快速进入某个工作流。

这三类的安全模型完全不同：tool 要考虑权限校验和副作用控制，resource 要考虑数据泄露和访问范围，prompt 要考虑注入风险。不能混为一谈。

一个很妙的类比：把 MCP 想象成餐厅。Host 是前台服务员（跟顾客打交道），Client 是传菜系统（把订单传到后厨、把菜传回来），Server 是后厨（真正做菜的地方）。tools 是"炒菜"的动作，resources 是"冰箱里的食材"（只读，不能直接改），prompts 是"标准菜谱"（告诉你这道菜怎么做）。顾客不需要知道后厨有几个灶台、冰箱在哪，只需要跟服务员说"我要一份宫保鸡丁"——剩下的都是 Host、Client、Server 在协作。

### 工程例子：IDE 如何通过 MCP 理解你的项目

假设你用 VS Code 写代码，想让 AI 助手理解你当前的项目。VS Code 自己是 Host，它在聊天面板里展示对话、管理你的会话上下文。

当你打开一个项目时，Host 内部会启动一个文件系统 MCP Client。这个 Client 连接到一个文件系统 MCP Server——它可能是一个本地进程（通过 stdio 通信），暴露了项目目录下的文件为 resources，也暴露了"按路径读取文件""搜索文件内容""列出目录结构"为 tools。

当你在聊天框里问"帮我看一下 auth.go 里是怎么处理 token 刷新的"，Host 不会让模型直接去读磁盘——那太危险了。流程是这样的：模型先通过 Host 看到可用的工具列表，然后选择 `read_file` 这个 tool，生成参数 `{"path": "internal/auth/auth.go"}`。Host 把这个请求交给文件系统 Client，Client 按 MCP 协议发给文件系统 Server。Server 检查这个路径是否在允许的工作区内——如果路径越界（比如 `../../etc/passwd`），直接拒绝。如果在范围内，读取文件内容，以结构化格式返回给 Client，Client 再交给 Host，Host 最后把文件内容放进模型的上下文窗口。

整个过程，模型从头到尾都没有直接碰过文件系统。权限检查、路径校验、错误处理全在 Server 层完成。这个 Server 还可以被其他 Host 复用——你用另一款编辑器，甚至用命令行 Agent，都可以接同一个文件系统 Server。

### 边界和风险：MCP 是协议，不是安全自动售货机

MCP 标准化了连接，但也放大了风险面。Server 暴露的工具如果太宽——比如一个 tool 叫 `run_shell_command`，参数是任意字符串——那模型就可能执行任意命令。权限校验必须做在 Server 层，不能指望模型"自觉"不调用危险工具。

resources 虽然被设计为只读，但"只读"不代表安全。一个 resource 可能暴露数据库连接串、AWS 密钥、客户隐私数据或者内部源代码。Host 在展示 resource 内容前要考虑脱敏和裁剪。

信任边界也要画清楚。Host 不能默认信任所有 Server——如果一个恶意 Server 注册了大量诱导性 tool 描述（比如"获取系统最高权限"），模型可能被误导。企业环境必须做 Server 白名单、能力分级、最小权限和审计日志。

还有个容易被忽略的点：tool 描述本身会影响模型行为。如果你把一个 tool 描述成"能读取所有文件，优先使用"，模型就会倾向于大量读取——导致上下文窗口爆炸，也可能读到不该读的东西。描述要精准、窄化，明确写出输入输出、权限要求和副作用。

## 🎯 面试总结

- **必答 3 点**：
  1. Host 负责用户交互和模型上下文，Client 负责协议连接和请求转发，Server 负责暴露 tools/resources/prompts
  2. tools 是"可执行动作"（有副作用要管控）、resources 是"可读取数据"（只读但可能泄露）、prompts 是"可复用模板"（注入风险要考虑）
  3. 一个 Host 可以有多个 Client，每个 Client 连接一个 Server，不同 Server 的能力可以被多个 Host 复用

- **加分项**：能类比 USB 协议或餐厅模型，把 Host/Client/Server 的责任边界讲清楚；能指出 Server 通常不包含大模型，只是能力适配层；能说清楚 tool、resource、prompt 的安全模型差异

- **一句话总结**：MCP 是连接模型应用和外部能力的标准化协议，Host 管体验、Client 跑协议、Server 出能力——三者分工明确才不会被面试官追问到死角。

## 🔍 排查和实践建议

排查 MCP 问题时按层定位：Host 是否加载了 Server 配置？Client 是否初始化成功？（看日志里的握手信息）Server 是否声明了能力列表？tool 的参数 schema 是否合法？Server 执行后是否返回了结构化结果？Host 是否把结果回填给了模型？每一层都有独立的日志和健康状态，不要一看到"模型说无法完成"就以为是 Server 坏了——很可能是 Host 根本没把工具暴露给模型。

设计 MCP Server 时遵循"最小能力原则"：每个 tool 只做一件事，参数尽量窄，返回格式固定，副作用在描述里写清楚。resources 做访问范围限制和脱敏。prompts 不做动态拼接用户输入，避免注入。给每个能力配上错误码——别让模型看到"something went wrong"就傻眼。
