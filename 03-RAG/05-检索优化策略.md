# 检索优化策略

## 🎭 面试官这样问

> 面试官：RAG 系统上线后效果不好，用户反馈答非所问、漏掉关键信息、引用旧政策，你怎么优化？

> 你：可以调大 top_k 多召回一些候选，或者换个更强的 embedding 模型，再不行就加 rerank。

> 面试官：你说的都是调参。我问一个具体场景——用户问「我买了 10 天的耳机还能退吗」，系统回答「超过 7 天不支持退货」，但知识库里最新的政策是 15 天可退。我告诉你，正确文档确实在知识库里，也被向量检索召回了，但排在候选第 8 位，top_k 设的是 5。你现在觉得问题是调大 top_k 吗？

> 你：呃……top_k 从 5 调到 8 可以暂时解决这个 case，但根本问题可能是排序层——正确文档分数不够高。需要看为什么旧政策排在新政策前面，是不是元数据没参与排序……

> 面试官：对。那你再想：如果旧政策分数高是因为它的分块更大、关键词更密集，而新政策的 chunk 被切得太碎——这还算排序层的问题吗？还是分块层的问题？

> 你：这是分块层的问题影响了排序。也就是说，RAG 是链路系统，任何一环都可能影响最终答案，优化要按链路逐层排查。

> 面试官：好，你就从这个思路出发，把 RAG 检索优化的七个层面串起来讲一遍。记住：不要只列工具，要说清每一层查什么、怎么修。

这个追问串起来，面试官要听的不是「调 top_k」这种单点答案，而是从数据到生成的完整排查链路。

## 💡 简要回答（面试话术）

> RAG 检索优化不是调一个参数的事，而是按链路逐层排查。我把整个流程拆成七层：数据质量 → 分块策略 → Query 改写 → 多路召回 → Rerank 精排 → 上下文组装 → 评测闭环。

> 数据层先确认正确资料有没有入库、清洗有没有丢内容、元数据准不准。分块层看关键条件有没有被切断，chunk 大小是否合适。Query 层看多轮对话有没有补全、口语表达有没有改写。召回层看是否需要混合检索——向量召回覆盖语义、关键词召回命中精确符号。排序层用 reranker 把真正能回答问题的片段排到前面。上下文组装层控制放入 prompt 的片段数量和顺序，去重、带来源、标版本。每一层的优化目标不同：召回阶段追求不遗漏，排序阶段追求精准排序，上下文阶段追求信息可用。

> 最关键的是要有评测集。没有评测集，优化就是打地鼠——修好这个 case 搞坏那个 case。我会给每次线上请求记录 query、召回列表、rerank 分数、最终上下文和模型答案，定期把失败样本回流到评测集，按问题类型统计失败原因。

## 📝 详细解析

### 核心矛盾：召回、排序和上下文可用性

RAG 优化的本质是同时平衡三个互相制约的目标：

**第一，召回率（Recall）。** 正确答案必须在候选集里。如果相关 chunk 根本没被召回，后面 rerank 和生成再怎么强也救不了。这个阶段追求「宁可错杀一千，不可放过一个」。

**第二，相关性（Relevance）。** 召回的内容不能只是主题接近，而要真正能回答当前问题。「如何申请退款」和「退款多久到账」都和退款有关，但不是同一个问题。这个阶段追求「精准过滤，把答非所问的踢出去」。

**第三，上下文可用性（Context Usability）。** 最终放入 prompt 的资料要完整、少噪声、有来源、版本正确。如果上下文里新旧政策混在一起，模型可能选错；如果缺少引用编号，用户无法验证。这个阶段追求「信息干净、可追溯」。

一句话记住 RAG 优化三阶段目标：**召回阶段别漏答案，排序阶段把答案放前面，上下文阶段让模型看得懂、用得上。**

![RAG 检索优化链路](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/03-RAG/05-retrieval-optimization/02-运行机制.png)

### 第一层优化：数据质量和元数据

很多 RAG 问题看着像检索不行，其实是数据根本没准备好。这一层查三件事：

**文档是否完整入库。** 政策更新了，但同步管道断了——知识库里还是旧的。高频变化的数据（政策、价格、库存）必须有自动同步、增量更新、失败告警和版本记录，不能靠人工偶尔上传。

**清洗是否干净。** PDF 解析中，页眉页脚混进 chunk，表格被拆成乱码，标题层级丢失，版权声明和广告文字污染了知识库。你召回的「正确文档」可能已经被清洗环节毁了。

**元数据是否准确。** 文档来源、更新时间、版本号、权限范围、产品线、地区、生效时间——这些字段决定了片段是否适用。用户问 2026 年耳机政策，但召回的是 2024 年旧版——很多时候不是检索不聪明，而是版本元数据没参与排序或过滤。

### 第二层优化：分块策略

分块是 RAG 的地基，地基不稳后面全是白搭：

- chunk 太大 → 多个主题压成一个向量，检索模糊，噪声多
- chunk 太小 → 条件被切断，模型只看到半条规则
- 标题丢失 → 片段离开原文后失去上下文锚点
- overlap 太大 → 重复片段占索引也占 prompt

优化手段：按标题层级切分、保留上级标题和元数据、对不同文档类型使用不同切分器、使用父子块检索。要紧密结合失败样本——模型总是漏条件 → 检查条件是否被切到其他块；召回太长太杂 → 检查 chunk 是否包含多个主题。

### 第三层优化：Query 改写和问题理解

用户的问题常常不适合直接检索。多轮对话里：「我买了个耳机」→「这个还能退吗」——「这个」指什么？不补全就搜不到东西。口语化表达：「这玩意退货有啥要求」→ 需要改写成「耳机退货条件和流程」。

常见改写操作：补全省略信息（指代消解）、提取实体和关键词、改写口语为书面语、扩展同义词、拆分复杂问题为多个子问题、结合历史对话消歧。

**风险提示：** 改写模型可能引入用户没说过的条件，或者把问题范围扩大。生产上最好保留原 query 和改写 query 做多路召回，并记录改写结果方便回放排查。

### 第四层优化：多路召回和混合检索

单一路径容易漏答案，生产系统常见多路召回组合：

```
向量检索（语义覆盖）
关键词检索（精确匹配）
元数据过滤（版本 / 权限 / 地区）
规则召回（高频固定问题模板）
历史热门问题召回（高频 query 缓存）
```

召回阶段宁可多拿候选，但也不能无限扩大 top_k——候选越多，精排成本越高，最终上下文也可能越吵。通常召回 50-100 条候选，精排后取前 3-5 条。

### 第五层优化：Rerank 精排

向量相似度不等于答案相关——这是 RAG 里最容易踩的坑。用户问「退款多久到账」，向量检索把「如何申请退款」排到前面——两者语义都跟退款有关，但后者根本答不上到账时间。

Reranker 的意义就在这：它会同时读 query 和候选文档全文，判断「这个文档能不能回答这个问题」，而不只是「这个文档语义上像不像」。常见做法是召回 50 条候选，rerank 选出前 5 条送入 prompt。精排也要看成本——reranker 比单纯向量相似度慢很多，高并发场景要控制候选数量、做缓存、分层精排，或只对低置信度请求启用。

### 第六层优化：上下文组装和生成约束

很多系统召回和排序都对了，最后仍答错——问题出在上下文组装：

- 放了太多片段 → 模型注意力被稀释
- 旧政策排在新政策前面 → 模型选了过时信息
- 多个 chunk 冲突 → 模型不知道信哪个
- 没带来源和标题 → 答案无法追溯
- 没要求资料不足时拒答 → 模型编造答案

优化手段：去重相似 chunk、按相关性和时效性排序、保留标题来源版本、控制总 token 数、对长片段做压缩、prompt 明确要求「仅基于提供的资料回答」、要求输出引用。对于冲突资料，要设定优先级规则——生效时间最新的优先、权限匹配的优先、来源权威级别高的优先。

### 第七层优化：评测集和闭环

没有评测集，RAG 优化就是玄学。一个基本评测集至少包含：

- 用户问题 + 标准答案
- 应召回文档 ID 列表
- 问题类型标签（精确符号 / 同义改写 / 版本时效 / 多条件 / 多跳推理）
- 失败原因记录

评测要分层看：检索层看 Recall@K（相关文档是否被召回），排序层看 MRR/NDCG（相关文档排多靠前），生成层看答案是否正确、完整、有引用，安全层看资料不足时是否拒答。每次改分块、换 embedding、调 top_k、加 rerank 或改 prompt，都要跑同一套评测集前后对比。

### 边界和风险：优化不是越多越好

- **top_k 不是越大越好。** 扩大提高覆盖但也增加精排成本和上下文噪声。
- **overlap 不是越大越好。** 过大增加索引体积和重复召回。
- **query 改写不是越复杂越好。** 改写错误会污染检索方向。
- **rerank 不是无成本。** 高并发下可能成为延迟瓶颈。
- **只看最终答案不保留中间结果。** 线上失败无法定位，这是最大的工程风险。必须记录 query、改写 query、召回列表、rerank 分数、最终上下文、模型输出——可回放才能定位。

## 🎯 面试总结

- **必答 3 点：** ① RAG 优化要按链路逐层排查，从数据、分块、改写、召回到排序、组装、评测，不能一上来就调 prompt；② 每层优化目标不同——召回追求覆盖，排序追求精准，上下文追求可用；③ 评测集是优化的前提，没有评测集就是凭感觉打地鼠。
- **加分项：** 能清晰讲出七层各自的问题现象和修复手段。能举例说明「向量相似不等于答案相关」如何用 reranker 解决。能讲出上下文组装中的版本冲突处理规则。能提到全链路日志回放和灰度上线。
- **一句话总结：** RAG 优化不是换模型换参数，而是沿着数据→分块→改写→召回→排序→上下文→评测这条链路，找到真正的瓶颈并精准修复。

## 🔍 排查和实践建议

线上 RAG 答错，按这个顺序排查，90% 的 case 能定位：

1. **数据层**：正确资料是否存在？是否同步入库？清洗有没有丢内容或混噪声？
2. **分块层**：chunk 是否包含完整规则？关键条件、例外有没有被切断？
3. **Query 层**：多轮对话是否补全？口语化是否需要改写？用户实际意图是什么？
4. **召回层**：正确 chunk 是否进入候选集？是否需要混合检索？
5. **排序层**：正确 chunk 是否排在前面？旧版本是否排在新版本之前？
6. **上下文层**：prompt 是否太长或重复？多个资料是否冲突？来源和标题是否保留？
7. **生成层**：模型是否被约束基于资料回答？资料不足时是否允许拒答？
8. **归类**：这个失败个案是否能代表一类问题？需要回流到评测集吗？

实践上，建立全链路日志——每次请求记录 query、改写 query、召回列表、rerank 分数、最终上下文、模型答案和引用。定期复盘失败样本，按问题类型统计失败原因。只有从「RAG 不准」定位到「哪一层不准」，优化才是可复现、可验证的工程实践。
