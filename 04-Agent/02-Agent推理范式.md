# Agent 推理范式

## 🎭 面试官这样问

> 面试官：你们项目里 Agent 用的是什么推理范式？ReAct 还是 Plan-and-Execute？
> 你：我们主要用 ReAct，就是让模型边想边做，每步先推理再行动，拿到观察结果后再推理下一步。
> 面试官：那如果一个排障 Agent 反复执行同一个 explain 命令，每次的观察结果几乎一样，但它就是不停——你的 ReAct 怎么处理？
> 你：这个……可以设最大步数限制吧。
> 面试官：设步数限制是兜底，不是推理范式本身解决的问题。我再问你，如果一个 Plan-and-Execute 的 Agent 生成了第一步计划就开始执行，执行到第三步发现初始计划的前提错了，它怎么办？
> 你：应该支持 replanning，也就是重新规划。
> 面试官：那重新规划的时候，已经执行完的前两步的结果要不要保留？要不要回滚？谁来判定"计划错了"？这才是工程上真正要解决的问题。

面试问推理范式，面试官不是在考你背得出几个名词——他是在看你能否判断：**在什么任务特征下、把"思考"放在"行动"的哪个位置、用什么机制兜底**。ReAct、Plan-and-Execute、Reflection、Tree of Thoughts 这些不是产品标签，而是不同的控制策略。能说出每种范式的适用条件和不适用条件，才是面试要的答案。

## 💡 简要回答（面试话术）

我理解 Agent 推理范式的本质是**在一个不可靠的语言模型外面，套一层可靠的认知脚手架**，把模型的开放生成约束为"什么时候观察、什么时候推理、什么时候行动、什么时候修正、什么时候停止"的可控流程。

从任务特征角度看，选范式主要看三点：信息完整度、路径确定性和动作风险。信息不完整、需要边查边做时用 ReAct，模型每步先判断需要什么信息，再调用工具获取观察，然后根据新信息决定下一步——它像一个在陌生环境里摸索的人。但 ReAct 容易走一步看一步，必须有步数、无进展检测和成本上限兜底。目标明确、步骤相对稳定时用 Plan-and-Execute，先整体规划再逐步执行，效率高且可审计，但必须支持中途 replanning——当观察结果和初始计划矛盾时，要能停下、回看、重新规划。

Reflection 不是独立范式，而是加在执行后的自检环节——代码生成后自查是否满足需求、是否引入风险，能有效提升质量但模型也可能"自信地修错"，必须和真实测试、规则校验结合。Tree of Thoughts 适合方案设计、数学推理等搜索空间大的场景，但成本高，要设分支数上限和剪枝规则。

总结来说，推理范式的价值不在复杂，而在合适——**信息越不确定越需要观察驱动，路径越稳定越可以计划驱动，风险越高越需要多层校验**。工程上常常混合使用，没有银弹。

## 📝 详细解析

### 核心矛盾：LLM 会续写，不天然会控制流程

咱们先回到 LLM 的本质——它就是一个根据上文预测下一个 token 的续写机器。你给它一个上下文，它给你一个看起来合理的续写。问题在于：**"看起来合理的续写"不等于"正确的决策时序"**。

没有推理范式约束时，Agent 会出三类经典问题。**第一类是过早行动**，模型还没有确认用户身份和操作权限，没有理解完整的任务目标，上来就调用工具。比如用户说"帮我看看最近的订单"，模型直接调 `query_order(limit=10)`，但用户其实只关心"待退款"的订单。**第二类是反复行动**，模型陷入"搜索→总结→再搜索"的循环，每一步都在做事，花了很多 token，但实际上没有推进目标——我管这叫"看起来很努力的摸鱼"。**第三类是不可解释**，模型最终给了一个结论，但你不知道它依据了哪些观察、在哪个推理步骤做出了关键判断、有没有遗漏关键信息——出了问题没法定位。

这三类问题的根因是同一个：**LLM 没有内置任务状态机**。而推理范式就是给模型外挂一个状态机。ReAct 把状态抽象为"思考 → 行动 → 观察"的循环，Plan-and-Execute 把状态抽象为"规划 → 执行 → 验证"的阶段，Reflection 在"生成 → 检查 → 修正"之间插入自检节点。每一种范式都是在回答同一个问题：模型的推理能力应该放在流程的哪个位置？

### 底层机制：四种范式的控制策略对比

我做一个类比帮你记住这四种范式。想象你到了一个陌生的城市，要找一家餐厅。

**ReAct：走一步看一步**。你站在路边，先观察周围有什么（思考），然后决定往前走 50 米（行动），走到之后再看周围有没有餐厅招牌（观察），发现了就进去，没发现就再决定下一步。这种方式的优点是灵活，你不知道这个街区什么布局，边走边看很合理。缺点是可能走回头路，可能在一个区域打转。ReAct 在工程上适合信息不完整的探索型任务，比如"线上接口 500，帮我定位原因"——你不知道是代码 bug、数据库问题、配置变更还是流量突增，只能先查日志（行动），看日志里有什么线索（观察），再决定下一步查什么（思考）。

ReAct 的工程兜底必须包括：**步数上限**（比如最多 20 步）、**无进展检测**（连续 5 步观察结果没有产生新信息就停止）、**成本上限**（token 消耗超阈值就暂停）、**循环检测**（如果连续调用同一个工具且参数相同，强制要求模型解释为什么要重复）。

**Plan-and-Execute：先看地图再出发**。你打开手机地图，看清整个路线：先走到地铁站、坐 3 号线、在人民广场换 2 号线、出站后走 200 米。然后按计划一步步执行。好处是效率高，每到一个节点你都知道接下来干什么；坏处是如果地铁停运了，你的整条路线都得重来。在 Agent 场景里，Plan-and-Execute 适合目标明确、步骤相对稳定的任务。比如"每晚 8 点生成当日销售报表"——拉数据、算指标、画图表、发邮件，这四步很少变化。

但工程上纯粹的 Plan-and-Execute 很危险，因为模型可能生成了一个看起来合理但实际不可行的计划，然后机械地执行到底。所以必须加 **replanning 机制**：每执行完一步，把实际结果和预期结果对比，如果偏差超过阈值，触发重新规划。重新规划时，已完成步骤的成果要保留（比如已获取的数据），但据此推导出的假设要重新评估。这里有一个面试金句：**plan 是假设，execution 是验证，replanning 是纠偏**。

**Reflection：做完之后自我审查**。你把菜做好了，端上桌前自己尝一口——咸了加水、淡了加盐。Reflection 是在模型输出后加一轮自检，让模型评估自己的输出是否满足要求。比如代码生成后，Reflection 检查：代码是否能编译通过？是否满足用户提出的所有约束？是否引入了安全漏洞（SQL 注入、硬编码密钥）？是否需要补充测试？

Reflection 能有效提升质量，但有三个硬伤。第一，**模型可能把正确结果改错**——它自信满满地"优化"了一段正常代码，引入了新的 bug。第二，**模型可能在自己的错误里打转**——它不认识自己真正的错误，反复修改但问题依旧。第三，**Reflection 增加 token 成本**——每次自检都要把原输出和分析要求再送进模型。所以 Reflection 必须和真实的外部验证结合：代码要真正编译、测试要真正跑、SQL 要做语法检查。模型自检是辅助，不是替代真实验证。

**Tree of Thoughts：多想几条路再选**。你不仅看一条路线，而是同时规划三条：打车、地铁、公交。每条路预估时间、费用、舒适度，然后选最优。在 Agent 里，Tree of Thoughts 适合搜索空间大的任务——比如方案设计（微服务拆分有 5 种方案）、数学推理（一道题有多个解题路径）、复杂排障（多个可能根因需要并行排除）。代价是成本高——每个分支都是一次完整的推理链，三条分支就是三倍成本。工程上必须限制分支数量（比如最多 5 个候选）、设置剪枝规则（置信度低于 0.3 的分支直接砍掉）、以及最终由 Judge 或投票机制选出最优解。

### 工程例子：慢查询诊断怎么选范式

咱们走一个实际场景——数据库慢查询诊断 Agent，帮你把四种范式的选择和混合讲清楚。

用户给出一条 SQL："这个查询最近突然从 50ms 变成 5s，帮我看原因。"

**纯 ReAct 的做法**：Agent 先想"可能和执行计划有关"，调用 `explain(sql)`，拿到执行计划后发现走了全表扫描。再想"为什么索引没生效"，调用 `show_indexes(table)`，发现索引还在。再想"是不是统计信息过期了"，调用 `check_stats(table)`，发现确实过期了。整个过程灵活，一步步推进。但 ReAct 的问题可能出现在：它可能遗漏"最近数据量有没有暴涨"这个方向，因为它没在一开始做全局规划。

**纯 Plan-and-Execute 的做法**：先生成完整计划——收集 SQL 文本 → 查看执行计划 → 检查索引状态 → 查看数据分布变化 → 比较历史执行耗时 → 检查统计信息 → 输出诊断报告。然后严格按计划执行。但如果执行到第二步发现"没有这个表的查询权限"，整个后续计划就悬空了。

**混合范式的工程实践——这才是面试要的答案**：第一阶段用 Plan-and-Execute 生成粗粒度的诊断框架（"收集基础信息 → 分析执行计划 → 排查根因 → 生成建议"），第二阶段用 ReAct 执行每个子任务（在"排查根因"里灵活地查索引、统计信息、锁等待、配置参数），第三阶段用 Reflection 做最后的质量检查——建议是否包含回滚方案？是否提示了索引的写入放大？是否给出了验证 SQL？

面试官听到你讲出"粗计划 + ReAct 执行 + Reflection 终检"三层混合，就知道你不是只背过概念。

### 边界和风险：范式越复杂，不等于越可靠

很多工程师有个误区——觉得推理范式越复杂就越好，Tree of Thoughts 一定比 ReAct 高级。这是错的。**范式的复杂度必须和任务的不确定性匹配**。给一个"查天气"的任务上 Tree of Thoughts，除了浪费钱没有任何好处。

各范式的翻车点要记住。ReAct 最怕的是**循环无进展**——模型在不同工具间切换，但每次观察结果都没有新信息。Plan-and-Execute 最怕的是**错误计划被机械执行**——所以 replanning 不是可选项，是必需品。Reflection 最怕的是**自欺欺人**——模型把好的改坏了、把错的放过了，所以必须搭配外部验证。Tree of Thoughts 最怕的是**成本爆炸和选择困难**——分支太多，Judge 挑不出来，成本翻了几倍。

安全上有一个极容易踩的坑：**模型的"思考文本"和"动作指令"要严格分离**。模型在 ReAct 的"思考"阶段可以自由写出"我认为可能需要删除这个文件"，但这个思考不能直接变成 `rm -rf` 的系统调用。思考归思考，动作归动作——工具调用必须经过 schema 校验、权限检查、沙箱隔离和人类确认。产品里通常也不展示完整的内部思维链给用户，而是展示可审计的计划、工具调用记录、观察来源和结论依据。

还有一个判断标准：**如果任务本身就是确定性的，就不要用不确定的推理范式**。固定审批流、固定 ETL、固定报表生成——这些用 Workflow 比 Agent 推理范式更可控、更便宜、更可审计。面试时能主动说出"这个场景不适合用 Agent 推理范式，用 Workflow 更合适"，比把所有范式背一遍要强得多。

## 🎯 面试总结

- **必答3点**：① 推理范式的本质不是"让模型更聪明"，而是给模型外挂一个任务状态机，控制"观察→推理→行动→修正→停止"的时序；② 选范式看三个变量：任务信息是否完整（决定用观察驱动还是计划驱动）、路径是否稳定（决定是否需要 replanning）、结果是否可验证（决定 Reflection 是否有意义）；③ 每种范式都有系统级兜底——步数上限、无进展检测、成本上限、权限隔离缺一不可。

- **加分项**：① 能讲出混合范式：粗计划 + ReAct 执行 + Reflection 终检，且能说出每层的切换条件；② 能说出 ReAct 的无进展检测机制——不是简单数步数，而是判断连续 N 步的观察结果是否产生了新信息；③ 能区分"模型的思考"和"系统的动作"，说明思考文本不能直接变成命令，必须经过工具 schema 和权限层。

- **一句话总结**：推理范式不是让模型自由发挥的舞台，而是给它划定"什么时候想、什么时候做、什么时候改、什么时候停"的行为边界——**好的范式让模型的聪明用在刀刃上，差的范式让模型的聪明变成瞎忙**。

## 🔍 排查和实践建议

排查推理范式问题时，看四样东西：当前目标是什么（模型在这一步到底想解决什么）？模型为什么选这个工具（它的推理依据是什么）？工具观察带来了什么新信息（观察结果和之前有什么不同）？下一步决策是否基于观察（有没有无视工具返回、继续按自己的猜测走）？

如果日志里只有最终的答案，没有中间的计划、工具选择理由、观察结果和推理路径，这个 Agent 基本没办法维护。一个好的 trace 系统应该能让你回放 Agent 的每一步"思考 → 行动 → 观察"循环，就像回放一盘棋一样。

设计时的决策树：先判断任务信息完整度。如果信息完整、流程固定 → 不用 Agent 推理范式，用 Workflow。信息不完整但动作低风险 → ReAct，重点做无进展检测。步骤稳定但可能有异常 → Plan-and-Execute + replanning。结果需要高质量保证 → 加 Reflection，但必须搭配外部验证。只有在方案空间很大、需要比较多个候选路径时，才上 Tree of Thoughts，并严格控制分支数和成本。
