# 大模型从输入文本到输出文本的运行机制

很多线上问题，只有理解"大模型是怎么生成文本的"才能解释。为什么同一个问题多问几次答案不同？为什么流式输出先卡一会儿再开始冒字？为什么 prompt 很长时首 token 特别慢？为什么设置了 max_tokens 后答案被截断？这些都不是玄学，而是 token 化、自回归生成、解码采样和停止条件共同造成的现象。

![输入输出概念总览](https://gitee.com/linkio666/image/raw/main/02-llm-framework/04-llm-io/01-概念总览.png)

---

## 🎭 面试官这样问

**面试官**："你做过大模型应用对吧？我问你，一个用户发来一段话，模型是怎么一步一步吐出答案的？"

**候选人**："就是先 tokenize，然后过 Transformer，最后解码生成。"

**面试官**："那为什么同样的 prompt，有时候返回 3 条有时候返回 5 条？为什么长 prompt 的时候第一个字等半天才出来，后面就快了？你说流式输出，模型是一边想一边输出吗？"

**候选人**："呃……跟采样参数有关系吧？温度设高一点就随机了……"

**面试官**："那 max_tokens 和 context window 有什么区别？用户说答案讲到一半停了，你怎么排查？是 prompt 太长、还是输出预算不够、还是 stop token 被误触发了？"

**候选人**："……"

**面试官**："好，咱们换一个问题。大模型生成是自回归的，每一步只预测下一个 token。那你告诉我——它是什么时候决定整段回答的结构的？是一开始就想好了大纲，还是走一步看一步？"

---

## 💡 简要回答

大模型从输入到输出可以概括为四个步骤：**tokenize → 前向计算 → 解码采样 → 循环生成**。文本先被 tokenizer 转成 token id，再映射成 embedding 向量，经过多层 Transformer 计算得到 hidden state，语言模型头输出词表维度的 logits。logits 经过 temperature、top-k、top-p 等解码策略处理后选出下一个 token，追加到上下文，再预测下一个，直到触发停止条件。

关键认知：**模型不是"想好了再写"，而是每写完一个字才决定下一个字**。它本质上是"下一个 token 预测器"，所有看起来的规划、推理、结构化输出，都是这套逐 token 生成机制在上下文信息的驱动下涌现出来的行为。

---

## 📝 详细解析

### 核心矛盾一：语言是字符串，模型只认 token

模型不能直接处理自然语言字符串。你输入的是"帮我写一首七言绝句"，模型看到的是一个 token id 序列。**token ≠ 中文词 ≠ 英文单词**。它可能是一个汉字、一个词片段、一个空格加单词、一个标点，甚至是一段高频出现的固定搭配。

这意味着什么？几个工程现象你必须知道：

**第一，token 密度不同。** 同样 1000 个字符，中文可能只有 300-500 个 token，英文可能是 200-400 个 token，但代码、JSON、表格可能高达 600-800 个 token——因为空格、缩进、花括号、方括号可能每个都被单独切成一个 token。你的 token 预算在不知不觉间会被这些"看起来不长"的内容吃掉。

**第二，context window 限制的是 token 数，不是字符数。** 面试官问你"prompt 明明才 3000 个字，为什么报超出上下文窗口"，你要能回答：因为 3000 个字里面有大量特殊符号、换行、代码块，tokenize 之后可能变成 9000 个 token，加上历史对话轻松过万。

**第三，tokenizer 的选择直接影响成本。** 同样一段中文，不同模型的 tokenizer 切出来的 token 数可能差 30% 以上。API 按 token 计费，你没算好这一步，成本估算全是错的。

把 token id 查 embedding 表得到向量，向量进入多层 Transformer。最后一层输出的 hidden state 被语言模型头映射到词表大小（比如 32000 或 50000）的 logits 向量。**logits 是"下一个 token 候选"的未归一化分数**，还不是最终答案——好戏才刚开始。

---

### 核心矛盾二：自回归生成的"串行诅咒"

大模型生成的最大特点是什么？**每一步只能预测一个 token，而且必须等上一步生成完才能预测下一步。** 这就是"自回归"的本质——自己的输出变成下一步的输入。

这带来一个根本性的工程矛盾：**模型知道的信息越来越多，但计算必须是串行的。** 你不能并行生成 10 个 token，因为第 3 个 token 是什么，取决于前两个 token 选了谁。

举个例子。假设模型要输出"中国的首都是北京"，它先生成"中"，然后把"中"拼回上下文，预测下一个 token；看到上下文里有"中"，它判断下一个是"国"；拼回去，判断下一个是"的"……直到 EOS（end of sequence）。每一步都是"看完之前自己写了什么，再决定下一个写什么"。这个机制决定了：

- **首 token 延迟和后续 token 延迟是两个完全不同的东西**，排查方式也不同。
- **流式输出不能绕过 prefill**——模型必须先"读完"整个 prompt，才能知道第一个输出 token 该是什么。
- **生成长度越长，总的 decode 时间线性增长**，并发越高，排队越严重。

---

### 底层机制：prefill 和 decode 两个阶段

一次完整的推理过程分为两个阶段，这是面试必考点。

#### 第一阶段：prefill（预填充）

prefill 处理**完整 prompt**。系统提示词、历史对话、用户问题、RAG 检索到的文档片段、工具返回的结果——所有这些都会被拼成一段超长上下文，一次性 tokenize 后送入模型。

模型对**整段 prompt 所有位置**并行做前向计算，在每一层为每个 token 产生 Key 和 Value（存入 KV Cache），最后输出**最后一个位置**的 hidden state，映射到 logits。

这个阶段的特点是：
- **计算量大，但并行度高**：所有 prompt token 的注意力可以一次性算完。
- **决定首 token 延迟**：prompt 越长，prefill 时间越长，用户等第一个字的时间越久。
- **显存占用高**：所有 prompt token 的 KV Cache 在这个阶段被创建。

#### 第二阶段：decode（解码/自回归生成）

decode 阶段是**逐 token 循环生成**。模型根据 prefill 最后输出的 logits，经过解码策略选出一个 token，把它追加到上下文序列末尾，然后**只对新 token 做前向计算**（因为历史的 K/V 已经在缓存里），再输出下一个 logits，再选 token……循环往复。

这个阶段的特点是：
- **单步计算量小**：每次只处理一个新 token + 历史 KV Cache。
- **步骤必须串行**：第 N+1 个 token 等第 N 个 token 生成完才开始。
- **决定吞吐量**：每秒能生成多少个 token（tokens/s）主要由 decode 阶段的效率决定。

![输入输出运行机制](https://gitee.com/linkio666/image/raw/main/02-llm-framework/04-llm-io/02-运行机制.png)

伪代码流程：

```text
prompt_ids = tokenize(prompt)
cache = prefill(prompt_ids)          # 一次性处理全部 prompt，建立 KV Cache
while not stop:
    logits = model.forward(last_token, cache)   # 只对最新 token 前向
    logits = apply_temperature(logits)
    candidates = apply_top_p_or_top_k(logits)
    next_token = sample(candidates)
    append(next_token)
    update_cache(next_token)
return detokenize(generated_ids)
```

注意一个细节：detokenize 是在 token 级别做的，流式输出通常每生成一个或几个 token 就立刻返回给前端，用户看到的是**逐步出现的文本**，而不是等完整答案生成好再一次性显示。这改善的是**用户感知延迟**，不是总计算量。

---

### 解码策略：从 logits 到具体 token 的"最后一公里"

logits 出来以后，怎么变成具体 token？这是大模型应用里最容易出问题也最容易被忽视的一环。

**贪心解码**：每次选 logits 最高的 token。结果是确定性的、可复现的，但容易死板、重复、陷入循环。比如模型说了"我今天很高兴"，贪心可能接着输出"，很高兴"，再输出"，很高兴"……死循环了。

**Temperature**：在 softmax 之前把 logits 除以一个温度值。温度越低（比如 0.1），概率分布越尖锐，高概率 token 被选中的概率接近 1，效果趋近贪心。温度越高（比如 1.5），分布越平坦，低概率 token 也有机会，输出更多样但也更容易胡言乱语。**temperature 不改变模型知道什么，只改变从候选里怎么选**——这是面试金句。

**Top-k**：只在 logits 最高的 k 个候选里采样。k=1 等价贪心，k=50 允许一定多样性。问题是 k 是固定的：一个"确定性强"的上下文和"高度不确定"的上下文用同一个 k 不合适。

**Top-p（nucleus sampling）**：选择累计概率达到 p 的最小候选集合，在这个集合里采样。p=0.9 意思是"只保留概率最高的那一批，直到它们的概率加起来超过 90%"。它的好处是动态调整候选集大小：确定性高时只保留 2-3 个候选，不确定性高时保留几十个候选。

**工程建议**：如果你做代码生成、结构化 JSON、知识库问答，降低 temperature（0.1-0.3），收紧 top-p（0.8-0.9），甚至可以加格式约束。如果你做创意文案、对话闲聊，适当提高 temperature（0.7-1.0）。但永远记住——**采样参数是调味料，不是食材**，知识不足、检索错误、上下文污染，靠调采样只能部分缓解。

### 工程例子：首 token 慢和输出截断怎么定位

这是面试高频场景题。

**场景一：用户说"机器人反应慢"**

你要拆成两段看。如果请求发出后很久才出现第一个字，排查重点依次是：① prompt token 数（RAG 是否塞了太多片段？历史对话是否太长？）→ ② 排队时间（是否有长请求堵在前面？）→ ③ prefill 时间 → ④ 模型冷启动/加载。如果第一个字很快出现，但后面一个字一个字地慢，排查重点依次是：① decode tokens/s → ② 并发 batch 策略 → ③ KV Cache 读写效率 → ④ 显存带宽。

**场景二：用户说"答案讲到一半停了"**

三个常见原因：① `max_tokens` 设太小，输出预算用完；② stop sequence 被命中——比如你设了 `"\n\n"` 做停止符，但模型输出正文里恰好有两个连续换行；③ 上下文窗口被输入占满，留给输出的空间不足。面试时把 **context window（输入+输出总上限）** 和 **max_tokens（本次最多生成多少）** 分清楚，这是区分背概念和真做过的人的分水岭。

**场景三：同一个 prompt 多次调用结果不同**

这是采样在起作用。temperature > 0 时每次采样都有随机性。如果你需要复现结果，设 `temperature=0` 或固定随机种子 `seed`。但注意：temperature=0 不是所有推理引擎都支持，有些平台在 temperature=0 时仍然有浮点精度导致的不确定性。

---

### 边界和风险：几个高频误区

**误区一："token 就是字符数"**。错了。中文、英文、代码、表格的 token 占用差异巨大。一个 emoji 可能占 2-3 个 token。算成本、算截断、算上下文窗口，都按 token。

**误区二："流式输出减少了计算量"**。不是。流式输出改善的是用户感知——不用等全部生成完才看到结果。模型要生成的 token 数一样多，总计算量基本不变。

**误区三："温度低 = 更正确，温度高 = 更聪明"**。不一定。温度低可能在事实性问答上更稳定，但如果模型本身知识有误，低温只会让它更"自信地犯错"。温度高的"创造性"也可能是系统性幻觉的来源。

**误区四："模型是先生成大纲再逐字填充的"**。从底层机制看，模型就是逐 token 预测的，每一步决策都是局部的。虽然模型可以在上下文中展现出规划能力（在 chain-of-thought 中先列出步骤再展开），但底层接口仍然是"每一步预测下一个 token"。这个区别在面试中非常重要。

**误区五："幻觉是温度太高导致的"**。幻觉的根因远比采样策略深。模型每一步都是在训练分布和当前上下文的共同作用下选择 token，没有天然的事实校验器。当 prompt 缺证据、问题有歧义、训练数据有偏差，或者模型面对从未见过的知识边界时，它会"编造"看起来合理但实际错误的内容。RAG 的作用就是把证据放进上下文，缩小模型需要"猜测"的空间，但最终仍要靠检索质量、评测体系和生成约束共同降低风险。

## 🎯 面试总结

### 必答三点

1. **四个步骤一条链**：tokenize → 前向计算 → 解码采样 → 循环生成。文本先转 token → 过 Transformer 得 logits → 用 temperature/top-k/top-p 选 token → 拼回上下文继续生成，直到触发停止条件。

2. **prefill vs decode，两个阶段两个瓶颈**：prefill 处理完整 prompt 建立 KV Cache，决定首 token 延迟；decode 逐 token 循环生成，决定后续吞吐量。排查性能问题必须先区分是哪个阶段慢。

3. **解码策略是"调味料"不是"食材"**：temperature 控制随机性、top-k/top-p 控制候选范围。调参数能改变输出风格和多样性，但解决不了知识缺失、检索错误和上下文污染。

### 加分项

- 能说清楚 token 密度差异（中文/英文/代码/表格的 token 占用不同）
- 能区分 context window 和 max_tokens（一个限制总长，一个限制生成长度）
- 能解释为什么模型看起来有规划能力但底层是逐 token 自回归

### 一句话

**大模型不是一个"想好再写"的作家，而是一个"每写完一个字才决定下一个字怎么写"的接龙高手，所有优雅的推理和规划都是这套机制在超大规模训练数据和上下文信息驱动下涌现出来的。**

## 🔍 排查实践

线上输出不稳定时，**先把能被记录的东西全部记下来**：完整 prompt、模型版本、采样参数（temperature、top-p、top-k）、随机种子、max_tokens、stop sequence、输入 token 数、输出 token 数。别等到复现不了才后悔没留现场。

排查按**三层模型**来做：

**输入层**：prompt 是否缺关键信息？RAG 片段是否相关、是否过期？历史对话是否引入了错误的上下文？system prompt 是否被覆盖？

**解码层**：temperature 是否过高导致抖动？top-p 是否过宽让低质量 token 混入？stop sequence 是否误触发？max_tokens 是否太小？

**服务层**：是否发生了截断、超时、降级或模型版本切换？并发是否过高导致排队超时？是否命中了限流？

性能排查同样分层。**首 token 慢**：查 prompt token 数 → prefill 时间 → 排队 → 冷启动 → 网络。**后续生成慢**：查 decode tokens/s → batch 策略 → KV Cache 效率 → 显存带宽 → 并发调度。

面试里能把**质量、延迟、成本**分别归因，把同一个现象拆到具体的阶段和参数上，比只背"输入到输出流程"更像一个真正上线过大模型应用的人。
