# 向量检索和关键词检索区别

## 🎭 面试官这样问

> 面试官：向量检索和关键词检索有什么区别？RAG 里用哪个更好？

> 你：向量检索更智能，基于语义相似度找相关内容。关键词检索是比较老的 BM25，只能做字面匹配。RAG 里肯定用向量检索，效果好。

> 面试官：那你说说，用户问「E1024 报错怎么解决」，向量检索召回「蓝牙连接失败常见问题」和「蓝牙模块故障排查」，但标题精确包含「E1024」的技术文档没有被召回。为什么会这样？

> 你：因为向量模型对 E1024 这种精确错误码不够敏感，它更关注整体的语义方向……

> 面试官：对。再问你，用户问「我买了 10 天的耳机还能退吗」，文档写的是「签收后 15 天内可申请无理由退货」——关键词重合很少，关键词检索可能漏掉，但向量检索能抓到。现在你还觉得关键词检索没用吗？

> 你：不能，两种检索各有擅长场景。错误码、型号、函数名适合关键词检索；口语化表达、同义改写适合向量检索。应该混合使用。

> 面试官：方向对了。再往深问：混合检索后两路结果怎么融合？BM25 分数和向量相似度分数不在一个量纲，你怎么处理？

这个问题问到点子上了。面试官要的不是「混合检索」四个字，而是你能讲清两路召回各自的原理、优点、盲区和融合策略。

![向量检索与关键词检索适用场景](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/03-RAG/04-vector-vs-keyword/01-概念总览.png)

## 💡 简要回答（面试话术）

> 向量检索和关键词检索不是新旧替代关系，而是互补关系。关键词检索基于词项匹配，看查询词是否出现在文档中，对错误码、产品型号、函数名、法规编号这类精确符号非常强——必须精确命中才有意义。向量检索基于语义相似度，把 query 和文档映射到同一个语义空间，擅长处理同义表达和口语化问题——用户说「还能退吗」，系统能召回「无理由退货」。两者各有盲区：关键词检索对同义改写不敏感，向量检索对精确符号不敏感，还容易召回「语义接近但不能回答问题」的内容。

> 生产系统通常用混合检索：BM25 和向量检索分别召回 top_k 候选，合并去重后交给 reranker 统一精排。融合不能简单把 BM25 分数和向量相似度相加——它们不在一个量纲上。常见做法是用 reranker 重新打分，或者做分数归一化后加权，还可以按问题类型动态调整两路权重：检测到错误码提高 BM25 权重，检测到口语化提高向量权重。

## 📝 详细解析

### 核心矛盾：语义相似 vs 精确命中

两种检索方式看文档的角度完全不同：

**关键词检索（如 BM25）看的是词项。** 查询词是否出现在文档中？出现多少次？这个词在整个语料库里是否稀有？「E1024」这样的错误码很稀有，一旦出现，区分度极高。但「退款」几乎所有文档都有，区分度就低。BM25 不需要在面试里推公式，但要记住它考虑三件事：词项匹配、词频、逆文档频率（IDF）。

**向量检索看的是语义。** 把 query 和文档 chunk 都转成向量，计算向量之间的 cosine 距离。即使字面完全不同——「还能退吗」和「无理由退货」——只要语义接近，向量距离就小。

核心矛盾在这：**关键词检索要求「字面命中」，向量检索只要求「意思接近」。** 字面命中精确但死板，意思接近灵活但容易跑偏。两者像是在用两套完全不同的「相似」定义，在一个系统里协作。

### 底层机制：BM25 和向量相似度各看什么（类比教学）

这两个检索方式，可以类比成**两种找书的方式**：

**关键词检索 = 图书馆的卡片目录。** 你按书名、作者名、ISBN 精确查找。你输入「E1024」，目录系统只会返回书名里包含「E1024」的书。精确、快速、可解释——你看到了命中了哪些词。但如果你输入「怎么能退」，目录里没有「能退」这两个字，你就找不到《无理由退货政策》。字面没对上，就找不到。

**向量检索 = 图书馆里的主题推荐。** 你描述想要什么类型的书，系统推荐一批主题相关的。你说「我想找关于退货的书」，系统会推荐《无理由退货政策》、《退换货流程》、《退款到账时间》——它们都和退货有关。但这里有个陷阱：推荐的《退换货流程》讲的怎么操作流程，而你要问的是「退款多久到账」，推荐的书主题相关但答不上你的问题。

所以总结一下：

| | 关键词检索（BM25） | 向量检索 |
|---|---|---|
| 优势 | 精确符号强、快速、可解释 | 同义表达强、处理口语化 |
| 盲区 | 字面没对上就找不到 | 精确符号不敏感、相似≠相关 |
| 适合场景 | 错误码、型号、函数名、法规号 | 口语化问题、概念解释、同义改写 |
| 典型失败 | 「还能退吗」vs「无理由退货」 | 「E1024」被「蓝牙故障」淹没 |

### 工程例子：为什么生产常用混合检索

单独用关键词检索，用户一句「这玩意退货有啥要求」搜不到「无理由退货政策」——「这玩意」和「耳机」没对上，「有啥要求」和「条件」没对上。

单独用向量检索，用户问「E1024 报错」可能召回一大堆蓝牙相关文档，但那条标题就是「E1024：蓝牙模块初始化失败处理指南」的精确文档却排在后面——因为向量模型把 E1024 这个罕见符号「稀释」到了蓝牙语义里。

混合检索的做法是两条路都走：

```
用户问题
  → 向量检索召回 top_k（语义覆盖）
  → 关键词检索召回 top_k（精确覆盖）
  → 元数据过滤（版本、权限、时效性）
  → 合并去重
  → Rerank 统一精排
  → 取最终上下文
```

用户问「E1024 报错，蓝牙连不上怎么办？」：BM25 召回了标题含 E1024 的文档，向量检索召回了蓝牙连接相关文档。两边合并去重，reranker 重新判断每个候选「是否能回答当前这个问题」，把最相关的前几条塞进 prompt。

### 结果融合：不要直接把分数相加

混合检索最关键的工程难题是融合——BM25 分数和向量相似度不在一个量纲，不能直接相加。如果简单相加，量纲大的一方会主导排序。

**方案一：简单合并去重。** 两路各取 top_k，合并后交给下游 reranker。最简单，但跳过融合直接丢给 reranker，排序压力全在精排上。

**方案二：分数归一化后加权。** 把两路分数分别归一化到相同范围（如 0-1），再按业务权重线性组合。权重要用评测集调：错误码类样本加大 BM25 权重，口语化类样本加大向量权重。

**方案三：Rerank 统一重排。** 这是生产系统最推荐的做法。召回阶段让两路尽量多给候选（比如各 50 条），精排阶段用 reranker 同时看 query 和文档内容，重新打分排序。reranker 不关心候选来自哪一路，只关心「谁能回答问题」。

更进一步，可以**按问题类型动态调权重**。比如用一个轻量分类器检测 query 中是否包含错误码、订单号、函数名、型号——有则提高 BM25 的初始权重和召回数量；检测到口语化、概念解释、自然问句——提高向量召回的权重。

### 边界和风险

关键词检索的盲区是「字面没对上就找不到」。文档写「款项返还」，用户问「多久到账」→ 没命中。解决方案是同义词词典、query 改写、搭配向量检索。

向量检索的盲区是「看起来像但答不上」。用户问「退款多久到账」，召回「如何申请退款」→ 语义接近但答案不相关。解决方案是 reranker 精排和更细粒度的分块。

混合检索也不是 top_k 越大越好。候选越多，rerank 成本越高；最终上下文越多，噪声越大。召回阶段追求覆盖，上下文阶段要控制质量——这是一组新的矛盾。

**元数据问题不容忽视。** 旧政策和新政策可能都包含「退款」，BM25 和向量检索都可能同时召回。纯粹靠分数排序不够，必须依赖版本、生效时间、权限等元数据参与过滤或加权。

## 🎯 面试总结

- **必答 3 点：** ① 关键词检索看词项匹配，向量检索看语义相似——两者互补，不是替代；② 关键词检索擅长错误码、型号、函数名等精确符号，向量检索擅长口语化和同义改写；③ 生产系统常用混合检索 + rerank 精排，两路分数不直接相加。
- **加分项：** 能讲清楚 BM25 的三个核心因素（匹配、词频、IDF）。能讲出三种融合方案及其适用场景。能提动态权重调整和元数据参与排序。
- **一句话总结：** 向量检索和关键词检索就像两个人各看半张地图——一个看路名，一个看地形，合在一起才完整。

## 🔍 排查和实践建议

按失败模式定位问题：
- **精确符号总召回泛文档：** BM25 权重不足，或没有对错误码/型号建字段索引。
- **口语化问题召不到：** 向量召回或 query 改写不足。
- **召回主题相近但答非所问：** 需要 reranker，或分块粒度需要优化。
- **新旧政策混在一起：** 元数据过滤未生效，版本/时间字段没参与排序。

实践上，先给失败样本分类——精确符号类、同义表达类、版本权限类、排序错误类——再针对性地加强对应召回路径。好的检索系统不纠结「向量还是关键词」，而是按问题类型动态组合多种召回方式，让每条路径做自己最擅长的事。
