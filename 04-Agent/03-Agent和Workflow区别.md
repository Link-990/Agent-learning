# Agent 和 Workflow 区别

## 🎭 面试官这样问

> 面试官：你们项目里什么时候用 Agent，什么时候用 Workflow？
> 你：嗯……复杂任务用 Agent，简单任务用 Workflow。
> 面试官："复杂"和"简单"怎么界定？发票报销——OCR 识别、金额校验、预算匹配、审批流转——算复杂还是简单？
> 你：这个流程比较固定，应该用 Workflow。
> 面试官：那如果用户在报销被拒后问"为什么被拒，我该补什么材料"，这个用 Workflow 还是 Agent？
> 你：这个……路径不固定，应该用 Agent。
> 面试官：你看，同一个业务里 Workflow 和 Agent 可能同时存在。它们不是二选一的对立关系，而是控制权的分配问题——**确定性流程交给代码，不确定性探索交给模型**。能讲清楚这个边界，才是面试要的答案。

很多企业项目踩过一个大坑：明明是固定审批流，为了"上 AI"硬改成 Agent 自由决策，结果成本飙升、结果飘忽、审计没法做。反过来也有团队对开放探索场景写了 500 条 if-else 规则，补一条漏三条，没人敢改。所以面试问 Agent 和 Workflow 的区别，面试官真正想听的是：**你如何判断一段业务逻辑该由代码控制还是由模型控制**。

## 💡 简要回答（面试话术）

我理解 Agent 和 Workflow 的核心区别在于**控制权的归属**，而不是任务"简单"还是"复杂"。

Workflow 是开发者提前编排好的确定性流程——每一步的输入、输出、异常分支都在设计阶段就确定了，模型可以作为某个节点的能力来用，但流程的控制权在代码里。比如发票报销的 OCR 识别、金额校验、预算匹配、审批流转，每一步都有明确的规则，同样输入必须产生同样输出，审计要求极高——这天然适合 Workflow。

Agent 则把下一步的决策权交给模型——模型根据目标和中间反馈动态判断该调用哪个工具、需要什么信息、什么时候停止。它像一个动态调度器，适合信息不完整、路径不可预期、需要灵活组合工具的任务。比如"帮我定位这个接口为什么突然变慢"，你无法提前枚举所有可能原因，需要 Agent 根据日志、监控、配置的反馈边走边判断。

从面试角度，最关键的认知是：**二者不是对立关系**。工程上常用 Workflow 承载主链路（确定性流程），用 Agent 处理非标准问题（解释、排查、动态探索）。高风险动作——无论在哪一种架构里——都必须由权限、规则和人工确认兜底，不能让模型自主拍板。

## 📝 详细解析

### 核心矛盾：确定性流程和不确定性探索

咱们用一个简单的类比来建立直觉。Workflow 像一个地铁线路图——从 A 站到 B 站，线路固定，换乘站固定，你按图走就一定能到。Agent 像一个老司机在陌生的郊区开车——不知道前面是堵车还是封路，需要根据路况实时决定左转还是右转。

两者的根本差异来自**任务不确定性**。如果任务的目标、输入、步骤、异常都能在设计阶段穷举，那 Workflow 是更优解——稳定、可控、成本低、好审计。如果任务的信息不完整、路径无法预知、需要动态组合工具来探索，Agent 才有价值。

很多团队在这个问题上摔倒，是因为用错了判断标准。他们问"这个任务复杂吗"——复杂用 Agent，简单用 Workflow。但真正该问的是"**这个任务的每一步，我能在设计阶段完整定义吗？**"发票报销很"复杂"——涉及 OCR、规则匹配、金额校验、预算管理、多级审批——但每一步都能在设计阶段定义清楚，所以它是 Workflow 的好场景。而"用户问为什么被拒"这个任务"简单"——就是解释一个原因——但用户的问法多变、需要查询的数据不固定、解释的逻辑也因人而异，所以反而适合 Agent。

### 底层机制：控制流在哪里

把两个架构拆开看，区别就一目了然了。

**Workflow 的控制流在代码里**。以客服质检流程为例：上传录音 → 语音转文本 → 抽取关键词 → 判断情绪（这一步可以调用模型）→ 匹配违规规则 → 生成质检报告 → 人工复核。模型在"判断情绪"和"生成报告"两个节点提供了智能能力，但流程本身——是否进入复核、是否触发处罚、是否升级投诉——由规则引擎或业务代码决定。模型只是一个聪明的节点，不是流程的决策者。

**Agent 的控制流在运行时产生**。模型不仅生成文本，还会决定调用哪个工具、下一步查什么、是否需要追问用户、什么时候任务已经完成。以排障 Agent 为例：拿到的唯一输入是"接口响应变慢了"，它的每一步决策都依赖上一步工具返回的结果——先查监控发现 CPU 正常，排除了一个方向；再查慢查询日志发现某条 SQL 扫描行数暴增，聚焦到数据库；再查执行计划发现索引失效——这条路径不是预设的，是 Agent 自己在运行时"走"出来的。

测试和可观测性也不同。Workflow 可以为每个节点写单元测试：输入 A，期待输出 B；异常输入 C，期待走到降级分支 D。Agent 则依赖轨迹评测：给一批真实任务，看 Agent 的完整执行路径是否合理——工具选择是否准确、是否出现循环、是否需要人工接管、最终结果是否满足用户需求。一个看节点级正确性，一个看端到端合理性。

### 工程例子：发票报销助手怎么拆

拿发票报销助手来串联——这是面试中最经典的例子，能讲清楚就能拿分。

整个报销业务需要拆成两部分。**主体流程走 Workflow**：用户上传发票 → OCR 识别发票信息（金额、日期、税号、商品）→ 校验发票真伪和重复报销 → 匹配预算科目和部门额度 → 进入审批流（按金额分级：500 以下直属领导、500-5000 部门负责人、5000 以上财务总监）→ 审批通过后记账。这个链路里的每一步都是确定性的——OCR 的字段是固定的、校验规则是明确的、审批流是按金额枚举的。同样一张发票，今天提交和明天提交，走同样的 Workflow 应该得到同样的结果。

**非标问题走 Agent**：用户收到驳回通知后问"为什么被拒，我要补什么材料？"这个任务没法提前写死流程，因为用户可能问的是"金额不对"（需要查 OCR 结果和发票图片）、可能问的是"预算不够"（需要查部门预算余额和月初分配）、也可能问的是"税号错了"（需要查企业信息库）。Agent 拿到这个问题后，先判断需要查哪些信息，再调用对应工具，把结果组织成用户可以理解的解释和建议。路径是动态的。

同一个产品里，Workflow 和 Agent 可以同时存在——Workflow 做骨架，承载确定性流程；Agent 做血肉，处理非标准和解释性任务。面试时讲出"**用 Workflow 承载确定性，用 Agent 处理不确定性**"，面试官就知道你有工程判断力。

### 边界和风险：什么时候不用 Agent

这个问题面试官一定会追问，因为答不好就暴露了你在"为了技术用技术"。

**强规则场景不要用 Agent**。审批、转账、删库、发公告、改权限、提交生产配置——这些动作的决策依据必须在设计阶段就明确，不能交给模型在运行时"灵活判断"。模型可以提供建议（"基于历史数据，建议批准这个报销"），但最终执行必须经过规则引擎校验、权限系统控制和人类确认节点。面试金句：**模型负责建议，规则负责拍板**。

**开放探索场景也不要硬写 Workflow**。"帮我定位性能下降原因"——如果你试图用 if-else 覆盖所有可能：查代码变更、查数据库索引、查缓存命中率、查流量峰值、查依赖服务、查配置变更……每遇到一个新 case 就加一条规则，系统会越来越脆，规则之间还会互相冲突。这种场景就交给 Agent 探索，但要配好兜底：最大步数、成本上限、人工接管按钮。

**混合架构里最隐蔽的坑**：开发者以为用了 Workflow 就安全，但实际上把关键判断交给了模型自由输出。比如一个报销 Workflow 第 5 步是"模型判断是否合规"，模型输出"合规"两个字，代码就直接触发付款——这跟让模型自由决策没有区别，只是外面套了个 Workflow 的壳。正确做法是让模型输出结构化字段（违规类型、违规条款、置信度），Workflow 拿到这些字段后，用规则引擎二次校验，再决定是放行还是送人工复核。

还有一个行业共识值得记住：**Agent 的上限是灵活，下限是不稳定；Workflow 的上限是固化，下限是可靠**。选型的本质是在"灵活但可能出错"和"可靠但不够灵活"之间找到平衡点。没有谁比谁高级，只有谁比谁合适。

## 🎯 面试总结

- **必答3点**：① 核心区别是控制权归属——Workflow 的控制流在设计阶段由代码确定，Agent 的控制流在运行时由模型动态产生；② 选型标准不是任务"复杂"还是"简单"，而是"能否在设计阶段穷举所有步骤和异常分支"——能穷举的走 Workflow，不能穷举的考虑 Agent；③ 二者可以共存，Workflow 做主链路，Agent 做非标准任务，但高风险动作必须回到规则和人工确认的框架里。

- **加分项**：① 能讲出混合架构里最隐蔽的坑——"模型输出自由文本，代码直接当指令执行"，并给出结构化输出 + 规则二次校验的正确方案；② 能把测试和可观测性的差异讲清楚——Workflow 看节点级正确性，Agent 看轨迹级合理性；③ 能主动说出 Agent 不适合的场景（强规则、高风险、需审计），展现工程克制力。

- **一句话总结**：**用 Workflow 承载确定性，用 Agent 处理不确定性，用规则和人工兜底两者共有的高风险地带**。

## 🔍 排查和实践建议

做方案设计时，先画任务状态机。如果你能在白板上把所有状态、转移条件、输入输出和异常分支枚举清楚——那就用 Workflow。如果画着画着发现"这里可能有很多种情况、那里要看具体数据才能判断"——那这个子任务适合 Agent。

排查线上问题时也按这个思路分层。Workflow 的问题看节点：哪个节点超时了？哪个节点的规则没有命中？补偿任务有没有执行？Agent 的问题看轨迹：目标抽取是否准确、计划是否合理、工具选择是否正确、观察结果是否被有效利用、停止条件是否合理触发。定位问题时先判断出问题的环节是在确定性层还是动态决策层，排查效率会高很多。

设计 Agent 和 Workflow 的衔接点时，关键是信息传递的格式。Workflow 交给 Agent 的数据应该是结构化的——订单 ID、状态、时间戳、错误码，而不是一段自然语言描述。Agent 交回 Workflow 的结果也应该是结构化的——决策类型、决策依据、置信度、建议动作——而不是"建议通过"四个字。
