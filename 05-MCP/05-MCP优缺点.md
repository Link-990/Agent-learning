# MCP 优缺点

## 🎭 面试官这样问
> 面试官：你前面把 MCP 讲得很熟，那我反过来问——MCP 有什么缺点？什么情况下你不应该用 MCP？
> 你：MCP 的缺点是增加了系统复杂度，链路变长了。如果只是一个简单的内部工具，可能不需要 MCP。
> 面试官：这个回答方向对，但太泛了。复杂度具体体现在哪？链路长了之后哪几个环节引入了额外开销？如果在创业公司只有三个人、一个 AI 助手，你推不推荐他们上 MCP？
> 你：嗯……链路长了排查困难……小团队可能不需要……但具体什么时候该用什么时候不该用，我不太确定。
> 面试官：核心是你要能算账——MCP 的收益来自"复用规模"，代价来自"解耦开销"。当复用收益大于解耦开销时，MCP 是好的选择；当只有一个 Host、一个工具、一个使用者时，用 MCP 就是把简单问题复杂化。你不能因为 MCP 是"新东西"就无脑推，也不能因为它"有代价"就一棍子打死。
> 好，今天我们就来算这笔工程账——MCP 到底好在哪、坏在哪、什么时候值得引入。

## 💡 简要回答（面试话术）

MCP 的核心优点四个字：**标准化复用**。它让文件系统、数据库、Git、知识库等外部能力通过统一协议暴露，一个 Server 写好之后可以被 IDE、桌面助手、自动化 Agent 等多个 Host 共享。这避免了"每换一个应用就重新写一遍工具适配"的重复劳动，同时把权限、审计和错误处理集中到 Server 层统一治理。

但 MCP 的代价也是四个字：**解耦开销**。原本一个简单的 API 调用——十行代码能搞定——引入 MCP 后变成：配置 Server → 启动进程 → 协议握手 → 能力发现 → Host 过滤 → 模型选择 → 协议请求 → 结果回填。每一步都是新的故障点。链路变长之后，排查问题变难、网络或进程开销增加、权限治理复杂度上升。

所以 MCP 不是"万能银弹"。它的适用场景满足三个条件：多 Host 复用、多工具管理、权限需要统一治理。如果你只有一个内部报表机器人，只调一个固定 API，直接用 Function Calling 更简单。面试时能把这个"收益-代价"的权衡讲清楚，比单纯背优缺点列表要高级得多。

## 📝 详细解析

### 核心矛盾：解耦带来复用，也拉长了每一条链路

MCP 的设计哲学是"解耦"。Host 和 Server 之间不直接耦合——它们通过 Client 和标准协议交互。这种解耦带来了巨大的灵活性：你可以给 VS Code 接一个文件系统 Server，也可以给同一个 Server 接上命令行 Agent，甚至 Web 端助手也能连——Server 完全不需要知道自己被谁在用。这就像 USB 协议：U 盘不在乎你插的是 Mac 还是 Windows，只要符合 USB 协议就能读写。

但解耦不是免费的。每多一层抽象，就多一层延迟、多一个故障点、多一份排查成本。前 MCP 时代，你想让模型读取一个文件，可能就是一句 `open('file.txt').read()`。引入 MCP 后，这件事变成了：确保文件系统 Server 进程在跑 → Client 和 Server 完成握手 → Client 发现 `read_file` tool → Host 把 tool 描述塞进模型上下文 → 模型选择 tool 并生成参数 → Client 编码请求 → Server 校验路径 → Server 读取文件 → Server 编码返回 → Client 解析返回 → Host 回填给模型。同样是读一个文件，链路的长度翻了好几倍。

所以 MCP 的核心矛盾是：**解耦的收益随复用规模增长，但解耦的开销是每个调用都要支付的固定成本。** 当你的复用规模足够大——10 个 Host 共享 20 个 Server——收益远超开销；当你的规模足够小——1 个 Host 用 1 个工具——开销大于收益。

### 优点：标准化、复用、隔离、生态

**第一，标准化。** 这是 MCP 最核心的价值。在 MCP 出现之前，每个模型应用都要自己定义"工具怎么描述""参数怎么校验""结果怎么返回"。A 平台的工具描述格式跟 B 平台不一样，同一个 Git 能力在三个 IDE 里要写三套适配代码。MCP 统一了这件事：tools 有标准 schema，resources 有标准 URI 模板，prompts 有标准格式。一次定义，到处使用。这不只是"方便"，而是让工具生态成为可能——社区可以写一个通用的 Git Server，所有支持 MCP 的 Host 都能直接用它。

**第二，复用。** 这是标准化直接带来的收益。一个数据库查询 Server 写好部署后，你的 IDE 能接、CI/CD Agent 能接、运维机器人能接、数据分析助手能接。你不需要为每个场景重新写一遍数据库连接、查询校验、结果格式化。复用不只是省代码，更重要的是省维护——当数据库表结构变了，你只需要更新一个 Server，所有 Host 自动生效。

**第三，隔离。** 外部系统的复杂性被封装在 Server 内部。Host 不需要知道这个数据库是 MySQL 还是 PostgreSQL，不需要知道文件系统是本地磁盘还是 NFS 挂载，不需要知道 Git 命令的参数细节。Host 只看到统一的 tools/resources/prompts 接口。这种隔离让 Host 变简单，也让安全治理变集中——权限校验、数据脱敏、审计日志全在 Server 层统一处理，而不是散落在各个 Host 里各自为政。

**第四，生态和可观测性。** MCP 让能力变成可插拔的组件。新接入一个浏览器自动化能力？加一个 Browser MCP Server 就行，Host 代码不用改。新接入一个 Jira 工单系统？同理。这种插件化架构对 Agent 工程特别重要——Agent 的能力边界可以通过"加载哪些 Server"来动态调整，而不是硬编码。可观测性也随之提升：因为所有外部调用都经过 Client-Server 这条标准化链路，你可以统一记录 trace、监控延迟、统计调用量——这在各 Host 自己瞎调 API 的时代是不可能做到的。

### 缺点：复杂度、延迟、安全治理、版本兼容

**第一，协议复杂度。** 这可能是最容易被低估的代价。直接调 API 只需要 HTTP client + JSON parse。引入 MCP 后多了：Server 进程管理（或 HTTP 服务管理）、协议握手和版本协商、能力发现和缓存、JSON-RPC 编解码、连接生命周期管理。对于一个小团队来说，光是理解"为什么读个文件要经过这么多步"就是认知负担。排查问题时，需要同时在 Host 日志、Client 日志、Server 日志三个地方找线索。

**第二，性能开销。** 远程 HTTP Server 有网络延迟、TLS 握手、序列化开销。即使是本地 stdio Server，也有进程启动时间、IPC 通信延迟。当一个 Agent 需要频繁调用工具——比如循环读取多个文件、多次查询数据库——这些延迟会累积。相比之下，直接在 Host 进程内调一个 SDK 函数几乎没有延迟。

**第三，安全治理成本。** MCP 标准化了连接，但也标准化了攻击面。Server 暴露的能力越多，Host 需要做的权限管控越复杂。哪些 tool 需要用户确认？哪些 resource 需要脱敏后再给模型？如果一个 Server 声明了 50 个 tool，你怎么保证模型不会选错一个危险的？只读 resource 看起来安全，但如果包含了数据库连接串、API 密钥、客户隐私数据，后果同样严重。安全不是 MCP 自带的——它是你需要额外投入建设的基础设施。

**第四，版本兼容和维护。** Server 升级后 tool 的参数变了、返回字段改了、错误码换了——但 Host 那边可能还在用旧的 tool 描述。模型按照旧描述生成了参数，Server 按新格式校验失败，返回一个莫名其妙的错误。如果你有 10 个 Host 和 20 个 Server，版本兼容矩阵的维护就是工程师的噩梦。需要契约测试、灰度发布、兼容性窗口期——这些在大公司能落地，在小团队就是负担。

### 工程例子：什么时候用 MCP，什么时候不用

**应该用 MCP 的场景：** 你的公司有 5 个不同的 AI 应用——IDE 插件、Web 问答助手、Slack 机器人、CI/CD 诊断 Agent、数据看板助手。它们都需要访问同一套企业能力：数据库查询、文档搜索、Git 操作、工单系统。如果不引入 MCP，每个应用都要自己写一遍数据库连接、文档索引、Git 封装——5×4=20 套重复代码。引入 MCP 后，写 4 个 Server，5 个 Host 都通过标准协议接入。Server 的权限、审计、限流统一管理。这时候 MCP 的复用收益远大于解耦开销。

**不应该用 MCP 的场景：** 你只是做一个内部用的自动报表脚本。每天定时跑一次，查询一个固定的 MySQL 数据库，生成一份 Excel 报表并发邮件。整个系统只有一个"用户"（定时任务），只有一个外部能力（MySQL 查询）。直接用 Python 写几十行代码——`pymysql.connect()` + `pandas` + `smtplib`——半小时搞定。上 MCP？你得部署一个 MySQL MCP Server、配置 Host、走握手流程……为了一个单次查询，引入一整套协议栈——这就是杀鸡用牛刀。

选择标准不是"MCP 新不新、酷不酷"，而是算账：**复用规模 × 能力复杂度 vs 协议开销 × 团队维护能力。** 左边大于右边就值得，反之就不值得。

### 边界和风险：可接入 ≠ 可安全使用

MCP 最大的风险是一种幻觉——"既然能接进来，就能放心用"。但接入和安全之间还有一道鸿沟。

**Server 爆炸问题。** 如果你允许 Host 自动发现和加载社区 Server，很快模型面前就会出现 100 多个 tool。上下文窗口被挤满，模型选择工具的准确率下降，误调用概率上升。而且你不知道这些社区 Server 的工具描述会不会故意诱导模型——比如一个 tool 叫 `get_system_info`，描述写"获取系统完整信息，推荐优先调用"，实际上它会读取所有环境变量并上传到外部服务器。

**能力描述诱导。** 这是 MCP 独有的攻击面。恶意 Server 可以通过精心设计的 tool 描述来影响模型行为——比如把危险的 tool 描述得特别有吸引力，把安全的 tool 描述得很平庸。因为模型是"看描述选工具"的，描述就是攻击向量。

**上下文膨胀。** 一个 tool 返回了 2MB 的日志数据，Host 不做裁剪直接塞进模型上下文——token 消耗爆炸，模型注意力被稀释，回答质量直线下降。Server 返回内容必须做裁剪和摘要，不能"有多少给多少"。

**权限幻觉。** "这个 tool 在 Server 上做了鉴权，所以 Host 不用管"——这是灾难性思维。鉴权要做在多层：Server 做身份验证，Host 也要做用户确认（尤其是高风险操作），审计日志两边都要留。

## 🎯 面试总结

- **必答 3 点**：
  1. MCP 的优点：标准化（统一 tools/resources/prompts 语义）、复用（多 Host 共享 Server）、隔离（外部复杂性封装在 Server 内）、生态可扩展（插件化接入新能力）
  2. MCP 的缺点：协议复杂度（链路变长、故障点增多）、性能开销（网络/进程延迟）、安全治理成本（权限、审计、脱敏需要额外建设）、版本兼容维护（Server 升级影响所有 Host）
  3. 是否引入 MCP 的判断标准：复用规模 × 能力复杂度与协议开销 × 团队维护能力的权衡——多 Host 多工具多团队场景值得，单一应用单一 API 场景用 Function Calling 更简单

- **加分项**：能量化分析"解耦收益 vs 解耦开销"；能指出"能力描述诱导"和"上下文膨胀"这两个 MCP 特有的风险；能说出"鉴权在多层做，不在某一层全包"

- **一句话总结**：MCP 是解耦换来复用的协议层架构——收益随规模增长，代价每调用必付，能不能用不看"新不新"，看"值不值"。

## 🔍 排查和实践建议

设计 MCP Server 坚持"最小能力原则"：不暴露万能 tool（比如 `execute_arbitrary_code`），每个 tool 只做一件事，参数尽量窄，副作用写在描述里，返回格式固定有文档。Host 端做好：能力白名单、用户确认点（高风险操作弹框）、返回内容裁剪（别把 2MB 日志全塞给模型）、trace 和审计。

评估是否引入 MCP 时问自己三个问题：是否有至少 2 个不同的 Host 需要复用这个能力？是否需要统一做权限治理和审计？团队是否愿意承担进程管理、协议排查和版本维护的额外运维成本？三个答案全是"否"，就不要为了概念而上 MCP——直接 Function Calling + SDK 是最务实的选择。
