# MCP 通信机制

## 🎭 面试官这样问
> 面试官：你刚才说 MCP 有 stdio 和 HTTP 两种通信方式，那你讲讲它们分别适合什么场景？
> 你：stdio 适合本地，HTTP 适合远程。
> 面试官：太表面了。我问你——stdio 模式下 Server 崩溃了怎么办？HTTP 模式下多个 Host 同时请求同一个 Server，你怎么做隔离？还有，MCP 协议消息本身是统一的，那通信层只是换个运输方式而已吗？
> 你：嗯……stdio 崩溃了 Host 应该能检测到退出码……HTTP 隔离可以用 session……
> 面试官：你已经在猜了。核心问题是：MCP 的协议语义（初始化、能力发现、tools 调用、resources 读取）是不变的，但 stdio 和 HTTP 这两种"运输方式"在进程生命周期、并发模型、安全边界和故障处理上有本质差异。你不能把 stdio 当成"简配版 HTTP"，也不能把 HTTP 当成"远程版 stdio"。它们是两种完全不同的工程范式。
> 好，我们就把通信层掰开揉碎讲清楚，让你面试时不会被追问到"网络抖动了怎么办"就卡壳。

## 💡 简要回答（面试话术）

MCP 的通信机制解决的是"协议消息怎么在 Client 和 Server 之间搬运"的问题。上层语义——初始化握手、能力发现、工具调用、资源读取——是统一的，但传输层有两种主流方式：stdio 和 HTTP。选错了通信方式，协议再标准也会出问题。

你可以把 MCP 协议想象成"信件内容"，stdio 是"面对面递纸条"，HTTP 是"快递邮寄"。信的内容格式一样，但递纸条不需要邮局、不需要贴邮票、不用担心快递员送错门——不过递纸条的人走了你就联系不上了。快递邮寄可以跨城市、可以多人同时寄、可以追踪物流——但需要地址、包装、签收流程。stdio 和 HTTP 的区别就是这个道理。

具体来说，stdio 适合本机单进程场景：Host 启动一个子进程作为 Server，通过标准输入输出传输 JSON-RPC 消息。它的优势是零网络开销、部署极简、进程生命周期跟 Host 绑定。HTTP（尤其是流式 HTTP/SSE）适合远程共享服务：Server 独立部署，多个 Host 可以同时连接，方便集中鉴权、限流和审计。代价是需要处理认证、TLS、网络重试、跨租户隔离和长连接管理。面试中不要只说"stdio 本地、HTTP 远程"，要能展开说进程管理、日志隔离、超时取消这些工程细节。

## 📝 详细解析

### 核心矛盾：协议语义统一，但运行环境天差地别

同一个 MCP Server——比如一个文件系统 Server——在本地 IDE 里通过 stdio 跑得好好的，部署到远程 Agent 平台后改用 HTTP，结果频繁超时、间歇性连不上。排查发现，本地 stdio 模式下 Host 和 Server 是父子进程关系，Server 挂了 Host 立刻知道（通过 SIGCHLD 或退出码）；但 HTTP 模式下 Server 是独立服务，网络抖动、负载均衡、连接池耗尽都可能让 Client 收不到响应。

这就是核心矛盾：MCP 在设计上把协议语义和传输层解耦了，但不同的传输方式有着截然不同的工程假设。stdio 假设了"一对一、同机器、共生命周期"；HTTP 假设了"多对一、可跨机器、独立生命周期"。你不能用 stdio 的思路去管 HTTP Server，也不能用 HTTP 的复杂度去要求本地 stdio。

理解这个矛盾的关键是认识到：通信层不只是"换一种运输方式"，它决定了进程模型、故障模型和安全模型。面试官追问通信机制，实质上是在考你的分布式系统基本功。

### 底层机制：stdio 和 HTTP 的工程本质

**stdio 模式——"贴身管家"模型。**

在 stdio 模式下，Host 直接 fork 一个子进程作为 MCP Server。Client 通过子进程的 stdin 写入 JSON-RPC 请求，从 stdout 读取 JSON-RPC 响应。这个模型的工程特点非常鲜明：

第一，部署零依赖。不需要配置端口、不需要 DNS、不需要 TLS 证书。Host 只要能执行一条命令，Server 就能起来。对于本地工具——文件系统、Git、本地数据库代理——这是最简单的方式。

第二，生命周期绑定。Host 退出时，子进程自动被回收。Server 不需要考虑"客户端断连了我还要不要继续跑"——父进程没了，子进程自然结束。反过来，Server 崩溃了，Host 也能立刻感知到 stdout 关闭。

第三，天然的进程隔离。每个 Host 启动自己的 Server 实例，不存在多租户问题。你的文件系统 Server 和别人的文件系统 Server 是独立进程，互不干扰。

但 stdio 有一个让人抓狂的坑：**stdout 污染**。Server 如果把普通日志、debug 信息也输出到 stdout，Client 会把这些乱七八糟的文字当成协议消息来解析——JSON-RPC 解析器看到一行 "Connecting to database..." 直接报错。所以 MCP 规范明确要求：协议消息走 stdout，日志走 stderr。这个坑踩过的人不在少数，面试中提一句"stderr 放日志"是很好的加分项。

**HTTP 模式——"服务中心"模型。**

HTTP 模式下，Server 是一个独立运行的服务进程，监听某个端口。Client 通过 HTTP 请求发送 JSON-RPC 消息。这个模型适合需要共享、集中管理的场景：

第一，多 Host 复用。一个数据库查询 Server 部署后，整个团队的 IDE、命令行工具、自动化 Agent 都可以连上来。不需要每人本地起一个数据库代理。

第二，集中治理。鉴权、限流、审计、版本管理全可以在 Server 端统一做。哪个 Host 在什么时候调了什么工具、消耗了多少资源，一目了然。

第三，独立运维。Server 可以独立扩缩容、滚动升级、监控告警。它不跟任何 Host 绑定生命周期。

代价也很明确：网络延迟、TLS 握手开销、连接池管理、重试和熔断、跨租户数据隔离。特别是长任务——如果一个 SQL 查询要跑 30 秒，HTTP 连接不能一直 hang 着。需要用流式返回（SSE 或 WebSocket）、心跳保活、超时取消机制来处理。

还有一个要特别注意的：HTTP Server 不能靠"知道 URL 就是授权"。必须做身份认证——JWT、OAuth、API Key 都行——并且按身份做能力分级。用户 A 只能查自己的订单，用户 B 可以查全公司订单但不能改——这些权限逻辑要落在 Server 层。

### 工程例子：文件系统 vs 企业知识库，两种选择两种命运

**文件系统 MCP Server → stdio。**

它的核心能力是读取、搜索、列出用户本机的项目文件。这个场景有三个特点：数据在本机、单用户使用、生命周期跟开发会话绑定。用 stdio 是最佳选择。Host 启动子进程时可以指定工作目录（比如 `/path/to/project`），Server 只允许访问这个目录及其子目录，超出范围一律拒绝。用户关闭 IDE 时，Server 子进程自动退出，不留残留进程。不需要网络配置、不需要担心其他人访问你的代码。

**企业知识库 MCP Server → HTTP。**

它的核心能力是搜索公司内部文档、Wiki、设计稿。这个场景有三个完全不同的特点：数据在服务端、多用户共享、需要按身份鉴权。用 HTTP + 集中部署是最佳选择。文档索引在服务器上，所有员工通过不同的 Host（IDE、Web 助手、移动端）都能查询。Server 根据用户的身份令牌返回不同范围的结果——实习生只能看公开文档，架构师能看全部技术设计。所有查询都有审计日志，安全合规。

这两个 Server 对模型暴露的 tool 可能长得差不多——都是 `search`、`read_resource`——但通信机制、运维模式和安全边界完全是两个世界。面试时能把这个对比讲出来，面试官就知道你是真的做过工程。

### 边界和风险：通信失败不能让 Agent 卡死

**stdio 的常见坑**：Server 启动命令写错——`npx` 没装、Python 版本不对、工作目录不存在。环境变量泄漏——Host 的环境变量会继承给子进程，如果里面有 AWS_ACCESS_KEY、DATABASE_URL 等敏感信息，Server 就能拿到。stdout 被日志污染——上面说过了。子进程僵死——Server 卡住了但不退出，Host 一直在等响应。

**HTTP 的常见坑**：认证 token 过期后 Client 没有自动刷新。网络分区导致连接断开，但没有重试机制。Server 限流后 Client 直接报错给模型，模型以为"功能不可用"。跨租户数据泄露——Server 根据某个 header 判断用户身份，但这个 header 可以被伪造。重放攻击——同一个请求被恶意重放多次，造成重复操作。

**通信层的通用风险**：超时和取消。模型不应该因为一个 Server 响应慢就被卡住 60 秒。每次工具调用都应该有 deadline——比如 30 秒。如果超时，Server 要返回一个结构化的超时错误，让模型知道"这个操作太慢了，要不要换个方式？"取消也是必须的——用户在 Host 上点了"停止生成"，对应的 MCP 请求也应该被取消，不能继续占用 Server 资源。

## 🎯 面试总结

- **必答 3 点**：
  1. MCP 协议语义统一，传输层分 stdio 和 HTTP 两种；stdio 适合本地子进程（部署简单、生命周期绑定、注意 stdout 污染），HTTP 适合远程共享服务（多 Host 复用、集中治理、注意认证和超时）
  2. 选通信方式的核心判断依据：数据在哪（本地 vs 服务端）、谁在用（单用户 vs 多用户）、怎么管（进程级 vs 服务级）
  3. 通信层必须处理超时、取消、重试和结构化错误，不能让一个慢 Server 卡住整个 Host

- **加分项**：能说出"stderr 放日志，stdout 放协议消息"这个实战细节；能对比两种模式下的进程生命周期和故障模型差异；能讲清楚 HTTP 模式下为什么要做身份认证而不能靠"知道 URL"

- **一句话总结**：MCP 通信层不是"换种运输方式"那么简单——stdio 是贴身管家，HTTP 是服务中心，选错了通信方式，协议再标准也会翻车。

## 🔍 排查和实践建议

排查 stdio 问题：先手动执行 Server 启动命令（不通过 Host），看能不能独立跑起来；检查 stdout 输出的是不是纯 JSON-RPC（有没有日志混进去）；检查 stderr 有没有报错；确认工作目录和环境变量是否正确。排查 HTTP 问题：先 curl 一下 Server 的健康检查端点；确认认证 token 有没有过期；看状态码是 401（没权限）还是 500（Server 内部错误）还是 504（超时）；查 Server 端的 trace_id 定位具体调用。

设计原则：本地私有能力优先 stdio，跨团队共享能力优先 HTTP。不管你选哪种，给每个请求设置 deadline、支持取消、返回结构化错误、记录审计日志——这四条是通信层的底线。
