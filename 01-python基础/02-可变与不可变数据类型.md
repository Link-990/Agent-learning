# 可变数据类型与不可变数据类型

> 很多人背"int 不可变，list 可变"背得滚瓜烂熟，但一被问到"函数传参是传值还是传引用"就卡壳。这道题真正考的是你对 Python 变量模型的底层理解。

---

## 🎭 面试官这样问

**面试官**："先来个简单的。Python 里哪些类型可变，哪些不可变？"

**很多同学**："int、float、str、tuple、frozenset 不可变；list、dict、set 可变。"

**面试官**："行。那你看这段代码，输出什么？"

```python
def add_item(item, box=[]):
    box.append(item)
    return box

print(add_item("a"))
print(add_item("b"))
```

**很多同学**："第一次输出 `['a']`，第二次输出 `['b']`。"

**面试官**："你确定？你在你电脑上跑一下。"

**很多同学**："等等……第二次是 `['a', 'b']`？为什么？这跟前面说的可变不可变有什么关系？"

这就是经典的"默认参数陷阱"——面试官用它检验你是否真懂可变对象的生命周期。

---

**面试官继续**："那我再换一个。Python 函数传参到底是传值还是传引用？"

**很多同学**："传引用！你看 list 传进去改了，外面也会变。"

**面试官**："那你再看这段："

```python
def reset(data):
    data = []

items = [1, 2, 3]
reset(items)
print(items)
```

**很多同学**："输出 `[]`……吧？"

**面试官**："跑一下。"

**很多同学**："啊？输出 `[1, 2, 3]`？怎么又没变？"

---

**面试官继续追问**："那 `tuple` 是不是一定能做 dict 的 key？"

**很多同学**："能！tuple 不可变，可以做 key。"

**面试官**："那如果 tuple 里面放了一个 list 呢？"

**很多同学**："……那就不行了。"

**面试官**："为什么？tuple 不是不可变吗？"

你看，"可变和不可变"这五个字背后，藏着一整套变量绑定和对象模型的知识。下面我们彻底讲透。

---

## 💡 简要回答（面试话术，可以直接背）

> "Python 中变量不存数据，存的是**对象引用**。可变对象（list、dict、set）可以原地修改内容，不可变对象（int、str、tuple 等）创建后内容不能变——修改操作实际上会**创建新对象**并重新绑定变量。
>
> 函数传参时，形参拿到的是**实参对象引用的副本**，所以形参和外部变量最初指向同一个对象。如果函数里对形参**重新赋值**（`data = []`），只是让局部名字指向新对象，不影响外部；如果**原地修改**可变对象（`data.append()`、`data.clear()`），外部也会看到变化。这不是传值也不是传引用，而是'**传对象引用**'。
>
> 默认参数用可变对象是大忌，因为默认值在**函数定义时只创建一次**，后续调用共用同一个对象。至于 tuple 做 key——tuple 本身不可变，但如果内部装了可变对象，整体就不可哈希。"

---

## 📝 详细解析

### 一、核心矛盾：变量不是盒子，是指针

**类比教学**：很多人学 Python 时，把变量想象成一个"盒子"，里面装着值。这个模型在简单场景下能应付，但一到函数传参、共享引用时就崩了。

正确的模型是：**变量是贴在对象上的便签**。一个对象可以贴很多便签，一个变量只是其中一张便签。赋值不是"把东西塞进盒子"，而是"把便签从一个对象上撕下来，贴到另一个对象上"。

```python
x = 10          # 把"x"这张便签贴在整数对象 10 上
y = x           # 把"y"这张便签也贴在同一个对象 10 上
x = x + 1       # 计算 10+1=11，把"x"便签从 10 撕下来，贴到 11 上
print(y)        # 10 — "y"便签还贴在 10 上呢！
```

这就是为什么"修改"整数不影响 `y`——`x = x + 1` 没有修改整数对象 10（也改不了，整数不可变），而是让 `x` 指向了一个新整数 11。

再看可变对象：

```python
items = [1, 2]      # "items"便签贴在列表对象 [1, 2] 上
other = items       # "other"便签也贴上去
items.append(3)     # 直接修改这个列表对象——贴在上面的所有便签都能看到
print(other)        # [1, 2, 3] — 同一个对象！
```

**金句**：变量变了，不等于对象变了；对象变了，不等于变量变了。一切取决于"修改"到底改的是哪一层。

### 二、底层机制：id、is、== 三兄弟

可变性问题的排查工具是三个东西：`id()`、`is`、`==`。

- `id()`：返回对象的**身份标识**（CPython 中是内存地址）。同一个对象 id 相同。
- `is`：判断两个变量是否指向**同一个对象**（比 id）。
- `==`：判断两个对象**值是否相等**（调用 `__eq__`）。

面试时要能讲清它们的区别：

```python
a = [1, 2, 3]
b = [1, 2, 3]

print(a == b)   # True  — 值相等
print(a is b)   # False — 不是同一个对象
print(id(a))    # 比如 140234567890
print(id(b))    # 比如 140234568901，不同！
```

对于不可变对象，"修改"之后 id 会变；对于可变对象，原地修改后 id 不变：

```python
# 不可变 —— id 变
name = "py"
print(id(name))     # A
name += "thon"
print(id(name))     # B, A != B —— 新对象

# 可变 —— id 不变
nums = [1, 2]
print(id(nums))     # C
nums.append(3)
print(id(nums))     # C, 还是同一个！
```

这就是区分"修改"和"重新赋值"最直接的办法。

### 三、默认参数陷阱：函数定义时只执行一次

回到开头的经典代码：

```python
def add_item(item, box=[]):
    box.append(item)
    return box
```

为什么第二次调用能看到第一次的结果？因为 **`box=[]` 这个 `[]` 在函数定义时就创建了，不是每次调用时创建**。之后每次调用 `add_item()` 不传 `box` 参数时，用的都是这个同一个列表对象。

**类比**：函数的默认参数就像你办公桌的抽屉——`def` 执行时你放了一个空抽屉在那里。每次调用不传参数时，你用的就是那个抽屉。第一次放了一支笔（`"a"`），第二次打开抽屉笔还在，又放了一个本子（`"b"`），所以抽屉里现在有两样东西。

正确的写法：

```python
def add_item(item, box=None):
    if box is None:
        box = []      # 每次调用都创建新的列表
    box.append(item)
    return box
```

这个坑不仅影响 list，dict、set 以及任何可变对象作为默认参数都有同样的问题。很多 ORM 的 session、数据库连接、缓存对象也容易掉进这个坑。

### 四、函数传参：不是传值也不是传引用

Python 函数传参的准确说法是 **"传对象引用"（pass by object reference）**，也叫"传引用的值"（pass by assignment）。

**类比教学**：想象你有一间房子的地址（对象引用）。你把地址写在便签上，递给朋友（函数传参）。朋友拿到的是地址的**副本**——你和你朋友拿着两张便签，但上面写的是同一个地址。朋友如果按照地址找到房子，在里面添家具（原地修改可变对象），你再去那间房子也能看到新家具。但朋友如果擦掉便签上的地址，写上另一个房子的地址（重新赋值形参），你的便签没变，你还是会去原来的房子。

```python
def rebind(data):
    data = []           # 擦掉便签，写上新的地址——只影响局部

def mutate(data):
    data.clear()        # 按照地址找到房子，清空——影响原对象

items = [1, 2, 3]
rebind(items)
print(items)            # [1, 2, 3] — 没变
mutate(items)
print(items)            # [] — 变了！
```

面试时别再说"传值"或"传引用"了，说"**函数拿到的形参和实参指向同一个对象，重新赋值只影响局部变量，原地修改会影响共享对象**"。

### 五、tuple 做 key 的边界

`tuple` 通常可以做 dict 的 key，因为它不可变，哈希值稳定。但有一个边界面试官特别爱追问：

```python
# ✅ 正常 tuple
d = {}
d[(1, 2)] = "ok"        # tuple 内所有元素都可哈希

# ❌ 包含可变对象的 tuple
# d[([1, 2], 3)] = "no"  # TypeError: unhashable type: 'list'
```

判断一个 tuple 能不能做 key，不是看它是不是"tuple 类型"，而是看**它的所有元素是否都可哈希**。如果 tuple 里包含了 list、dict、set 等可变对象，整个 tuple 就不可哈希。

**这又回到之前的金句**：tuple 的不可变是浅层的——它锁住的是它保存的引用位置，锁不住引用指向的对象。如果你在 tuple 里放了一个 list，list 本身仍然可以 `append`，但 tuple 的引用关系没变。然而哈希值要涵盖所有内容，包括那个 list——list 内容变了，按理说哈希值也变了，所以 Python 干脆不让这种 tuple 被哈希。

### 六、工程场景：配置被意外污染

```python
base_config = {"timeout": 3, "headers": []}

# 场景一：你以为 copy 了就隔离了
client_config = base_config.copy()
client_config["headers"].append("trace-id")

print(base_config["headers"])  # ['trace-id'] — 被污染了！
```

`dict.copy()` 是浅拷贝——复制了外层字典，但内部的可变对象 `headers` 列表还是同一个。这就是可变对象共享引用在工程中最常见的坑。

更稳的做法：

```python
client_config = {
    **base_config,
    "headers": list(base_config["headers"]),  # 显式复制嵌套可变字段
}
```

这样做的好处是你清楚表达了意图：哪些字段允许共享（只读配置），哪些字段必须隔离（每个请求独立的列表）。盲目 `deepcopy` 虽然省事，但一来有性能成本，二来某些字段（数据库连接、线程锁）根本不应该被复制。

## 🎯 面试总结

### 必答 3 点

1. **变量绑定模型**：变量不存数据，存对象引用。赋值是重新绑定，不是修改对象。
2. **可变与不可变的区别**：不可变对象"修改"产生新对象（id 变）；可变对象原地修改 id 不变。
3. **函数传参**：传的是对象引用，重新赋值不影响外部，原地修改影响外部——不要说传值或传引用。

### 加分项

- 能讲清默认参数只创建一次，并举出工程中的实际场景
- 能说清 tuple 做 key 的条件——所有元素可哈希，不是"tuple 本身不可变"
- 能用 `id()` 和 `is` 排查共享引用问题

### 金句

> "变量是贴在对象上的便签，不是装东西的盒子。"

> "默认参数就像你办公桌的抽屉——定义时放进去，每次调用都在用同一个。"

> "tuple 锁住的是引用关系，不是引用指向的世界。"

---

## 🔍 排查实践

**遇到"数据莫名被修改"的 bug，按这个顺序排查：**

1. **先打 `id()`**：确认修改前后的外层和内层对象是不是同一个。id 一样就是共享引用。
2. **找原地修改操作**：`append()`、`extend()`、`update()`、`clear()`、`dict[key] = value`、`list[i] = value`——这些操作修改对象内容，不是重新赋值。
3. **检查默认参数**：函数签名里有没有 `= []`、`= {}`、`= set()`？有就改成 `= None`。
4. **检查配置共享**：跨模块、跨请求复用的配置对象，嵌套可变字段有没有被意外共享？
5. **设计原则**：只读数据可以共享，请求/任务临时状态必须独立。让可变字段有明确的所有者。

面试回答按"**变量绑定 → 对象可变性 → 函数传参 → 默认参数 → 哈希边界**"展开，基本能覆盖所有追问。
