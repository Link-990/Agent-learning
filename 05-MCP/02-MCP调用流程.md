# MCP 调用流程

## 🎭 面试官这样问
> 面试官：用户在聊天框里问了一句"查一下最近的错误日志"，你从头到尾讲一遍，MCP 是怎么把这个请求变成结果的？
> 你：就是模型选了一个日志查询工具，然后调用 Server，Server 返回结果给模型。
> 面试官：你这个回答太粗糙了。那我追问一句——模型是怎么"看到"有哪些工具的？它怎么知道该选哪个？调用成功后，结果怎么回到模型手里？如果 Server 返回了日志但模型说"我无法回答"，问题出在哪一层？
> 你：嗯……工具是 Host 告诉模型的，结果也是 Host 放回去的……具体链路我不是特别清楚。
> 面试官：核心是你把 MCP 调用当成了一次"函数调用"，但它其实是一条六步的流水线——从配置启动、握手、能力发现、模型选择、协议请求到结果回填，每一步都可能挂。你不把这条链路拆开讲，面试官随便卡你一步你就答不上来。
> 好，我们现在就把这条链路从头到尾拆解清楚，让你面试时不被追问到死角。

## 💡 简要回答（面试话术）

MCP 的一次完整调用，不是"模型调函数"这么简单。它是一条六步流水线：**配置启动 → 初始化握手 → 能力发现 → 模型选择能力 → 协议请求 → 结果回填**。每一步都有独立的失败可能，排查问题必须按顺序逐层定位。

你可以把这条链路类比成"点外卖"。第一步是商家上线（Server 启动配置），第二步是平台跟商家确认菜单（初始化握手），第三步是菜单同步到你的 App 里（能力发现），第四步是你看菜单选菜（模型选择能力），第五步是商家接到订单开始做菜（协议请求与执行），第六步是骑手把菜送到你手里（结果回填）。任何一个环节断了——商家没上线、菜单没同步、App 没展示、订单发错了——你都吃不上饭。MCP 完全一样，模型看到"工具可用"不等于链路通畅。

面试时一定要把这条链路讲成"流水线"，不要讲成"函数调用"。函数调用只覆盖了第四步和第五步的一小部分，剩下四步是 MCP 独有的复杂度。

## 📝 详细解析

### 核心矛盾：模型看到了工具，不代表链路可用

这是一个让无数开发者抓狂的场景：你在 Host 的配置里加了一个新的 MCP Server，工具列表也显示成功了，模型也生成了正确的工具调用参数，但最后返回的是"抱歉，我无法完成这个操作"。你去看 Server 日志——Server 根本没收到请求。你看 Host 日志——Client 初始化失败了，但错误被吞了，界面没有任何提示。

这就是 MCP 调用流程的核心矛盾：模型看到的工具列表，是 Host 整理过的"菜单"，底层连接的真实状态模型根本不知道。模型更像是一个"坐在办公室里的决策者"，它根据面前的信息做判断，但不知道电话线断了没有、快递员在路上摔了没有。真正的网络连接、进程管理、权限确认、协议编解码，全由 Host 和 Client 在底层默默处理。

所以说，MCP 调用流程面试考的不是"你会不会调 API"，而是"你能不能把一个分布式系统的调用链路讲清楚"。六步中任何一步出了问题，最终表现都是"模型无法完成"，但根因可能天差地别。

### 底层机制：一次调用经历的六个关卡

咱们一步一步走一遍。

**第一步：配置和启动。** Host 根据配置文件决定启动哪些 MCP Server。配置里会指定 Server 的类型（stdio 还是 HTTP）、启动命令或远程地址、工作目录、环境变量、认证信息等。比如一个本地文件系统 Server，配置可能是一条命令 `npx @modelcontextprotocol/server-filesystem /path/to/project`。这里最容易出的问题是命令不存在、环境变量缺失、工作目录错误——Server 压根没起来，后面全白搭。

**第二步：初始化握手。** Client 跟 Server 建立连接后，第一件事是握手——协商协议版本、交换能力声明和元信息。Server 告诉 Client："我支持 MCP 协议版本 X，我提供以下能力类型。"Client 验证版本兼容性，确认 Server 的身份。如果版本不匹配或者 Server 返回了非法消息，握手失败，这个 Server 就不能用。这就像 USB 设备插上后，操作系统先要识别设备类型——是键盘还是 U 盘还是打印机——识别不了就没法用。

**第三步：能力发现。** 握手成功后，Client 向 Server 请求完整的能力清单：有哪些 tools、resources、prompts，每个 tool 的输入参数 schema 是什么，每个 resource 的 URI 模板是什么。Client 把这份清单交给 Host。Host 在这一步有重要的决策权：不是所有能力都要暴露给模型。比如某个 tool 标记了"需要管理员权限"，Host 可以把它从模型可见列表中过滤掉，或者弹出确认框让用户授权。

**第四步：模型选择能力。** 用户提出任务后，Host 把当前可用的工具描述（经过过滤和整理的）放入模型的上下文窗口。模型根据用户意图和工具描述，通过 Function Calling 机制选择一个 tool 并生成结构化参数。比如用户说"查最近七天的订单量"，模型可能先选 `list_tables` 看有哪些表，再选 `run_query` 执行 SQL。注意：模型在这一步完全不知道底层连接状态，它只是根据"菜单"做判断。

**第五步：协议请求。** Host 把模型生成的 tool 名称和参数交给对应的 Client。Client 按 MCP 协议格式打包请求，发给 Server。Server 收到后做参数校验、权限校验、实际执行——比如查询数据库、读取文件、调用企业 API。Server 必须做防御性编程：参数类型不对要拒绝，SQL 注入要拦截，超时要 kill，返回行数要限制。不能信任模型生成的任何参数。

**第六步：结果回填。** Server 返回结构化结果，Client 解析后交给 Host，Host 把它作为"工具观察"放回模型的上下文窗口。模型拿到结果后继续推理，可能再调一次工具，也可能直接生成最终回答。这里有一个高频踩坑点：Server 返回成功了，但 Host 没有把结果正确回填——模型就"看不到"结果，表现就是"我无法完成"。

### 工程例子：数据库 MCP Server 一次完整走通

假设你有一个只读数据库 MCP Server，配置启动后暴露了三个 tool：`list_tables`、`describe_table`、`run_readonly_query`，还有一个 `database_schema` resource。

用户问："最近七天订单量是不是下降了？"

模型先选择 `list_tables`，想确认有哪些表。Host 把请求通过 Client 发给数据库 Server，Server 返回表名列表，Host 回填给模型。模型看到有 `orders` 表，又选 `describe_table` 查表结构，确认了 `order_date` 和 `amount` 字段。最后模型生成一条 SQL：`SELECT DATE(order_date), COUNT(*) FROM orders WHERE order_date >= DATE('now', '-7 days') GROUP BY 1`，通过 `run_readonly_query` 发过去。Server 做了几层防护：限制 SQL 只能是 SELECT、设置 5 秒超时、限制返回最多 1000 行、脱敏敏感字段。查询成功，返回了七天的每日订单数。Host 把结果回填给模型，模型分析趋势后回答："最近七天订单量整体平稳，周三有小幅下降约 5%，但周末回升了。"

这整个过程中，如果 Server 查询成功了但 Host 没有回填——模型只能干瞪眼，回答不了。如果模型生成了 UPDATE 语句——只读 Server 直接拒绝，返回错误。如果查询超时了——Server 返回结构化超时错误，模型可以据此建议"数据量可能较大，要不要缩小时间范围？"

### 边界和风险：六步里每一步都可能挂

**启动阶段**：命令路径不对、Node.js 版本不够、环境变量没设、工作目录不存在。排查时先手动执行一下启动命令，看 Server 能不能独立跑起来。

**握手阶段**：协议版本不兼容是最常见的。MCP 还在快速迭代，Server 用的版本和 Client 期望的版本可能对不上。另外，远程 HTTP Server 还可能遇到 TLS 证书问题、认证 token 过期。

**能力发现阶段**：工具描述不完整、参数 schema 不符合 JSON Schema 规范、resource URI 模板写错。如果 Host 在这一步做了过度过滤，模型可能"看不到"本该可用的工具。

**模型选择阶段**：工具描述写得太差，模型选错工具；或者工具太多（比如 50 个），模型在上下文里迷路。还有种情况：模型生成了参数但值不对——比如把 `limit=1000` 写成 `limit="一千"`。

**协议请求阶段**：Server 执行时遇到真实错误——数据库挂了、文件被删了、API 返回 500。Server 要把这些错误结构化返回，不能只丢一个 "error" 字符串。超时和取消也要处理——不能让一个慢查询把整个 Host 卡住。

**结果回填阶段**：返回内容太大，超出上下文窗口，Host 做了截断但模型不知道；或者返回格式变化，Host 解析失败；或者多个工具调用的结果被 Host 合在一起，模型分不清哪个结果对应哪个调用。

还有一个全局风险：未知 Server。如果你允许 Host 自动加载社区里的 MCP Server，一个恶意 Server 可以声明一堆诱导性工具——比如"读取系统所有环境变量""执行任意 Python 脚本"。模型可能被误导去调用。企业环境必须做白名单。

## 🎯 面试总结

- **必答 3 点**：
  1. MCP 调用流程是六步流水线：配置启动 → 初始化握手 → 能力发现 → 模型选择能力 → 协议请求 → 结果回填，不是"模型调函数"那么简单
  2. 模型不直接连接 Server，所有通信经过 Host 和 Client 中转，模型看到的工具列表是 Host 过滤整理后的结果
  3. 排查故障必须按链路逐层定位——Server 是否启动、握手是否成功、能力是否返回、Host 是否暴露工具、参数是否合法、结果是否回填

- **加分项**：能类比"点外卖"流程讲清楚六步；能指出每一步独立的失败模式和排查方法；能说清楚模型"看不到"结果不等于 Server 没返回——往往是 Host 回填环节出了问题

- **一句话总结**：MCP 调用不是一次函数调用而是一条分布式流水线，每一步都可能挂，排查时必须从 Server 启动一路追到结果回填，中间少一环模型就傻眼。

## 🔍 排查和实践建议

排查时按时间线走：看 Server 进程在不在（或远程地址通不通）→ 看 Client 初始化日志有没有握手成功的记录 → 看能力发现返回的 tools 列表是否完整 → 看 Host 是否把工具写入了模型上下文 → 看模型生成的 tool 调用参数是否正确 → 看 Server 执行日志有没有报错 → 看 Host 有没有把结果回填。给每次调用打一个 trace_id，从用户问题 → 模型工具选择 → Client 请求 → Server 返回 → 最终回答，串成一条完整的链路。没有 trace，MCP 问题排查就是"各层互相甩锅"。设计 Server 时给每个 tool 返回结构化结果，成功有成功的格式，失败有失败的格式——别让模型猜"这个空响应是什么意思"。
