# 装饰器与生成器

> 装饰器和生成器是 Python 面试的"分水岭"题目——能讲清楚的人通常已经写过一定规模的工程代码。它们分别解决两类完全不同的问题：装饰器管"不改代码主体增强行为"，生成器管"按需产数据而不是一次性塞进内存"。

---

## 🎭 面试官这样问

**面试官**："先考个基础的。装饰器是干什么用的？`@decorator` 的等价写法是什么？"

**很多同学**："装饰器就是给函数加功能的。`@decorator` 等价于 `func = decorator(func)`。"

**面试官**："行。那装饰器里为什么总见到 `functools.wraps`？不加会怎样？"

**很多同学**："呃……好像是保留函数名？不加的话函数名会变成 wrapper？"

**面试官**："那你写个带参数的装饰器——比如一个 `@retry(3)`，重试三次。"

**很多同学**："这个……好像要三层函数？但我讲不清为什么。"

你看，第一轮就暴露了：你知道装饰器能"包装"函数，但不清楚它的本质是高阶函数+闭包。

---

**面试官**："换个话题。生成器是什么？跟普通函数有什么区别？"

**很多同学**："生成器用 `yield`，调用时返回生成器对象，可以用 `for` 或 `next()` 迭代……"

**面试官**："那你看这段代码："

```python
g = (x * 2 for x in range(3))
print(list(g))
print(list(g))
```

**很多同学**："两次都输出 `[0, 2, 4]` 吧？"

**面试官**："再跑一下。"

**很多同学**："第二次是 `[]`？怎么没了？"

**面试官**："那如果我文件很大，不想一次读完，但又需要多次遍历，怎么办？"

---

**面试官继续追问**："多个装饰器叠在一起，执行顺序是怎样的？比如 `@a` 在上面，`@b` 在下面，调用时先走哪个？"

**很多同学**："从上到下？"

**面试官**："你确定？装饰阶段和调用阶段是一样的顺序吗？"

这就是为什么"装饰器和生成器"是面试重灾区——它们涉及闭包、函数一等对象、迭代协议、延迟求值，每一层追问都可以很深。

---

## 💡 简要回答（面试话术，可以直接背）

> "装饰器本质是高阶函数——接收一个函数，返回一个**增强后**的函数。`@decorator` 等价于在函数定义后执行 `func = decorator(func)`。带参数装饰器需要三层：最外层接收装饰器参数，中间层接收被装饰函数，最内层是真正的 wrapper。`functools.wraps` 用于保留原函数的 `__name__`、`__doc__` 等元信息。
>
> 生成器是通过 `yield` 定义的**延迟求值**迭代器。调用生成器函数不执行函数体，而是返回生成器对象；每次 `next()` 或 `for` 迭代时执行到下一个 `yield` 暂停，同时保存局部状态。优点是**节省内存**（不需要一次性把所有数据放进列表），边界是只能**顺序消费**一次，不能随机访问。"

---

## 📝 详细解析

### 一、装饰器本质：高阶函数 + 闭包

**类比教学（外卖包装盒）**：装饰器就像外卖的包装盒。你有一份饭（原函数），包装盒（装饰器）把它包起来，外面贴个标签（`@functools.wraps` 保留名字），加上筷子纸巾（增强逻辑），但里面的饭还是原来的饭。更妙的是，包装过程发生在厨房（函数定义时），不是顾客打开时（调用时）。

**三步拆解装饰器：**

第一步：Python 中函数是一等对象——可以赋值给变量、作为参数传递、作为返回值返回。

```python
def greet():
    return "hello"

say = greet          # 把函数对象赋给另一个变量
print(say())         # "hello" — 可以正常调用
```

第二步：装饰器利用这个特性，接收函数返回函数：

```python
def log_cost(func):
    def wrapper(*args, **kwargs):
        print(f"calling {func.__name__}")
        result = func(*args, **kwargs)
        print(f"done")
        return result
    return wrapper

@log_cost
def query():
    return "data"

# 等价于：query = log_cost(query)
```

第三步：`wrapper` 是一个**闭包**——它记住了外层函数作用域里的 `func`。即使 `log_cost` 已经返回，`wrapper` 仍然能访问 `func`。

**金句**：装饰器的三个关键词——**高阶函数**（接收/返回函数）、**闭包**（wrapper 记住 func）、**语法糖**（`@` 等于重新赋值）。

### 二、为什么需要 `functools.wraps`

不加 `functools.wraps`，装饰器会把原函数的元信息覆盖成 wrapper 的：

```python
def bad_decorator(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@bad_decorator
def important():
    """这是重要文档"""
    pass

print(important.__name__)   # "wrapper" — 而不是 "important"！
print(important.__doc__)    # None — 文档丢了！
```

这会影响：
- **路由注册**（如 Flask `@app.route` 路由名可能变成 wrapper）
- **自动文档生成**（Sphinx 等工具读 `__doc__`）
- **单元测试**（pytest 找不到正确的函数名）
- **调试**（traceback 里显示的栈帧是 wrapper 不是原函数）

```python
from functools import wraps

def good_decorator(func):
    @wraps(func)           # 把 func 的元信息复制到 wrapper
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper
```

### 三、带参数装饰器：为什么需要三层

**类比教学**：带参数装饰器就像**定制包装盒工厂**。你要先告诉工厂"给我做一个重试 3 次的盒子"（`retry(3)`），工厂返回一个装饰器（中间层函数），再由这个装饰器去包你的饭（被装饰函数）。

```python
def retry(times):
    def decorator(func):            # 中间层：接收被装饰函数
        @wraps(func)
        def wrapper(*args, **kwargs):  # 内层：真正的增强逻辑
            last_error = None
            for _ in range(times):
                try:
                    return func(*args, **kwargs)
                except Exception as exc:
                    last_error = exc
            raise last_error
        return wrapper
    return decorator

@retry(3)
def call_api():
    pass
```

执行顺序：`retry(3)` → 返回 `decorator` → `@decorator` 装饰 `call_api` → `call_api = decorator(call_api)` → 最终 `call_api` 指向 `wrapper`。

面试时画清楚这个三层结构，比死记"带参数要三层"强得多。

### 四、多个装饰器的执行顺序

```python
@a
@b
def f():
    pass

# 等价于：f = a(b(f))
```

装饰阶段：**从下往上**——先 `b` 包装 `f`，再 `a` 包装 `b(f)`。
调用阶段：**从外往内**——先进入 `a` 的 wrapper，再进入 `b` 的 wrapper，最后执行 `f`。

**类比**：就像穿衣服——先穿内衣（`@b`），再穿外套（`@a`）。出门时，别人先看到外套。

### 五、生成器：像水龙头一样产数据

**类比教学（水龙头）**：普通函数像一个大水箱（list），一口气把所有水（数据）灌给你——水少了还行，水多了你根本接不住（内存爆掉）。生成器则像一个**水龙头**（yield），你拧一下（`next()`），流一点水（一个数据），不拧就不流。数据产出的节奏由调用方控制。

![装饰器与生成器运行机制](https://gitee.com/linkio666/image/raw/main/01-python-basics/03-decorators-generators/02-运行机制.png)

```python
def read_error_lines(path):
    with open(path, encoding="utf-8") as f:
        for line in f:
            if "ERROR" in line:
                yield line.strip()

for line in read_error_lines("huge_app.log"):
    process(line)
```

生成器函数被调用时**不执行**函数体，而是返回一个生成器对象。每一次迭代：
1. 执行到 `yield`，返回一个值，**暂停**执行
2. 保存局部变量和执行位置（类似程序的"断点"）
3. 下一次迭代从暂停位置**继续**执行

这就是为什么生成器内存占用低——它不需要在内存里同时保存所有数据。

### 六、生成器的边界：一次性消费品

生成器最大的限制是：**只能消费一次**。

```python
g = (x * 2 for x in range(3))
print(list(g))   # [0, 2, 4]
print(list(g))   # [] — 已经到末尾了，没东西了
```

就像你看完一部电影，进度条到了末尾，不会自动回到开头。需要重新创建一个新生成器，或者先转成 list（但这样就失去了内存优势）。

另外生成器不能像列表那样 `g[0]` 随机访问，不能 `len(g)` 获取长度——它是"现产现用"的模式。

### 七、工程场景

**装饰器场景：**

```python
# 鉴权装饰器
@require_login
def delete_post(post_id):
    pass

# 缓存装饰器
@cache(ttl=60)
def get_user(user_id):
    return db.query(user_id)

# 异常转换装饰器
@translate_error(ValueError, "输入不合法")
def parse_input(raw):
    pass
```

**生成器场景：**

```python
# 分页 API 封装
def paginate(fetch_page):
    page = 1
    while True:
        items = fetch_page(page)
        if not items:
            break
        yield from items    # 或 for item in items: yield item
        page += 1

# 数据库游标流式读取
def stream_users(batch_size=1000):
    offset = 0
    while True:
        batch = db.query(User).offset(offset).limit(batch_size).all()
        if not batch:
            break
        yield from batch
        offset += batch_size
```

### 八、边界和风险

**装饰器风险：**
1. 忘记 `return func(*args, **kwargs)`——wrapper 执行完没有返回原函数结果
2. 吞掉异常——比如重试装饰器在重试耗尽后要 `raise last_error`，不能静默失败
3. 改变函数签名——用 `*args, **kwargs` 保持通用性，用 `@wraps` 保留元信息
4. 装饰器叠加顺序错误——鉴权放外层还是缓存放外层，直接影响安全性

**生成器风险：**
1. 被提前转成 list 失去内存优势
2. 资源未释放——如果生成器里有 `with open()`，消费一半就 break，文件可能没有 clean close（用 `contextlib.closing` 或确保生成器被完整消费）
3. 多线程/多进程间传递生成器——生成器绑定执行状态，不能简单序列化

## 🎯 面试总结

### 必答 3 点

1. **装饰器**：高阶函数 + 闭包 + 语法糖，`@decorator` 等价于 `func = decorator(func)`，在定义时执行
2. **生成器**：`yield` 定义的延迟求值迭代器，调用时返回生成器对象，迭代时才执行函数体，一次一个值
3. **工程价值**：装饰器抽离横切逻辑（日志、鉴权、缓存），生成器节省内存（大文件、分页、流式）

### 加分项

- 能讲清楚带参数装饰器为什么是三层，每一层分别接收什么
- 能说出 `functools.wraps` 的三个影响场景（路由、文档、测试）
- 多个装饰器的执行顺序——装饰自下而上，调用自外而内
- 生成器只能用一次，不能随机访问——能讲清为什么以及工程对策

### 金句

> "装饰器就像外卖包装盒——包在函数外面，里面还是原来的菜。"

> "生成器像水龙头——拧一下出一点水，不拧不出。全拧完，就没了。"

> "装饰器改变的是调用路径，不是业务逻辑。"

## 🔍 排查实践

**装饰器出问题，按这个顺序查：**

1. 先确认被调用的到底是不是包装后的函数（打 `__name__`）
2. 检查 wrapper 有没有 `return func(*args, **kwargs)` ——忘了 return 返回值就是 None
3. 检查有没有吞异常——`try-except` 是否吃掉了不该吃的
4. 有没有加 `@functools.wraps`
5. 多个装饰器叠加顺序对不对——鉴权应该在最外层

**生成器出问题，按这个顺序查：**

1. 生成器是不是被消费过一次了——打 `list()` 之后变空
2. 是不是被意外转成了 list——`result = list(gen)` 之后内存暴涨
3. 文件/连接句柄 —— `with open()` 要确保在生成器生命周期内
4. 如果确实需要多次遍历——是重新创建生成器，还是权衡后转成 list

面试回答按"**场景 → 本质 → 执行时机 → 代码例子 → 边界**"展开，不会停留在背定义层面。
