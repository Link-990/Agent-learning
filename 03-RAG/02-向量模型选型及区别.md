# 向量模型选型及区别

## 🎭 面试官这样问

> 面试官：RAG 系统里向量模型怎么选？是不是排行榜第一的就最好？

> 你：选 embedding 模型主要看评测指标，比如 MTEB 排行榜的 Retrieval 分数，分数高的语义表示能力强，召回效果好。

> 面试官：那你解释一下，为什么某模型 MTEB 排名很高，但在我们客服系统里，用户问「E1024 报错怎么解决」，它召回的全是「蓝牙连接失败常见问题」，却漏掉了标题精确包含「E1024」的技术文档？

> 你：因为向量模型擅长语义相似，对 E1024 这种精确错误码不够敏感，可能需要结合关键词检索……

> 面试官：方向对了。那再说一个场景：我们有 1000 万个文档 chunk，上线后 QPS 要跑到 500。你选 768 维还是 1536 维？维度翻倍，存储、索引、延迟各涨多少？索引重建要多久？这些你算过吗？

> 你：呃……维度高的表达能力更强，但成本也会涨。具体数字需要根据向量库和硬件估算……

> 面试官：对。所以向量模型选型不是看一个分数，而是要在语义效果和工程成本之间做权衡。模型换了，索引要全量重建，不是说换就能换的。你能把选型要看的关键维度一个一个讲清楚吗？

这就是面试官想听的——你既懂语义表示，也懂生产成本。下面展开讲。

![向量模型语义空间示意](https://gitee.com/linkio666/image/raw/main/03-RAG/02-embedding-models/01-概念总览.png)

## 💡 简要回答（面试话术）

面试时你可以这样组织回答：

> 向量模型在 RAG 里的核心作用，是把用户问题和文档片段映射到同一个语义空间，让字面不同但意思接近的文本在向量距离上更近。这样用户说「买了 10 天还能退吗」，系统能召回「签收后 15 天内可申请无理由退货」——关键词重合很少但语义高度相关。

> 但选型不能只看排行榜。排行榜的分數是通用场景下的平均值，你的业务可能有一堆错误码、产品型号、函数名和医学术语，通用模型不一定 cover 得住。我会从七个维度综合评估：语言能力、领域适配、向量维度和存储成本、输入长度限制、相似度指标匹配、延迟和价格、以及索引重建成本。

> 工程上，我不会一上来就换模型。我会先用成熟通用模型建 baseline，用真实业务 query 构建评测集，对比候选模型在当前分块策略和向量库配置下的 Recall@K 和 MRR。只有业务效果提升能覆盖迁移成本时才换。如果效果不好，优先排查分块、相似度配置、混合检索和 query 改写，而不是直接换 embedding。

## 📝 详细解析

### 核心矛盾：语义召回 vs 工程约束

向量模型的本质是把文本压缩成一个固定维度的数字向量。这个压缩过程天然有损失：一段 500 token 的文字可能包含退款、维修、发票、物流四个主题，但最终只有一个向量，它只能是一个「平均」的语义表示。维度越高，表达能力可能越强，但存储、索引、内存和查询延迟也成倍增长。

这带来选型的核心矛盾：**你想让向量精准区分业务概念，但又不能花太多钱。** 1000 万个 chunk，1536 维 float32，光向量数据本体就近 60 GB，还没算索引结构。线上 QPS 一上来，embedding API 调用成本持续放大——每次用户提问都要做一次 query embedding，每次入库都要做文档 embedding。选模型本质是在「召回效果」和「成本」之间找平衡点。

### 底层机制：embedding 不只是转数字（类比教学）

把 embedding 模型想象成一个**专业的图书管理员**。这个管理员训练有素，你问「这本书讲了健康饮食」，他能把讲营养学、减脂、养生的书都找出来——即使书名里没有「健康饮食」四个字。他的本事是「理解」内容，而不是「匹配」词。

但这位管理员也有盲区。你问他找编号为「E1024」的技术手册，他可能给你拿一堆蓝牙相关的书，却漏掉那本标题就是「E1024：蓝牙模块初始化」的手册——因为在他的「理解」里，「蓝牙连接」和「E1024」是同一个语义方向，他分不清哪个是精确匹配。

这就是向量模型的运作机制：通过对比学习训练，让相关文本在向量空间距离近，不相关文本距离远。检索时用 cosine similarity、dot product 或 L2 distance 算距离。但这里有三个常常被忽视的细节：

**第一，相似度指标要和模型训练方式匹配。** 有些模型训练时用 cosine 优化，你如果在向量库里配置成 dot product，召回效果可能明显下降。向量有没有归一化也会影响 distance 计算。

**第二，输入长度有限制。** embedding 模型有最大 token 输入限制，比如 512 token。如果你的 chunk 是 1000 token，后半段直接被截断，关键条件根本没进入向量。你以为模型不懂业务，其实是分块太长被腰斩了。

**第三，一个向量表达多个主题会「语义稀释」。** 一个 chunk 里混合了退款和维修两个主题，向量就成了两者的平均值。用户问退款，检索既不精准也不稳定——这就是分块和向量模型的配合问题。

### 工程例子：怎么评估候选模型

不要只看公开榜单，榜单上的高分不保证在你的场景有效。我来演示一个最小可用评测流程：

先构造评测集，从真实日志里抽样，覆盖不同问题类型：

```
问题：耳机买了 10 天还能退吗？
标准相关文档：耳机售后政策 2026 版 - 15 天无理由退货
类型：同义改写

问题：E1024 报错怎么解决？
标准相关文档：E1024：蓝牙模块初始化失败处理指南
类型：精确符号

问题：退款多久到账？
标准相关文档：退款款项返还时间说明
类型：口语化表达

问题：如何申请退换货？
标准相关文档：退换货申请流程（网页端 / App 端）
类型：操作步骤
```

然后对每个候选模型，用同一套分块参数、同一个向量库配置、同一批 query 跑检索。看标准文档是否进入 top1、top3、top5。记录关键指标：

- **Recall@K**：正确文档有没有被召回进前 K 个候选？这是第一道关，答不上就全完。
- **MRR（Mean Reciprocal Rank）**：第一个正确文档排多靠前？排第 1 和排第 5 差距巨大。
- **NDCG**：当有多个相关文档时，排序质量如何？
- **Latency**：query embedding 耗时多少？直接影响端到端延迟。
- **Cost**：每千次 embedding 多少钱？QPS 高的场景下这是持续支出。

最后还要看**失败样本**——哪些 query 在两个模型上都答不对，哪些在老模型上对但在新模型上错。平均值好看不代表没退步。

### 选型必看的七个维度

**第一，语言能力。** 中文知识库要测中文语义、口语表达、同义改写。中英混合场景还要测跨语言检索——中文提问英文文档能不能召回。

**第二，领域适配。** 医疗里的「阴性/阳性」、法律里的「除外责任」、金融里的「授信额度」、代码里的「幂等键」——术语理解错一个，召回就全错。通用模型能做 baseline，但垂直场景可能需要领域微调模型。

**第三，向量维度和存储成本。** 维度不是越高越好。从 768 到 1536 维度翻倍，存储和计算也翻倍。要拿召回提升的幅度来 justify 成本增长。

**第四，输入长度。** chunk size 必须和 embedding 模型的 max token 一起设计。超过限制被截断，等于埋雷。

**第五，相似度指标和向量库兼容性。** cosine、dot product、L2 不能随便换。索引类型（HNSW、IVF、PQ）的精度和延迟也不同。

**第六，延迟、吞吐和价格。** 在线 RAG 每条 query 都要做 embedding。API 慢则全链路慢，QPS 高则成本持续放大。

**第七，索引重建成本。** 换 embedding 模型意味着全量重新向量化、重建索引、回归测试。1000 万级知识库可能要好几天，期间要有双索引方案和回滚计划。

### 边界和风险

**语义相似不等于业务相关。** 这是最大的陷阱。用户问「退款多久到账」，模型召回「如何申请退款」——两者都和退款有关，但一个是问到账时间，一个是问操作流程。语义接近但答不上问题。

**精确符号容易被语义稀释。** 错误码 E1024、产品型号 SKU-8821、函数名 getUserById——这些符号在语义空间里可能被周围的自然语言「淹没」，导致检索不稳定。

**embedding 效果和分块强相关。** 一个 chunk 混入多个主题，多强的模型也没办法把多件事精准表达成一个向量。

**模型升级有迁移风险。** 新模型整体指标可能更高，但某些老业务 case 可能退步。上线必须做灰度、保留旧索引做双路对比，不能直接覆盖。

## 🎯 面试总结

- **必答 3 点：** ① 向量模型把文本映射到语义空间，解决同义表达和口语化问题；② 选型看七个维度——语言、领域、维度、输入长度、相似度指标、延迟成本、重建成本；③ 选型要用业务评测集，不能只看公开排行榜。
- **加分项：** 能讲出「语义相似不等于业务相关」的陷阱。能区分 embedding 问题和分块问题。能估算大规模知识库的索引重建成本和回滚方案。
- **一句话总结：** 向量模型选型是语义效果和工程成本的平衡——不是选最强的，是选在你的语言、领域、规模和延迟预算下，能稳定把正确片段送到前面的那个。

## 🔍 排查和实践建议

向量召回效果不好时，按这个顺序排查：
1. **Query 是否完整？** 太短、缺主语、多轮对话没补全？
2. **Chunk 是否主题混杂？** 关键条件是否被切断？
3. **Embedding 是否适配语言和领域？** 用评测集验证。
4. **相似度指标和归一化是否匹配模型？** cosine/dot product 配置对吗？
5. **Top_k 是否过小？** 正确片段排在较后位置被截断？
6. **是否需要混合检索？** 精确符号场景加关键词检索补齐。

实践上，先用成熟通用 embedding（如 text-embedding-3-small 或 bge-large-zh）建 baseline，再用业务评测集决定是否需要领域模型、微调或混合检索。大规模知识库换模型前，务必估算重建索引时间、存储增长、回滚方案和灰度策略。
