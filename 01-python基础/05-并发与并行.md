# 并发与并行

> 同一个"加速"需求，线程池能让爬虫从 100 秒降到 15 秒，但对纯 CPU 计算几乎无效。说不清并发和并行区别的人，常常在选型上出事故——该用进程的任务上了线程，该用异步的任务用了多进程。

---

## 🎭 面试官这样问

**面试官**："并发和并行有什么区别？单核 CPU 能并发吗？"

**很多同学**："并发是交替执行，并行是同时执行。单核 CPU 只能并发，不能并行。"

**面试官**："好。那我有一个爬虫要请求 100 个网页，还有一个任务要算 100 万次 SHA256。分别适合用什么方案？"

**很多同学**："爬虫用多线程，SHA256 用多进程。"

**面试官**："为什么？底层原因是什么？"

**很多同学**："因为爬虫是 I/O 密集型，多线程在等网络时可以让出 CPU；SHA256 是 CPU 密集型，Python 有 GIL 所以多线程不行……"

**面试官**："那如果 SHA256 我用 `hashlib` 这个 C 扩展库呢？还能用多线程吗？"

**很多同学**："呃……好像 `hashlib` 会释放 GIL？那可能多线程也行？"

---

**面试官继续**："你把并发数从 5 调到 50，速度反而更慢了。为什么？"

**很多同学**："上下文切换开销？"

**面试官**："还有呢？如果你在调用一个外部 API，对方有 QPS 限制呢？如果你的数据库连接池只有 10 个连接，你开了 50 个线程呢？"

**很多同学**："那就不是并发模型的问题，是资源共享和下游瓶颈的问题……"

**面试官**："对。那协程和线程有什么区别？协程里调用 `time.sleep()` 会怎样？"

---

你看，"并发 vs 并行"的面试题，从概念到选型，到性能瓶颈分析，再到协程陷阱——每一层都可以追问到底。

---

## 💡 简要回答（面试话术，可以直接背）

> "并发是多个任务在一段时间内**交替推进**，核心是**调度和等待时间利用**，单核 CPU 也能并发。并行是多个任务在同一时刻**真正同时执行**，需要多个执行资源（多核、多进程、GPU、多机）。
>
> I/O 密集型任务（网络、磁盘、数据库）大部分时间在**等待**，适合多线程或协程——等待期间可以让出执行权。CPU 密集型任务瓶颈在**计算**，Python 中适合多进程、释放 GIL 的 C 扩展（如 NumPy、hashlib）、GPU 或分布式。
>
> 工程选型时，先判断瓶颈类型，再看第三方库是否支持异步、任务是否共享状态、是否需要限流和超时。并发数不是越大越好——超过系统承载能力反而会因调度、锁竞争和下游瓶颈导致性能下降。"

---

## 📝 详细解析

### 一、核心矛盾：调度 vs 资源

**类比教学**：并发和并行最经典的类比是"服务员和餐桌"：

- **并发**：一个服务员（单核 CPU）同时服务多张桌子。他在 A 桌点完菜（在等菜期间），立刻去 B 桌倒水，再去 C 桌收盘子——一段时间内，每桌都得到了服务。这就是"并发"：**通过快速切换，让多个任务都在推进**。
- **并行**：你雇了三个服务员（多核），每人服务一张桌子——三张桌子同时被服务。这就是"并行"：**通过增加执行资源，让多个任务真正同时进行**。

所以并发解决的是"**等待时间怎么利用**"，并行解决的是"**计算资源怎么增加**"。两者解决的不是同一个问题。

![并发与并行概念总览](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/01-python-basics/05-concurrency-parallelism/01-概念总览.png)

### 二、I/O 密集型 vs CPU 密集型：从现象判断瓶颈

面试时你要能讲清楚**怎么判断任务类型**：

**I/O 密集型的特征**：
- CPU 利用率低（经常 < 30%），但任务总耗时长
- 大部分时间卡在 `requests.get()`、`cursor.execute()`、`file.read()` 等 I/O 调用上
- 增加 CPU 核数不会明显加速
- **解决方案**：多线程、协程、异步 I/O——让等待的时间被其他任务利用

**CPU 密集型的特征**：
- CPU 利用率持续接近 100%
- 任务主要在做纯计算：循环、哈希、图像处理、加密、科学计算
- 增加 I/O 并发不会加速
- **解决方案**：多进程、C 扩展、NumPy、GPU、分布式——增加实际计算资源

```python
# I/O 密集型 —— 多线程
from concurrent.futures import ThreadPoolExecutor
import requests

def fetch(url):
    return requests.get(url, timeout=5).status_code

with ThreadPoolExecutor(max_workers=10) as pool:
    results = list(pool.map(fetch, urls))
```

```python
# CPU 密集型 —— 多进程
from concurrent.futures import ProcessPoolExecutor

def heavy_compute(n):
    return sum(i * i for i in range(n))

with ProcessPoolExecutor() as pool:
    results = list(pool.map(heavy_compute, [10_000_000] * 4))
```

### 三、Python 三种并发方案对比

| 方案 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **多线程** | I/O 密集型 | 简单、共享内存、`concurrent.futures` 成熟 | GIL 限制 CPU 并行、线程安全需加锁 |
| **协程 (asyncio)** | 高并发 I/O（万级连接） | 单线程内调度、开销极低、无需锁 | 要求全链路非阻塞、学习曲线陡、阻塞调用卡死事件循环 |
| **多进程** | CPU 密集型、隔离性要求高 | 真正利用多核、进程间隔离 | 创建开销大、数据需序列化传输、内存占用高 |

![并发与并行运行机制](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/01-python-basics/05-concurrency-parallelism/02-运行机制.png)

### 四、协程的核心：事件循环和非阻塞

**类比教学**：协程就像你在家里同时做三件事——烧水、洗衣服、回邮件。你把水壶放上去（发起 I/O 任务），然后去洗衣服；洗衣机开始转后（又一个 I/O），你回来回邮件；水开了（I/O 完成），你回来关火。全程只有你一个人（单线程），但三件事都在推进。

协程的关键是：**任何"等待"都必须是异步的**。如果你在等水开的时候站在厨房门口死等（`time.sleep()`），那衣服就洗不了，邮件也回不了——整个事件循环被你一个人卡死了。

```python
import asyncio

async def fetch(url):
    # ✅ 异步等待——事件循环可以切换到其他协程
    async with aiohttp.ClientSession() as session:
        async with session.get(url) as resp:
            return await resp.text()

async def main():
    tasks = [fetch(url) for url in urls]
    return await asyncio.gather(*tasks)

# ❌ 在协程里调同步阻塞函数——事件循环卡死
async def bad_fetch(url):
    import requests
    return requests.get(url)  # 同步阻塞！整个事件循环被卡住
```

### 五、为什么并发数不是越大越好

面试官追问"为什么并发调高了反而更慢"，你要给出多维度的分析：

1. **上下文切换开销**：线程/进程切换本身消耗 CPU——当切换开销大于等待时间收益时，多做无益
2. **共享资源竞争**：所有线程抢同一把锁、同一个数据库连接池、同一个限流器——并发增加排队，不增加吞吐
3. **下游瓶颈**：外部 API 有 QPS 限制、数据库连接数有限、带宽有限——继续加并发只是把更多请求堆在路上
4. **内存压力**：每个线程有独立的栈内存、每个进程有独立的内存空间——并发过高可能 OOM

**金句**：并发提升吞吐的本质是**利用等待间隙**。一旦所有等待间隙都被填满了，再增加并发就是纯负担。

### 六、工程场景

**场景一：批量调用外部 API**

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def call_model(payload):
    resp = requests.post(API_URL, json=payload, timeout=30)
    return resp.json()

with ThreadPoolExecutor(max_workers=5) as pool:
    futures = {pool.submit(call_model, p): p for p in payloads}
    for future in as_completed(futures):
        try:
            result = future.result(timeout=10)
            save(result)
        except Exception as e:
            log_error(futures[future], e)
```

这里 `max_workers=5` 不是随意写的——需要根据 API 的 QPS 限制、本机连接池大小、内存情况，通过压测找到收益拐点。

**场景二：多进程数据管道**

```python
from multiprocessing import Pool

def process_chunk(data_chunk):
    # 纯 CPU 计算
    return [heavy_transform(item) for item in data_chunk]

# 把大数据集分块，每个进程处理一块
chunks = split_into_chunks(dataset, num_workers=4)
with Pool(processes=4) as pool:
    results = pool.map(process_chunk, chunks)
```

### 七、边界和风险

1. **协程中的阻塞调用**：`time.sleep()`、同步 HTTP、同步数据库驱动——任何一个都会卡死事件循环。必须使用对应的异步库（`asyncio.sleep`、`aiohttp`、`asyncpg`）。
2. **多进程的数据传递**：参数和返回值通过 `pickle` 序列化传输——大对象传输成本高，不能序列化的对象（lambda、生成器、某些 C 扩展）无法传递。
3. **并发和锁导致性能退化**：频繁加锁可能让多线程退化成"排队执行"，收益归零。
4. **异常处理复杂化**：并发中的异常可能被延迟抛出、被吞掉、或在错误的上下文中处理——需要明确超时、重试和降级策略。

## 🎯 面试总结

### 必答 3 点

1. **并发 vs 并行**：并发是交替推进（调度）、并行是同时执行（资源）。单核能并发，不能并行。
2. **I/O 密集型 vs CPU 密集型**：前者等 I/O → 多线程/协程；后者等 CPU → 多进程/C 扩展/GPU。
3. **三种方案对比**：线程（I/O）、协程（高并发 I/O + 非阻塞）、进程（CPU + 隔离）

### 加分项

- 能从现象判断瓶颈类型（看 CPU 利用率、看耗时分布）
- 能分析"并发过高反而慢"的多维原因（切换开销、锁竞争、下游瓶颈、内存压力）
- 知道协程的"全链路非阻塞"要求和常见陷阱
- 能讲清楚多进程的序列化成本和适用边界

### 金句

> "并发是让一个服务员同时盯多张台，并行是雇多个服务员——解决的不是同一个问题。"

> "协程像你在家同时烧水、洗衣、回邮件——但如果你站厨房门口死等水开，后面两件事全黄了。"

> "并发不是越多越好——等所有等待间隙都被填满后，再加就是纯负担。"

## 🔍 排查实践

**线上并发性能问题，按四类排查：**

1. **任务是否被阻塞**：打日志看任务开始到结束时间，定位卡在哪里——是网络、数据库、锁，还是 RPC
2. **线程/协程是否堆积**：监控队列长度——如果持续增长，说明消费者跟不上生产者
3. **外部依赖是否变慢**：下游 API 延迟、数据库慢查询、连接池耗尽——瓶颈可能不在你的代码
4. **资源是否达到上限**：CPU 满了吗、内存够吗、文件描述符耗尽了吗、连接池满了吗

**实践中的黄金法则：**
- 先用小并发压测，找到吞吐提升开始变慢的"拐点"，那就是你的合理并发上限
- 设计时就考虑超时（永远不要无限等待）、重试（带退避）、降级（失败时返回兜底数据）
- 监控队列长度、任务耗时 P99、超时率、CPU/内存/连接数——这些比"并发数"本身更有意义

面试回答按"**现象 → 瓶颈类型 → 方案选择 → 并发上限 → 排查指标**"组织，会很落地。
