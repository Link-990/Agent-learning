# Skills 优缺点

## 🎭 面试官这样问

**面试官**："你们团队用 Skills 做 Agent 的能力沉淀，那你觉得 Skills 最大的好处是什么？最大的坑又是什么？"

**候选人**："好处是复用——写一次代码审查 Skill，以后所有 PR 都按这套流程走。坑嘛……就是维护成本，流程变了 Skill 要跟着改。"

**面试官**："这是标准答案。但我想听更深的东西。你说维护成本高——那你有没有想过，一个写坏的 Skill 造成的危害，可能比不用 Skill 还大？Skill 不是让你的 Agent 变聪明，是让它把你的经验稳定地执行出来。这个'稳定'是双刃剑——稳定地做好事，还是稳定地做错事？"

**候选人**："呃……确实，如果 Skill 里写了高危操作，模型会每次都执行。"

**面试官**："对。再考你一个场景：你们有十个 Skill，功能描述里都有'文档'两个字。用户说'帮我处理一下这个文档'，模型怎么知道该触发哪个？触发错了怎么办？你觉得十个 Skill 是比一个 Skill 更强，还是更弱？"

**候选人**："……可能更弱？触发冲突会让模型选错。"

**面试官**："行，有这个概念就好。那你讲讲怎么判断一个任务值不值得做成 Skill？"

## 💡 简要回答（面试标准话术）

"Skills 的本质是**把高频任务的经验产品化**。它像即插即用的技能 U 盘——你把代码审查、PPT 生成、上线检查这些经过验证的操作流程封装好，模型在匹配到对应场景时自动加载执行。好处是复用性强、流程稳定、节省上下文、能沉淀工程经验。但这里藏着一个核心矛盾：**经验可以复用，错误也会复用**。一个写坏的 Skill 会让模型稳定地做错事，十个边界模糊的 Skill 会让模型在触发时左右为难。

所以 Skills 不是越多越好。它适合沉淀**高频、边界清楚、流程稳定、有明确验收标准**的任务。变化快、低频、开放探索型的任务，为它们写 Skill 的维护成本可能超过收益。面试时要同时讲清复用价值和安全边界，不能只夸不防。"

## 📝 详细解析

### 一、核心矛盾：经验和错误会被同一条管道放大

Skill 这项技术的底层逻辑其实很简单：把过去成功的经验写下来，让未来的自己能复用。这在我们日常生活里再常见不过了——你第一次配环境踩了一堆坑，下次就照着笔记来；你第一次做上线发布紧张得手抖，后面就照着 Checklist 一步步走。Skill 就是把这种"经验笔记"从人的脑子里搬到 Agent 系统里，让模型在遇到同类任务时自动加载。

但问题就在这里。你的笔记写错了，你照着做也会错。而且因为你每次都照着笔记做，你错得很稳定——每次都错在同一个地方。Skill 也一样，它本质上是模型的"强上下文"。一旦加载，模型就会高度依赖这套流程做决策。**Skill 质量高，效率飞升；Skill 质量低，错误被系统性复制。**

这里用一个类比帮助你理解：Skill 就像工厂的自动化流水线。流水线设计得好，每个工位精准高效，产品良率 99%。但流水线设计有缺陷——比如某个焊接工位参数错了——那这条线生产的每一件产品都带有同样的焊接缺陷。你发现问题后要改流水线，还得停产调试。所以流水线适合生产标准品，不适合做高度定制化的单件。

同样道理，Skill 适合标准化操作（代码审查、上线检查、文档生成），不适合每次都不一样的新问题（用户随口问一个冷门问题，你非要写个 Skill 去处理它）。

### 二、优点：复用、稳定和工程沉淀——为什么大厂都在用

**优点一：真正意义上的复用，不是复制粘贴**

没有 Skill 的时候，让模型做代码审查这件事，你得在每次对话的 system prompt 里写一遍："请先阅读 diff，关注安全问题、性能问题、兼容性问题……"。如果团队有五个 Agent 产品都要做审查，你就要写五遍。更惨的是，A 同事写的 prompt 偏重安全，B 同事写的偏重性能，用户在不同 Agent 之间切换，体验割裂。

有了 Skill，你把代码审查的流程、工具使用规范、输出格式、禁止事项封装成一个独立的 Skill 文件。所有 Agent 在需要审查时加载同一个 Skill。这个 Skill 还可以版本化管理——1.0 只查安全，1.1 加了性能检查，1.2 加了国际化检查。每个 Agent 拿到的是同一份最新版。这就是工程复用的真正含义：不是 copy-paste，而是单一来源、统一更新。

**优点二：流程稳定，减少模型自由发挥**

大模型最大的优点是灵活，最大的毛病也是灵活。你让它审查代码，它有时候能给你写出一篇散文来，有时候就回一句"看起来没问题"。Skill 通过明确的步骤约束、输出模板和禁止事项，把模型的灵活性框在一个有边界的范围内。它告诉模型："这个任务你不需要创新，你按这张 Checklist 来，做完一项勾一项。"这种约束对新人特别友好——一个刚入职的工程师，用加载了代码审查 Skill 的 Agent，输出质量和老员工用同一个 Agent 几乎一样。

**优点三：节省上下文，配合渐进式披露**

把全部 Skill 的内容都塞进每次对话的上下文里是不可能的——token 贵而且窗口有限。Skill 的经典设计是**渐进式披露**：平时只加载 Skill 的简短描述和触发条件（可能就几十个 token），当系统判断用户意图匹配时，再加载 Skill 的主体内容，需要执行具体步骤时才读取相关资源文件。这套机制的上下文开销是"按需付费"，不是"一次性充值"。

**优点四：工程经验可以被版本化、被积累、被传承**

这一点往往被低估。一个团队做 Agent 做久了，会积累大量的"土经验"——某个 CI 环境里 `npm test` 会偶发超时、某个文档生成任务因为 Markdown 表格对齐总是出错、某个客户环境里 curl 必须加 `--insecure`。这些经验如果不系统化沉淀，每次新人来都要重新踩一遍坑。Skill 提供了一个天然的经验容器：你发现一个新坑，就在对应 Skill 里加一条检查或处理逻辑，Git commit 一下，全团队受益。

### 三、缺点：维护、触发和安全——三个绕不开的成本

**缺点一：维护成本——跟着业务一起老去**

Skill 不是写完就一劳永逸的。你的工具参数改了（MCP Server 升级了 tool schema），你的业务规则变了（上线审批流程从两步变成了三步），你的输出格式调了（报告里要加一个新字段），Skill 都必须跟着更新。否则它会稳定地产生"旧答案"——看似跑通了，实际漏了关键检查。更麻烦的是，当你有 50 个 Skill 的时候，你根本不知道哪个 Skill 引用了过时的工具，直到线上出了问题才后知后觉。

维护成本的一个隐形大头是**评测**。你怎么知道更新后的 Skill 比旧版更好？你怎么知道改了触发描述后不会误触发到别的场景？每改一次 Skill，理论上应该跑一遍回归评测集。但大部分团队做不到这一点。

**缺点二：触发歧义——Skill 太多，模型也会"选择困难"**

假设你有三个 Skill：一个叫"文档写作"，一个叫"PDF 处理"，一个叫"Markdown 转换"。它们的功能描述里都有"文档"两个字。用户说："帮我把这份文档处理一下。"模型需要从三个中选一个——它可能选了"Markdown 转换"，但用户其实想要"PDF 处理"。

触发歧义的根源在于，Skill 的触发是基于语义匹配而不是规则引擎。当多个 Skill 在同一个语义空间里"抢活干"的时候，模型的能力决定了匹配质量，而模型对于边界模糊的场景天然不擅长。更糟的是，如果选错了 Skill，模型的整个执行思路就走偏了，后续所有工具调用都跟着错。

**缺点三：过拟合流程——把灵活任务做死了**

有些任务需要模型保持探索的灵活度。比如用户说"帮我分析一下这个数据有什么异常"，你不需要让模型先做数据清洗、再画分布图、再跑统计检验、再写 2000 字报告——用户可能就想要一句话的结论："销售额在 3 月 15 号有一个 40% 的下滑，原因是当天服务器宕机了 6 小时。"但如果你的"数据分析 Skill"强制要求执行全部步骤，模型就会过度执行——画了一堆用户根本不想看的图，写了一堆用户不关心的分析，浪费了 token 也浪费了时间。

**缺点四：安全风险——Skill 不能突破权限边界**

这是最容易被忽视的问题。Skill 写的是任务流程，它天然倾向于"帮用户把事情做完"。但如果 Skill 里写了高危操作——"检查完成后自动提交代码"、"必要时执行清理命令"、"读取所有配置文件"——模型就会忠实地执行这些指令，而不管用户是否有权限、是否应该确认。

**记住一条铁律：Skill 只能沉淀方法，不能自动授权危险动作。** 权限控制必须在 Skill 之外——在 Harness 里、在 MCP Server 的权限策略里、在用户的主动确认里。Skill 可以说"检查代码后建议用户提交"，但不能说"检查后自动提交"。

### 四、工程例子：上线检查 Skill——收益和坑都在细节里

我们拿一个真实场景来分析。团队做了一个"上线检查 Skill"，流程如下：
1. 检查所有配置文件是否与环境匹配
2. 确认数据库迁移脚本已执行
3. 确认监控告警已配置
4. 确认回滚方案已准备
5. 确认灰度策略已制定
6. 运行冒烟测试
7. 输出检查报告和建议发布窗口

**收益非常明显**：以前上线靠老员工的脑子，每次发布评审要花两小时逐项确认，遗漏率大概 15%。现在新人也敢做发布评审了——Agent 带着 Skill 按 Checklist 逐项过，遗漏率降到 2%，评审时间缩短到 30 分钟。

**但坑也很快暴露：**

第一，公司发布了新的灰度平台，旧的灰度配置方式废弃了。Skill 没更新，导致连续三次发布都漏了灰度检查——因为 Skill 在检查一个已经不存在的东西，模型认为是"已配置"直接跳过了。

第二，Skill 的初版结尾有一条："如果所有检查通过，建议立即发布。" 有一天凌晨，一个新同事用 Agent 做上线检查，Agent 说"所有检查通过，建议立即发布"，他就真的点了发布——没注意到有一个监控告警的阈值配置错了（但 Skill 只检查了"是否配置"，没检查"配置值是否正确"）。结果上线后出了问题，回滚多花了 20 分钟。

**改进方案**：Skill 改为"生成检查报告和风险评估，由人工确认后执行发布"。检查项从"是否配置"升级为"配置值是否合理"。灰度检查绑定到发布平台 API 而非静态配置方式。这就是 Skill 迭代的正确姿势——不停发现问题，不停沉淀。

### 五、边界和风险：什么任务不适合做成 Skill

**不适合做 Skill 的任务类型：**

- **边界模糊的任务**：比如"帮我优化这个项目"，什么算优化？性能？代码质量？架构？边界太模糊，Skill 没法写清楚触发条件和执行流程。
- **低频任务**：一年多触发不了几次的任务，写 Skill 的投入产出比太低。Skill 的价值在于高频复用，低频不如直接写 prompt。
- **变化快的任务**：如果流程每个月都要变，Skill 的维护成本会吃掉所有收益。
- **开放探索型任务**：需要模型发挥创造力去探索新方案的任务，流程约束反而会压制好的输出。
- **一次性任务**：今天有人问了个偏门问题，你花半小时写个 Skill，这辈子可能再也不会触发。

**Skill 太多带来的系统问题：**

- 触发冲突增多，模型选错 Skill 的概率上升
- 上下文膨胀——即使只加载描述，100 个 Skill 的描述也要占不少 token
- 版本不一致——A 组更新了 Skill 但 B 组还在用旧版
- 权限难治理——每个 Skill 都要审计其是否有越权操作

**维护 Skill 的黄金法则：**

1. **定期淘汰**：Skill 不是越积越多越好，要有淘汰机制——三个月没触发过的 Skill 考虑归档。
2. **评测驱动**：每个 Skill 都要有对应的测试用例，改了就要跑回归。
3. **权限审计**：Skill 里不能出现"自动删除"、"自动提交"、"跳过测试"、"读取所有密钥"这类无条件高危动作。
4. **明确边界**：每个 Skill 必须声明：适用场景、不适用场景、需要的工具权限、失败处理方式。

### 🎯 面试总结

**必答三点：**

1. **Skills 的优点**：复用（单一来源，统一更新）、稳定（流程约束减少自由发挥）、节省上下文（渐进式披露）、沉淀经验（版本化积累团队智慧）。
2. **Skills 的缺点**：维护成本高（随业务变化持续更新）、触发歧义（多个 Skill 抢活）、过拟合流程（压制探索）、安全风险（无法替代权限控制）。
3. **适用边界**：适合高频、边界清楚、流程稳定、有验收标准的任务；不适合低频、模糊、快变、探索型任务。

**加分项：**
- 能说出 Skill 太多带来的系统性问题（触发冲突、上下文膨胀、版本不一致）
- 能提出 Skill 的评测和淘汰策略
- 能举出"上线检查 Skill"的具体迭代案例

**一句话**：Skill 把经验产品化——好的经验稳定地做好事，坏的经验稳定地做错事。设计 Skill 的核心不是让模型更聪明，是让你的 SOP 经得起重复执行。

## 🔍 排查实践

**评估一个 Skill 好不好，看四个指标：**

1. **触发准确率**：用户需要这个 Skill 时，它被正确触发的比例。如果命中率低于 80%，触发描述需要重写。
2. **任务完成率**：Skill 被触发后，流程走完并产生有效输出的比例。如果经常中途卡住，说明工具或步骤有问题。
3. **人工修改率**：用户对 Skill 输出的结果做了多少人工修改。修改越多，说明 Skill 流程与实际需求的 gap 越大。
4. **安全违规率**：Skill 执行中是否出现了越权操作、未确认的高危动作或信息泄露。

**排查 Skill 问题的标准流程：**

- 先确认是不是选对了 Skill（触发匹配是否准确）
- 再看 Skill 的流程步骤是否过时（工具参数、业务规则是否变了）
- 接着看工具调用结果是否符合预期（参数是否正确，返回是否正常）
- 最后看输出格式是否满足 Skill 约束

**设计新 Skill 时的 Checklist：**
- [ ] 这个任务过去三个月出现了多少次？（低于 10 次就别做）
- [ ] 任务流程是否稳定？（会不会下个月就变）
- [ ] 边界是否清楚？（能不能用一句话说清"什么时候用"和"什么时候不用"）
- [ ] 有没有高危操作？（自动删除、自动提交、绕过确认、读取密钥）
- [ ] 失败后怎么办？（流程中断、人工介入还是降级输出）
- [ ] 评测用例准备好了吗？（改 Skill 后怎么验证）
