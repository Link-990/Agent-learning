# 分块策略选型

## 🎭 面试官这样问

> 面试官：RAG 系统里文档分块怎么做？是不是设个 500 token 加 50 token 重叠就行了？

> 你：分块就是把长文档切成固定长度的片段，一般 500 token 左右，加一些 overlap 防止关键信息被切断。然后向量化入库就行。

> 面试官：那我问你，用户问「耳机买了 10 天，包装还在，可以退吗？」你的系统召回了一个 chunk：「耳机类商品自签收之日起 15 天内可申请无理由退货」。模型回答「可以退」。但原文还有后半句：「若商品存在人为损坏、缺少配件或超过 15 天，则不支持无理由退货。特殊促销商品以活动页面规则为准。」——你把限制条件切丢了，用户拿到一个过度承诺的答案。你说这个分块策略合格吗？

> 你：呃……那我把 chunk 设大一点，比如整章《售后政策》作为一个 chunk？

> 面试官：整章里有退款、换货、维修、发票、地区规则。用户问耳机退货，召回这一大坨，模型被维修规则、发票流程干扰，可能把「维修需要发票」写进退货答案里。而且一个 chunk 包含多个主题，向量表示会变成混合体，检索精准度也会下降。你明白分块的矛盾在哪了吗？

> 你：明白了。分块要在「召回精准」和「上下文完整」之间找平衡。太小丢条件，太大噪声多。还要看文档类型和问题粒度……

> 面试官：对。分块是 RAG 的地基。地基不稳，后面换 embedding、加 rerank、改 prompt 都只能部分补救。你能把常见分块策略和各自的适用场景讲清楚吗？

这才是面试要的深度——不是背参数，而是理解「知识粒度」的矛盾。

![分块策略的核心矛盾](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/03-RAG/03-chunking/01-概念总览.png)

## 💡 简要回答（面试话术）

> 分块是 RAG 里最容易被低估的环节。它的核心目标是把长文档切成适合检索和生成的知识单元，需要同时平衡两个互相矛盾的目标：chunk 小一点，主题集中，召回更精准，但容易切断上下文和限制条件；chunk 大一点，信息更完整，但向量表示混杂，噪声更多，也更占生成模型的上下文窗口。

> 常见的分块策略有五种：固定长度分块（适合 baseline 和结构混乱的文本）、按标题层级分块（适合技术文档和制度文档）、按段落分块（适合自然段规范的文章）、语义分块（按主题变化自动切分，效果好但成本高）、以及父子块检索（小块检索、大块生成，召回精准 + 上下文完整，是生产系统里的常见折中方案）。

> 选 chunk size 不能脱离文档类型、用户问题粒度和模型限制。FAQ 可以一问一答成块，技术文档按函数和接口切，制度文档按条款切，代码按类和文件切。最终策略要用真实 query 做评测：看正确片段能否被召回、排序是否靠前、模型最终是否答完整。

## 📝 详细解析

### 核心矛盾：召回精准 vs 上下文完整

分块不是简单的「把长文档切短」，而是在知识粒度和上下文完整度之间做 trade-off。这个矛盾可以这样记：

```
chunk 太小 → 精准但缺上下文 → 模型过度承诺
chunk 太大 → 完整但噪声多 → 模型张冠李戴
好的 chunk → 主题集中 + 足够回答问题
```

举个例子：用户问「耳机退货需要什么条件」，如果你的 chunk 只有「15 天内可退」，模型会自信地回答「可以退」，完全不提「配件齐全、包装完整、特殊促销除外」。用户照着操作被拒，就会觉得系统不靠谱。

但如果把整章《售后政策》——包含退货、换货、维修、发票、地区差异——全塞进 prompt，模型可能被维修流程干扰，把「维修需要发票」掺进退货答案，或者因为噪声太多直接忽略关键条件。

分块的黄金法则是：**每个 chunk 内部语义尽量单一，同时保留回答该类问题所需的完整条件、例外和来源信息。**

### 底层机制：分块影响两次关键压缩（类比教学）

把文档分块想象成**超市货架摆放商品**，分块至少影响 RAG 系统的两次关键压缩：

**第一次压缩：embedding。** 每个 chunk 被压成一个向量。如果 chunk 里混了退款、维修、发票三个主题，这个向量就会变成三件东西的「平均口味」——用户搜索「退款」，系统拿一个混合向量去匹配，就像顾客想买酱油，但你只给他一个「调味品混合包」，精准度自然下降。

**第二次压缩：上下文组装。** 召回的 chunk 塞进 prompt。生成模型虽然能读长上下文，但注意力和输出质量都受噪声影响。无关片段放得越多，模型越容易忽略关键条件，或者把多个片段的信息混在一起回答。这就像你同时给一个助手三份不同的说明书让他总结，他很可能张冠李戴。

所以分块的目标不是让每块长度平均，而是让每块的「信息密度」和「主题纯度」最大化。长度只是手段，知识粒度和完整性才是目的。

### 常见策略：五种切法，各有用武之地

**1. 固定长度分块。** 每 N 个 token 切一块，加 overlap。最简单、最稳定，适合早期 baseline 或结构混乱的纯文本。缺点明显：可能在标题中间、表格内部、代码块中间甚至句子里切断，破坏语义完整性。

**2. 按标题层级分块。** 适合技术文档、产品手册、制度文档。把文档的 h1/h2/h3 层级保留，每个小节单独成块，并把上级标题写入 chunk 前缀。例如 chunk 内容为「## 退货条件\n耳机签收后 15 天内……」，前缀带「售后政策 > 耳机 > 退货条件」。这样片段离开原文后仍有上下文锚点。

**3. 按段落分块。** 按自然段边界切分，配合长度控制——太短的段落合并相邻段落，太长的段落按句号二次切分。适合自然段规范的文章，但不适合业务文档（段落长度差异大、一个段落可能含多个主题）。

**4. 语义分块。** 基于内容主题变化切分：用 embedding 模型检测相邻段落之间的语义相似度，相似度骤降处就是分块边界。更贴近内容边界，但实现成本高，依赖模型质量，也需要评测验证。

**5. 父子块检索。** 这是生产里最值得掌握的折中方案。思想很简单：建两套索引——小的「子块」用于向量检索（精准度高），大的「父块」用于上下文组装（信息完整）。检索时命中子块，返回对应的父块给模型。用户问耳机退货，小 chunk 可能是「15 天无理由退货」，精准被召回；返回给模型的则是包含完整规则的大 chunk（含条件、例外、来源）。

### 工程例子：售后政策应该怎么切

看这段原文：

```
耳机类商品自签收之日起 15 天内，若商品外观完好、配件齐全、
包装完整，可申请无理由退货。若商品存在人为损坏、缺少配件或
超过 15 天，则不支持无理由退货。特殊促销商品以活动页面规则为准。
```

**❌ 差的分块：** 只切出第一句「15 天内可申请无理由退货」。能回答时间问题，但漏掉条件、例外。模型会过度承诺。

**❌ 更差的分块：** 把退款、换货、维修、发票整章塞进一个 chunk。召回后主题混杂，模型被噪声干扰。

**✅ 好的分块：** 把三句话放在同一个 chunk，标题标注为「售后政策 / 耳机 / 无理由退货」。这个 chunk 包含完整决策链：适用商品 → 时间范围 → 必要条件 → 不支持情况 → 特殊例外。用户问「买了 10 天能退吗」，模型能回答「在期限内，但需要外观完好、配件齐全、包装完整。特殊促销商品以活动规则为准。」

再举一个代码场景的例子。一段代码里有函数定义、参数说明、返回值、异常处理。如果按行切，模型可能只看到函数签名，看不到异常处理逻辑。好的做法是把整个函数（含文档注释、参数、返回值、异常）作为一个 chunk。

### chunk size 怎么选

选 chunk size 不能脱离实际场景，要同时看四个因素：

**第一，文档类型。** FAQ 一问一答成块（一般 100-300 token）。技术文档按标题/接口切（200-800 token）。制度文档按条款切（300-1000 token）。代码按函数/类切（200-1500 token）。

**第二，用户问题粒度。** 问题越具体（「E1024 报错怎么处理」），chunk 可以更小，因为答案通常集中在一个小段落。问题越综合（「退换修规则是什么」），chunk 需要带更多上下文。

**第三，模型限制。** embedding 模型的 max token 输入决定 chunk 上限——超了会被截断。生成模型的上下文窗口决定最终能放多少候选片段。长上下文模型不代表可以无脑大 chunk，噪声仍会影响生成质量。

**第四，评测结果。** 准备一批真实问题和标准相关片段，对比不同分块策略下：正确片段是否被召回？排序是否靠前？模型最终答案是否完整且不含无关信息？分块策略必须用失败样本迭代优化，不能凭经验值一次定死。

### 边界和风险

**小 chunk 容易丢条件。** 政策、法律、医疗、金融场景里，限制条件比主规则更重要——丢了条件就是过度承诺，后果不只是答错，可能是合规风险。

**大 chunk 容易污染向量。** 多个主题压成一个向量，检索变模糊，就像在一个拥挤的房间里找一个人。

**overlap 不是越大越好。** overlap 能缓解关键句被切断，但太大会增加索引体积、重复召回和上下文噪声。一般设 chunk size 的 10-20%。

**元数据不能丢。** 标题、版本、生效时间、权限、产品线——这些元数据常常决定片段是否适用。没有元数据，召回旧政策时模型很难判断时效性。

## 🎯 面试总结

- **必答 3 点：** ① 分块要平衡精准召回和上下文完整，小 chunk 精准但丢条件，大 chunk 完整但噪声多；② 五种策略——固定长度、按标题、按段落、语义分块、父子块检索，各有适用场景；③ chunk size 要看文档类型、问题粒度、模型限制和评测结果，不能拍脑袋。
- **加分项：** 能讲清「父子块检索」的原理和适用场景。能举出代码、合同、FAQ 等不同文档的分块方案。能讲出 overlay、元数据保留、评测驱动迭代。
- **一句话总结：** 分块是 RAG 的地基——地基不稳，后面换 embedding、加 rerank、改 prompt 都只能修修补补。

## 🔍 排查和实践建议

怀疑分块影响效果时，按这个顺序排查：
1. **抽样看 chunk 内容：** 是否主题集中？关键条件和例外是否被切到不同块？
2. **检查标题和元数据：** 片段离开原文后是否仍有上下文锚点？
3. **用真实 query 验证：** 正确 chunk 是否进入 top_k？排序是否靠前？
4. **看 prompt 里的上下文：** 是否有大量重复（overlap 过大）？是否有噪声（chunk 太大）？
5. **看模型回答：** 漏条件 → chunk 太小。答非所问/掺杂无关信息 → chunk 太大或主题混杂。

实践上，先用按标题或段落的结构化分块建 baseline，然后根据失败样本迭代调整。对于「召回要精准、回答要完整」的场景，优先尝试父子块。分块策略确定后，要对关键样本做回归测试，确保改 chunk size 不会破坏已有的好效果。
