# Agent 核心组件

## 🎭 面试官这样问

> 面试官：说说你理解的 AI Agent 核心组件有哪些？
> 你：嗯，就是大模型加上一些工具调用，比如搜索、数据库查询，再加个记忆存历史对话，基本就能叫 Agent 了。
> 面试官：那你觉得一个线上客服机器人，用户说"查一下我上周退款失败的订单，如果符合规则就重新提交"，你的 Agent 能做到吗？
> 你：理论上没问题，模型理解意图，调用查订单的工具，拿到结果，再调用退款接口就行。
> 面试官：那你有没有想过——如果查订单的工具超时了怎么办？如果退款接口返回 200 但业务上说"余额不足"怎么办？如果模型把"符合规则"判断错了，直接提交了退款，责任算谁的？
> 你：这……需要加一些错误处理和权限控制吧。
> 面试官：这就对了。Agent 的核心组件远不止"模型加工具"。你把一个会生成的模型和一套能对行动负责的工程系统搞混了。真正要回答的是：目标怎么定义、规划怎么产生、工具怎么控权、记忆怎么写和读、执行怎么兜底、反馈怎么闭环——六个组件缺一个，线上都会出事故。

这就是面试问 Agent 核心组件时，面试官真正想挖的东西。不是让你把"规划、记忆、工具"像名词解释一样背一遍，而是看你能否站在工程系统角度，讲清楚每个组件解决什么问题、不解决会出什么事故。

## 💡 简要回答（面试话术）

我理解 Agent 核心组件不能简单理解成"LLM + 工具调用"，而是一个围绕目标进行多步决策和行动的工程系统。从工程落地角度，至少需要六个组件协同工作。

第一是**目标层**，把用户的自然语言意图转换成可验收、可拆解的任务目标。比如"帮我处理退款"必须拆成身份校验、订单查询、政策匹配、退款申请、结果确认等可验收的子目标，不能只写成"让用户满意"这种模糊描述。目标不能验收，后面所有步骤都无法判断是否完成。

第二是**规划层**，决定先做什么后做什么，每一步要有明确的输入、输出和退出条件。第三是**工具层**，连接外部系统，但工具必须分只读和可写、有限流和幂等保护。第四是**记忆层**，分短期记忆（当前任务状态）和长期记忆（稳定偏好），写入必须谨慎，错误记忆会长期污染后续决策。第五是**执行控制层**，真正调用工具时做鉴权、参数校验、沙箱隔离和回滚——模型只提出"想调用什么"，能不能调由宿主系统决定。第六是**反馈闭环**，根据工具返回、规则校验和人工确认来修正下一步，没有反馈层，Agent 就会把失败当成功汇报。

总结来说，Agent 的本质矛盾是：**模型负责理解和生成，系统负责对行动负责**。组件设计的关键不是堆功能，而是确保每一层都有兜底——知道什么时候该停、什么时候该回退、什么时候该交给人。

## 📝 详细解析

### 核心矛盾：模型会生成，但系统要负责行动

咱们先想一个根本问题：LLM 本质上是什么？它是一个根据上下文预测下一个 token 的概率模型。它能写出非常像样的计划、能生成格式正确的工具参数、能组织出流畅的最终回答——但它没有"任务完成"这个概念，也没有"我的操作造成了真实后果"这个意识。

如果把 `LLM + 工具调用` 直接叫 Agent，你会漏掉三个致命问题。**第一个问题是目标没有被显式表示**。用户说"帮我处理退款"，到底是希望你先查原因、再判断政策、等用户确认后才提交——还是直接操作？模型没有目标状态机，它只会沿着上下文续写，可能在第一步就调用了退款接口。**第二个问题是工具没有工程边界**。接口返回 HTTP 200 不代表业务成功，参数通过语法校验不代表用户有权限，一次调用成功不代表可以重复调用。模型不管这些，它只看到"调用成功"四个字就继续往下走了。**第三个问题是缺少反馈闭环**。工具超时、返回空数据、返回权限错误时，Agent 必须知道下一步是重试、换工具、追问用户、还是停下来上报——而不是编一句"已为您处理完毕"。

这三个问题背后的根因是一样的：**LLM 是语言引擎，不是责任引擎**。把语言引擎直接暴露给真实业务操作，没有中间层兜底，翻车是必然的。

### 底层机制：一个工程化 Agent 的六层架构

我把它比作一家公司的运转，逐层拆开看。

**目标层——相当于公司的 OKR 拆解**。老板（用户）说"把客户满意度提上来"，你不能直接冲出去做事，要把这句话拆成可验收的子目标：响应时间降到 2 分钟、退款处理降到 1 天、投诉闭环率 95%。Agent 同理：把"处理退款"拆成 `verify_identity → query_order → check_policy → confirm_with_user → submit_refund → verify_result`，每一步有明确完成条件。面试官追问"目标层和规划层有什么区别"——目标定义"做成什么"，规划定义"按什么顺序做"。目标是验收标准，规划是执行路径。

**规划层——相当于项目经理排工期**。简单任务一次性生成计划，开放任务支持动态调整。一个有效规划每步都有三个要素：**输入**、**输出**、**退出条件**。缺了退出条件，Agent 可能在某一环无限循环。面试时你说"Agent 会规划"，面试官紧接着问"规划错了怎么办"——你要答出 replanning 和退出条件才能加分。

**工具层——相当于公司的外部供应商**。每个工具要有明确边界：名称、描述、参数 schema、权限等级（只读 / 低风险写 / 高风险写）、超时、重试、幂等键和错误码。工具描述越详细，模型越不容易选错——"用于发送通知邮件，仅内部邮箱，需用户确认后调用"比"发送邮件"靠谱得多。容易被忽略：**返回格式必须包含成功/失败语义**，模型需要知道 `{"status": "success"}` 和 `{"status": "failed"}` 的区别才能决定下一步。

**记忆层——相当于文档系统和白板**。短期记忆是白板：当前任务的目标、工具结果、待办动作。长期记忆是文档库：稳定偏好、项目约定、经验证的事实。写入要极其谨慎——把一次超时写成"该接口不稳定"，后续所有任务都会绕过它。每条写入经抽取、去重、来源标注和置信度判断。

**执行控制层——相当于合规和安保部门**。模型提出"调用退款接口，order_id=123"，执行控制层要校验：用户是订单主人吗？退款政策允许吗？有幂等键防重复吗？超时是重试还是暂停？**模型负责表达意图，宿主系统负责承担后果**——面试金句。高风险工具必须过人类确认节点：模型生成"建议退款 50 元"的提案，系统展示详情，人点"确认"才执行。

**反馈层——相当于质检和复盘机制**。工具返回不代表任务完成。反馈层要做：返回"调用成功"，检查业务字段是否真的成功；返回空数据，判断是真没有还是查询条件有问题；返回异常，决定重试、降级、追问还是停止；最终回答用规则或评审模型检查是否含幻觉、是否遗漏关键信息。没有反馈层，Agent 就是一个"输出即完成"的盲系统。

### 工程例子：自动周报为什么不能只让模型写

咱们走一个真实场景，帮你把这六层串起来。

假设你要做"自动生成项目周报"的 Agent。最粗糙的做法是什么？把团队聊天记录、提交日志丢给模型，让它总结一下。一周内你可能会发现：模型把上周已完成的工单写成本周进展，引用了一个根本不存在的需求编号，对高优缺陷只字不提，对风险描述用"持续推进中"这种废话覆盖。

用六层架构重新设计：

- **目标层**明确周报的输出标准：必须覆盖"本周进展（按项目）、风险与阻塞、下周计划"三个板块；风险必须标注负责人和截止日期；下周计划必须能关联到具体任务 ID。
- **规划层**生成执行计划：先读取日历确定本周范围 → 拉取代码仓库的合并请求 → 查询工单系统的需求状态变更 → 查询缺陷库的本周新增和关闭 → 读取会议纪要中的待办项 → 交叉校验生成草稿 → 检查数据完整性 → 输出终稿。
- **工具层**注册工具：`read_calendar`、`fetch_merge_requests`、`query_stories`、`query_bugs`、`read_meeting_notes`。每个工具有明确的参数和错误处理——如果工单系统挂了，工具要返回 `{"status": "error", "reason": "jira_unavailable"}`，而不是让模型猜。
- **记忆层**短期记忆记录：本周起止日期、已拉取的仓库列表、已查询的项目 ID，避免重复查询。长期记忆记录：周报模板偏好（"团队习惯按模块分组"）、项目代号到项目名的映射。
- **执行控制层**：所有工具都是只读的，不需要人类确认。但如果未来加了"自动发送周报邮件"，就必须确认。
- **反馈层**是重头戏。模型生成周报草稿后，反馈层做几件事：检查每个需求编号是否真实存在（去工单系统验证），检查风险项是否有负责人，检查下周计划是否为空，检查数据来源标注是否完整。如果工单系统查询失败，反馈层标记为"缺失数据"，强制模型在周报中写明"工单系统查询失败，本周缺陷数据可能不完整"，而不是用通用话术补齐。

面试里你能把这个例子从"模型直接写"到"六层架构兜底"讲清楚，比背任何定义都更有说服力。面试官最怕的就是候选人只会说"Agent 有规划、记忆、工具"，但问"如果工具返回失败怎么办"就卡住了。

### 边界和风险：什么时候不要让 Agent 自己决定

Agent 最危险的地带是**可写工具**。发邮件、提交代码、删除文件、发起转账、审批工单、修改数据库——这些动作一旦由模型自主决定，后果可能是灾难性的。工程上必须把工具分成三类：**只读**（查询类，模型可以自由调用）、**低风险写**（创建草稿、添加备注，需要日志审计）、**高风险写**（发送、删除、转账、审批，必须有用户确认、审计日志和回滚方案）。一个设计原则是：**模型可以建议，不能拍板**。

第二个风险是**上下文膨胀**。Agent 运行越久，累积的历史消息、工具结果越多。上下文太长导致模型注意力分散——被第 10 步的无关联返回带偏，忘了第 2 步的核心目标。解决方法是状态压缩：把已完成步骤摘要化，只保留关键证据和未解决问题，类比"你不需要记住会议每句话，只需要会议纪要"。

第三个风险是**无限循环**。模型可能反复搜索、总结、再搜索——每一步都看起来"在做事"，但实际上没有推进目标。这叫"看起来很努力但没有产出"的陷阱。系统必须设置硬性约束：最大步数（比如 20 步）、最大费用（比如 0.5 美元）、最大耗时（比如 3 分钟）、连续失败次数（比如 3 次失败就暂停），以及无进展检测（如果最近 5 步都没有产生新信息，强制停止或转人工）。

还有一个常被忽略的点：**不是所有场景都需要 Agent**。流程固定、规则明确、风险高的业务——工资计算、合规审计、审批流转——优先用 Workflow。强行把确定性流程交给模型自由决策，浪费成本还引入不稳定。面试时说"什么时候不用 Agent"，比盲目鼓吹 Agent 万能加分得多。

## 🎯 面试总结

- **必答3点**：① Agent 的核心组件是目标、规划、工具、记忆、执行控制、反馈闭环六层，不是"模型加插件"；② 每层的本质是权责分离：模型负责理解和生成，宿主系统负责校验、执行、审计和回滚；③ 组件之间必须有明确的输入/输出契约，任何一层失败都能追溯到具体环节，不能是黑盒。

- **加分项**：① 能把工具按风险等级（只读/低风险写/高风险写）做权限分级，说明高风险动作必须经过人类确认节点；② 能把无限循环问题讲透——不仅说"要设最大步数"，还能说出无进展检测、连续失败阈值、成本上限三重兜底；③ 能主动说"流程固定的场景用 Workflow，不是所有任务都该用 Agent"，体现工程判断力而非技术狂热。

- **一句话总结**：Agent 不是让模型自由发挥，而是用工程框架把模型的生成能力装进一个可控、可审计、可回退的责任系统——**模型负责聪明，系统负责靠谱**。

## 🔍 排查和实践建议

排查 Agent 线上问题，不要只看最终回答——看完整 trace：用户意图被抽取成了什么目标？每步选了哪个工具、传了什么参数？工具返回了什么？模型如何理解返回？失败触发了哪个兜底策略？终止条件在哪步生效？

排查顺序：先确认终止条件是否合理，再追溯工具调用，再检查目标和规划，最后看记忆和上下文是否被污染。

设计时先问四个问题：**目标能不能验收**？**工具能不能控权**？**状态能不能回放**？**失败能不能恢复**？四个问题答不上来，做出来的只是演示系统。
