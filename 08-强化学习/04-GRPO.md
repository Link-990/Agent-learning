# GRPO

面试官问："GRPO 和 PPO 最大的区别是什么？"候选人说："省掉 value model，用组内比较。"追问来了：省掉的是哪个模型？优势怎么估计？为什么要同一题采样多个答案？组内相对优势有什么好处也有什么风险？和 DPO 的离线偏好对比有什么不同？如果答不上来，说明你只是记住了 DeepSeek 用 GRPO 这个热点。

GRPO（Group Relative Policy Optimization，组内相对策略优化）的关键在于用**同一 prompt 下的一组多个候选回答做相对比较**，用组内奖励均值和标准差归一化来估计优势函数，从而彻底去掉对独立 value model 的依赖。它特别适合奖励可自动验证的任务（数学、代码、推理），但不适合奖励主观的场景。

## 🎭 面试官这样问

**面试官**："最近 DeepSeek 的 GRPO 很火，你说说它和 PPO 的本质区别。"

**候选人**："GRPO 不需要 value model，用同一 prompt 采样多个回答，组内归一化得到优势，更新策略。省显存。"

**面试官**："省显存，代价呢？你采样 8 个回答，每个回答 500 token，生成成本比 PPO 高 8 倍——这个账怎么算？"

**候选人**："确实采样成本高了，但省了一个价值模型的训练和维护……"

**面试官**："好。那你说，如果一道题采样了 8 个答案，8 个全对或者 8 个全错，GRPO 还能学吗？"

**候选人**："组内方差太小的话，优势信号很弱……基本学不到东西。"

**面试官**："对。那你怎么保证组内有多样性？温度和 top-p 设多少？如果全对，是不是采样策略需要调整？如果全错，是不是题目太超纲了？"

**候选人**："需要调温度让候选更多样化，同时控制题目难度……"

**面试官**："最后一个问题。奖励函数如果是规则判题器，模型有没有可能学会投机取巧——绕过推理过程直接猜答案？你怎么防？"

---

## 💡 简要回答

GRPO（Group Relative Policy Optimization）是一种**去掉独立 value model 的策略优化算法**。核心创新是：对同一 prompt 采样一组（比如 4-16 个）回答，用奖励函数给每个回答打分，然后在组内做**均值-标准差归一化**，用归一化后的相对分数作为优势函数来更新策略。

和 PPO 相比：PPO 用 value model 估计 baseline → 需要额外模型、显存、训练。GRPO 用组内均值替代 baseline → 省模型但多样本采样成本高。和 DPO 相比：DPO 用离线固定偏好对 → 不探索。GRPO 对当前策略在线采样 → 能探索，且奖励可以来自规则/判题器/环境交互而不仅是偏好数据。

GRPO 最适合**奖励可自动验证、可重复的任务**——数学推理（答案对错可判）、代码生成（测试用例可跑）、可验证问答。不适合奖励主观的场景——"亲切"、"有帮助"在组内比较时噪声太大。

---

## 📝 详细解析

### 核心矛盾一：PPO 的 value model 太重了

PPO 为什么需要 value model？因为优势函数的计算需要 baseline：`Advantage = Reward - Baseline`。没有 baseline，直接拿奖励做策略梯度，方差大到几乎学不动——同一个奖励 0.8 在"这道题通常能拿 0.9"和"这道题通常能拿 0.2"的语境下含义完全不同。value model 的作用就是估计"在这个状态下，按照当前策略，我预计能拿多少分"。

但 value model 在语言模型 RLHF 中是**又贵又难训**的：

- **额外显存**：大语言模型的 value model 通常和 policy model 同规模或接近，多一个模型的权重、优化器状态、激活缓存。
- **训练不稳定**：value model 要和策略同步更新，策略一变，value 的估计目标就变。目标在不断移动，value model 追着跑，像在猜一个不停变换的谜底。
- **信用分配困难**：value model 要对每个 token 位置估计未来期望回报。但语言生成中 token 级别的价值信号往往稀疏——只有最后几个 token 决定了奖励高低，前面的 token 都在"铺垫"，value model 很难学到有意义的区分。
- **超参敏感**：value loss coefficient、value model 学习率、GAE lambda 等多了一堆要调的东西。

GRPO 的核心思路是：**如果没有 value model，能不能用别的东西做 baseline？** 答案：让同一道题出多个答案，取它们的平均分当 baseline。

---

### 核心矛盾二：用"同题多答"替代 value model

GRPO 的训练数据单位和 PPO/DPO 完全不同：

```text
PPO:      (prompt, response, reward)                     —— 一个 prompt 一个回答一个分
DPO:      (prompt, chosen, rejected)                     —— 一个 prompt 一个好回答一个坏回答
GRPO:     (prompt, [answer_1, ..., answer_k], [r_1, ..., r_k])  —— 一个 prompt 一堆回答一堆分
```

**训练流程**：

1. 给策略模型一个 prompt。
2. 用当前策略采样 k 个不同的回答（比如 k=8，通过调整温度和 top-p 保证多样性）。
3. 对每个回答用奖励函数打分（可以来自规则、测试用例、判题器、奖励模型）。
4. 在组内做归一化：**优势_i = (r_i - mean(r_1..r_k)) / std(r_1..r_k)**
5. 用归一化后的优势更新策略：正优势的回答提高概率，负优势的回答降低概率。

**组内归一化解决了三个问题**：

- **Baseline 替代**：组均值就是自然 baseline。比平均分高的就是好回答，比平均分低的就是差回答。
- **跨题尺度统一**：一道简单题所有答案都在 0.9-1.0 之间，一道难题在 0.1-0.3 之间。不做归一化，简单题的"中等回答"（0.95）的原始奖励可能比难题的"优秀回答"（0.25）高得多，模型会过度学习简单题。归一化后，每道题内部比，全局信号更公平。
- **自适应难度**：标准差天然反映题目的区分度。区分度高的题目（答案有对有错、分差大）标准差大，归一化后优势的绝对值反而被压缩；区分度低的题目（全对或全错）标准差小，归一化后微弱差异被放大——但这个放大可能是噪声。

### 省掉 value model 的代价是什么

没有免费的午餐。GRPO 省掉 value model（省显存、省训练的复杂度），但代价也很明确：

**代价一：采样成本暴增。** 每个 prompt 要生成 k 个回答。k 通常取 4-16。如果每个回答平均 300 token，k=8 就是 2400 token 的生成量——是 PPO 单次采样的 8 倍。对于大模型推理，生成 token 的成本远高于处理 token 的成本，这是 GRPO 最大的隐性花销。

**代价二：奖励函数必须可靠。** PPO 的 value model 虽然难训，但它毕竟是从数据中学出来的，有一定泛化能力。GRPO 完全靠奖励函数驱动。如果奖励函数有漏洞——比如代码判题器只检查最终输出不检查过程、数学判题器只比对数值不比对推导——模型会学会投机取巧。**错误奖励比没有奖励更危险，因为它稳定地强化错误路径。**

**代价三：组内方差决定学习效率。** 如果一组全对（所有 r_i = 1.0），std = 0，归一化退化成除以 0（实际实现通常会加 epsilon 或跳过）。如果一组全错（所有 r_i = 0），同样没有学习信号。如果一组只有一个对、七个错，对的获得巨大正优势，错的一样惨——但"差点对"和"完全瞎写"的错被同等惩罚，信号粒度粗。

**代价四：多样性需求和成本矛盾。** 要组内有差异才能学到东西，所以温度不能太低、top-p 不能太窄。但温度高了低质量回答增多，浪费 token。温度和 k 的 tuning 是双刃剑——温度低 = 候选相似 = 组内方差小 = 学不动；温度高 = 废物多 = 成本浪费。

### 工程例子：代码题强化训练

最典型的 GRPO 应用场景：训练代码模型解 LeetCode 风格的算法题。

**设置**：
- 每道题采样 k=8 个回答（不同的代码实现）
- 奖励函数 = 编译通过（0 或 0.3）+ 样例测试通过（0 或 0.3）+ 隐藏测试通过率（0 到 0.4）
- 温度 0.8，top-p 0.95，保证组内多样性

**训练动态**：

- 早期：模型弱，8 个回答可能 6 个编译失败、1 个编译通过但逻辑错、1 个勉强通过部分测试。组内方差大，GRPO 强力推高那个"勉强对的"概率。
- 中期：模型强了，8 个回答 4-5 个通过全部测试。组内仍有方差，GRPO 继续优化——推高"代码更优雅、更高效"的变体（如果奖励函数包含了效率维度）。
- 晚期：模型很强，8 个回答 7-8 个满分。组内方差变小，GRPO 信号减弱——这时候可能需要上更难的题或者更细粒度的奖励（比如代码风格、注释质量）。

**需要警惕的陷阱**：

- **答案泄漏**：模型发现判题器只检查最终 return 值 → 学会硬编码答案而不是推导过程。
- **过拟合判题器**：隐藏测试覆盖不足 → 模型学会了恰好通过已知测试的 trick。
- **语言退化**：KL 约束太弱 → 模型语言质量下降，代码注释变得敷衍或消失。

**好的奖励设计应该多维度**：最终答案正确性 + 推理过程完整性 + 代码格式合规 + 效率（时间复杂度）+ 安全性（不执行危险操作）。单独依赖任何一个维度都可能被 exploit。

### 和 PPO、DPO 的全面对比

| 维度 | PPO | DPO | GRPO |
|------|-----|-----|------|
| **Value Model** | 需要 | 不需要 | 不需要 |
| **奖励模型** | 需要 | 不需要 | 可选（规则/模型都可） |
| **参考模型** | 需要（KL） | 需要（KL 的影子） | 需要（KL） |
| **采样方式** | 在线（当前策略） | 离线（固定偏好数据） | 在线（当前策略） |
| **每次采样数** | 1 个回答 | 1 对偏好 | k 个回答（4-16） |
| **优势估计** | Value Model + GAE | 不需要（隐式在损失里） | 组内归一化 |
| **探索能力** | 强（在线采样） | 弱（离线数据） | 强（在线多样本采样） |
| **工程代价** | 最高（4 模型） | 最低（2 模型） | 中等（3 模型 + 高生成成本） |
| **适合任务** | 通用对齐 | 风格/安全/帮助性对齐 | 可验证任务（数学/代码） |
| **数据需求** | 奖励模型训练数据 | chosen/rejected 偏好对 | 可自动打分的 prompt 集 |

**选择指南**：

- 你有大量高质量 chosen/rejected 偏好数据 + 奖励是主观的 → **DPO**
- 你有可靠奖励模型或规则奖励 + 需要在线探索 → **PPO**（预算充足）或 **GRPO**（想省 value model）
- 你的任务是数学推理、代码生成、可自动验证 → **GRPO** 是天然适配的首选
- 你没有偏好数据也没有自动奖励函数 → 先去解决数据/奖励问题，别急着选算法

### 失败模式和排查

**常见失败**：

1. **组内方差太小**（全对或全错）→ 学不到有效信号。排查：检查题目难度分布，增加温度提升多样性，或对全对/全错的组跳过更新。
2. **采样多样性不足** → 候选高度相似，组内比较没有意义。排查：提高温度、放宽 top-p、增大 k。
3. **KL 太弱** → 语言质量下降、出现重复、格式崩塌。排查：加大 KL 系数，或在奖励中加入格式/语言质量维度。
4. **KL 太强** → 策略几乎不动，等于白训。排查：减小 KL 系数。
5. **奖励函数被 exploit** → 模型学会了投机取巧。排查：检查高分回答的真实质量，增加奖励维度，加入对抗性测试。
6. **题目难度分布不合理** → 简单题全对（无信号）、难题全错（无信号）、只有中等难度题有学习价值。排查：动态调整题目池，根据模型当前能力匹配题目难度。

**排查的话术模板**：

"GRPO 排查我先看三条线。第一，奖励分布——每组是否有明显的正负样本区分？组内均值、标准差是多少？全对/全错的组占比多大？第二，生成质量——输出长度、KL、重复率、语言流畅度有没有退化？第三，任务指标——pass rate、测试准确率是否在持续上升？如果 pass rate 涨但人工抽检发现答案在投机取巧，回头审查奖励函数。"

## 🎯 面试总结

### 必答三点

1. **GRPO 用"同题多答 + 组内归一化"替代 value model**：对同一 prompt 采样 k 个回答，用奖励函数打分后组内做均值-标准差归一化，用相对优势更新策略。省掉 value model 的显存和训练复杂度，但生成 token 成本变为 k 倍。

2. **GRPO 的边界是奖励必须可自动验证**：数学（答案判对错）、代码（测试用例 pass/fail）、推理（规则校验）是天然场景。奖励主观的任务（聊天质量、创造性评估）组内比较噪声大，不适合 GRPO。

3. **组内方差是 GRPO 的生命线**：方差太小（全对/全错）= 学不到；方差来自多样性（温度、top-p、题目的区分度）。温度调得太低候选相似、调得太高候选太烂——需要在多样性和质量之间找平衡。

### 加分项

- 能写出组内归一化公式：`advantage = (r - mean) / std`
- 能分析"全对组"和"全错组"的处理策略（跳过 or epsilon）
- 能对比 GRPO 和 DPO/PPO 的适用边界，给出具体场景的选择建议
- 能举例说明奖励函数设计的陷阱（答案泄漏、过拟合判题器）
- 能定量估算 GRPO 的生成成本：k × avg_response_tokens × cost_per_token

### 一句话

**GRPO 的价值模型不在显存里，而在采样策略里——每一次"同题多答"就是一次局部的 self-play，让模型从"这道题我考了几分"变成"这道题我比其他版本的自己考得好还是差"，用相对比较替代了绝对估值。**

## 🔍 排查实践

**第一步：画奖励分布图。** 统计所有 group 的组内均值、标准差、最大值、最小值的分布。理想状态是：大部分 group 均值在中等范围（0.3-0.8 之间）、标准差在 0.1-0.3 之间，表示题目有区分度、组内有差异。如果均值集中在 0 或 1，标准差接近 0——题目太简单或太难，需要调整题目池。

**第二步：监控生成指标。** 输出长度是否稳定？KL 散度是否在目标范围？重复率（ngram repeat）是否上升？entropy 是否快速下降？任何一个异常都可能指向采样策略需要调整。

**第三步：做人工抽检。** 从高优势回答、低优势回答、中等优势回答中随机各抽 20 条，人工判断"好的回答确实更好吗"、"差的回答确实更差吗"。如果高优势但人工觉得差 = 奖励函数有问题。如果低优势但人工觉得不错 = 奖励函数漏掉了好的模式。

**第四步：审查奖励函数。** 对代码任务，检查隐藏测试的覆盖率；对数学任务，检查判题器是否只比对最终答案；对需要格式的任务，检查格式规则是否细致到不会误伤合理变体。

**第五步：调整采样策略。** 如果组内方差太小 → 提温度、宽 top-p、加 k。如果低质量回答太多浪费成本 → 设最低质量阈值，过滤掉明显无意义的回答后再训练。

**面试话术**："GRPO 排查的核心是确认组内比较是否有效。奖励分布决定学习信号质量——全对全错都没用，要有区分度。生成指标决定策略健康度——KL、长度、重复率任何一项异常都要回头调参数。人工抽检是最终验证——奖励函数说有差异但人看不出来，那奖励函数就是错的。"
