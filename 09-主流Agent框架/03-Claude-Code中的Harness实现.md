# Claude Code 中的 Harness 实现

## 🎭 面试官这样问

**面试官**："Claude Code 我用了，体验很好。它为什么能像真正的工程师一样改代码、跑测试、根据报错继续修——而不是像普通 chatbot 一样只给建议？"

**候选人**："因为 Claude 模型本身很强，推理能力好。"

**面试官**："模型强是必要条件，不是充分条件。GPT-4 也很强，但扔一个 GitHub repo 给它，它只能给你一段补丁建议，不能自己去文件系统里搜文件、读代码、精准编辑、跑测试、看报错、再改。这些能力是从哪来的？"

**候选人**："呃……它有一些内置工具？"

**面试官**："对，工具。但不是简单的工具列表，是一套完整的 Harness。你知不知道 Claude Code 有哪些核心工具？`Edit` 工具为什么要求传 old_string 和 new_string，而不是直接传整个文件内容？Bash 命令为什么不会把整个项目删掉？长任务里 `TodoWrite` 是干嘛的？如果我本地有未提交的修改，它会不会覆盖掉？"

**候选人**："这些……我确实没细想过。"

**面试官**："好，那我们从 Harness 的角度拆一拆 Claude Code。记住：Claude Code 的能力 = 模型推理 + Harness 执行。模型负责'知道怎么做'，Harness 负责'安全地做到'。缺了任何一个，它也就是个聊天机器人。"

## 💡 简要回答（面试标准话术）

"Claude Code 的能力来源是**模型推理 + Harness 执行**的双引擎。模型提供了代码理解、问题定位和修改方案的推理能力，但真正让它从'给建议的辅助工具'变成'能动手的工程师'的，是它的 Harness 层。

Claude Code 的 Harness 提供了六类核心能力：**搜索定位**（Glob、Grep）、**精确阅读**（Read）、**安全编辑**（Edit、Write）、**受控终端**（Bash）、**任务管理**（TodoWrite）、**知识注入**（Skill、CLAUDE.md）。这些工具不是简单地暴露给模型，而是在执行前做校验（路径白名单、编辑唯一性检查）、执行中做隔离（工作目录限制、命令类型约束、超时控制）、执行后做裁剪（日志截断、关键信息提取）和审计（操作记录）。

其中 `Edit` 的设计是点睛之笔——它要求传 old_string 和 new_string，编辑器通过精确匹配 old_string 来定位修改位置。这比整文件重写安全得多，既避免了误覆盖（匹配不到就不改），也减少了格式漂移（只改了该改的部分）。面试时如果能把这个设计讲透，说明你真的理解了 Harness 的精髓。"

## 📝 详细解析

### 一、核心矛盾：代码任务需要持续感知环境，但模型天生看不到环境

普通聊天模型处理代码任务就像**一个戴着降噪耳机、蒙着眼睛的程序员**。你通过文字告诉他项目结构，他凭记忆和推理给你一段补丁。他不知道你的 `node_modules` 里装了哪个版本，不知道 `package.json` 的 scripts 是怎么写的，不知道测试框架的配置文件在哪，也不知道你上一分钟刚改了什么。他只能基于你告诉他的信息（往往不完整）和你训练数据里见过的模式（可能已经过时）来给出建议。

真实世界的代码任务不是"一锤子买卖"。一个典型的工作流是：搜索 → 阅读 → 理解 → 修改 → 验证 → 根据报错继续修改 → 重复直到通过。每一步都要看到真实的反馈，才能做出正确的下一步决策。

Claude Code 的 Harness 解决的就是这个问题：**它把模型的"盲人摸象"变成了"睁开眼睛干活"**。通过 Harness 提供的工具，模型能主动探索代码库、精准获取文件内容、在安全边界内修改代码、运行测试获取反馈、根据反馈迭代修复。核心是建立了一个**观察—计划—行动—验证—再观察**的闭环。

### 二、逐层拆解 Claude Code Harness 的六大能力

**第一层：搜索定位——Glob 和 Grep**

在大型代码库里，让模型靠"猜"来找文件是灾难性的。一个项目可能有几百个文件，模型猜文件路径的准确率可能不到 30%。Glob 和 Grep 让模型能像工程师一样用工具搜索：`Glob(pattern="**/*.test.ts")` 找到所有测试文件，`Grep(pattern="login", path="./src/auth/")` 找到认证模块中所有跟登录相关的代码。

这两个工具看似简单，但它们是 Harness 闭环的"眼睛"。没有搜索，模型就无法感知代码库的真实结构，只能基于训练数据中的"常见模式"猜测——这在项目结构稍微特殊一点时就会出错。面试时一个很好的追问点：**为什么不直接用 `ls` 和 `grep` 的 Bash 命令？** 因为 Bash 命令是通用工具，权限控制粒度粗（一个 Bash 工具什么都能做），而 Glob 和 Grep 是专用工具，天然就限制了只能做搜索和文件匹配，减少了权限风险。

**第二层：精确阅读——Read**

Read 看起来就是"读一个文件"，但它的工程细节值得注意。首先，Read 返回的是精确内容（带行号），不是模型的记忆或近似内容。这解决了模型"基于幻觉改代码"的问题——模型必须先 Read 再 Edit，而不是凭记忆猜文件内容。其次，Read 支持 offset 和 limit 参数，大文件可以分段读，避免上下文浪费。

一个关键的 Harness 约束是：**修改已有文件之前必须先 Read。** 这不是建议，是硬性规则。如果模型跳过 Read 直接 Edit，Harness 会拒绝执行。为什么？因为如果不先读，模型改的是它"记忆中的版本"，而不是"磁盘上的真实版本"——可能文件已经被别人改了，或者模型记忆中的代码是错的。

**第三层：安全编辑——Edit 和 Write（这个设计值得单独讲深）**

`Edit` 是 Claude Code Harness 最精妙的设计之一。它的接口是 `Edit(file_path, old_string, new_string)`，其中 `old_string` 是你想要替换的原始代码片段，`new_string` 是替换后的内容。

为什么不用更直观的 `Edit(file_path, line_start, line_end, new_string)`？因为行号是脆弱的——文件被改过一次后行号全变了。为什么不用 `Edit(file_path, new_content)`（全文件替换）？因为全量替换风险太大，容易因为格式漂移（模型改变了没想改的空白、注释、import 顺序）引入意外变更。

`old_string` 匹配机制的妙处在于：它是一种**自验证的安全编辑**。如果 `old_string` 在文件中是唯一的，Harness 精确替换那一处；如果不唯一，Harness 拒绝并报错"匹配不唯一，请提供更多上下文"；如果不存在，Harness 拒绝并报错"文件可能已变化，请重新 Read"。这三层防护让误改率大幅下降。

`Write` 是完整覆盖文件内容，主要用于新建文件或需要完全重写的场景。但 Write 也有安全措施：如果目标文件已存在且未被 Read 过，拒绝覆盖。

**第四层：受控终端——Bash**

Bash 让模型能运行命令，这是 Claude Code 区别于普通代码助手的核心能力之一。但它的 Bash 不是原生的 shell，而是被 Harness 严密管控的执行环境：

- **工作目录限制**：命令默认在项目根目录执行，不能随意切换到系统目录
- **命令审核**：危险命令（包含 `rm -rf`、`git push --force`、`chmod 777` 等）会被拦截或要求确认
- **超时控制**：长运行命令有超时限制，防止无限等待
- **输出裁剪**：返回的命令输出会被裁剪（终端返回几千行日志，Harness 提取关键部分给模型）
- **后台任务管理**：支持后台运行长时间任务（如 dev server），Harness 管理其生命周期

最妙的是**测试反馈闭环**：模型改完代码后调用 Bash 跑 `npm test -- --testPathPattern=login`，如果失败，Harness 把退出码、失败栈和关键错误信息结构化返回给模型。模型基于这些真实反馈继续修改，形成"改一点，测一下，看报错，再改"的迭代节奏。这就是它像工程师的原因——工程师也是这么干活的。

![Claude Code 执行链路](https://gitee.com/linkio666/image/raw/main/09-agent-frameworks/03-claude-harness/02-运行机制.png)

**第五层：任务管理——TodoWrite**

代码任务往往有很多步骤：修改 A 文件 → 更新 B 文件的引用 → 更新测试 → 运行测试 → 检查 linter。模型在多步操作中很容易忘记"我做到哪了"和"还有什么没做"。TodoWrite 让模型维护一个任务列表，完成一项勾一项。

这是一个看似简单但极其有效的设计。它解决的不只是"组织问题"，更是**上下文管理问题**——当一个任务跨越十几步操作、几十次工具调用时，模型有限的上下文窗口很难记住所有步骤的状态。TodoWrite 把"进度"外化到任务列表中，模型每次只需要看"还有哪几项没做"，而不需要在有限的"脑容量"里记住全部细节。

**第六层：知识注入——Skill 和 CLAUDE.md**

Claude Code 支持通过 CLAUDE.md（项目级配置）和 Skill（任务级能力包）注入特定领域的知识和规范。CLAUDE.md 像项目的"README for AI"：代码风格、测试命令、构建流程、常见约定写在这里，模型每次启动都会加载。Skill 则是按需加载的能力包：处理 PDF、生成文档、验证部署——在需要时才注入上下文。

### 三、权限和安全边界——为什么 Claude Code 很少搞出大事故

Claude Code 的安全不是靠"模型自觉"，而是靠 Harness 的分层防御：

1. **路径白名单**：模型只能访问项目目录内的文件，不能碰 `/etc/passwd`、`~/.ssh/`、`.env` 等敏感路径
2. **编辑安全**：Edit 的 old_string 唯一性校验 + 修改前必须先读，双重保护
3. **Git 保护**：自动提交前的安全检查——查看 status、diff、log，防止把密文、大文件、未完成的修改提交进去。高危 Git 操作（reset --hard、clean -fd、force push）需要确认或被限制
4. **命令黑名单**：Bash 工具内置了危险命令的拦截规则
5. **确认机制**：高风险操作在执行前要求用户确认
6. **审计日志**：每次操作都有记录，可以回溯

这些防御组合在一起的效果是：即使模型判断失误或被用户诱导，它能造成的破坏也在可控范围内。它可能改错一个函数、跑废一个测试、生成一段烂代码——但它删不掉你的整个项目，提交不了你的密钥文件，也连不上你的生产数据库。

### 四、上下文管理和长任务处理

Claude Code 面临的一个核心挑战是上下文窗口管理。一个代码任务可能涉及几十个文件、几百次工具调用、几万行日志。如果不管理，上下文会在几分钟内被塞满。

Harness 的上下文管理策略：
- **按需加载**：模型先搜索再阅读，不一次性加载无关文件
- **输出裁剪**：测试日志只返回关键错误和摘要，不是完整输出
- **步骤分解**：长任务拆成多个子任务，配合 TodoWrite 管理进度
- **增量编辑**：Edit 只传修改片段而非整个文件，减少每次调用的上下文消耗

### 五、工程例子：修复登录测试失败的完整过程

用户说："登录测试 red 了，帮我修。"

**TodoWrite**：模型列出任务——[搜索测试] [读取测试文件] [读取实现文件] [定位问题] [修改代码] [运行测试] [验证修复]

**Grep**：搜索 "login" 相关的测试文件 → 找到 `tests/auth/login.test.ts`

**Read**：读取测试文件和对应的源文件 `src/auth/login.ts` → 发现测试期望返回 `{token, user}` 但实现只返回了 `{token}`

**Edit**：用 Edit 精确替换相关代码行，增加 user 对象的返回

**Bash**：运行 `npm test -- --testPathPattern=login` → 测试通过

**TodoWrite**：所有任务完成

整个过程模型做了约 8 次工具调用，每次都是真实的代码库交互。Harness 确保每一步都在安全边界内：搜索只读、Read 只读、Edit 在项目目录内且唯一匹配、Bash 在工作目录限制内且超时受控。

### 六、失败模式和排查方式

**Claude Code 类 Agent 的常见失败：**

| 失败类型 | 原因 | Harness 能做的 | Harness 做不到的 |
|---------|------|--------------|-----------------|
| 搜索不充分 | 模型没搜到关键文件就猜 | 提供搜索工具，但不能强迫模型搜全 | 需要模型自身推理判断是否需要更多搜索 |
| 上下文理解不足 | 读取的代码片段太少 | 提供精确 Read，但不能判断模型是否理解正确 | 需要模型正确分析代码逻辑 |
| 编辑匹配失败 | old_string 不唯一或已过期 | 拒绝执行并要求重试 | 依赖模型重新选择合适的 old_string |
| 测试环境缺依赖 | 项目需要特定环境才能跑测试 | 执行测试命令，返回错误信息 | 环境的搭建不在 Harness 职责内 |
| 日志过长 | 测试输出几万行 | 裁剪输出，但不能判断哪些是关键信息 | 裁剪策略可能丢掉重要错误信息 |

排查时的关键原则：**看完整轨迹而非只看最终结果。** 检查模型搜了什么、读了什么、为什么改这一处、测试是否真的运行了、失败日志是什么、修改后模型有没有正确地"观察"到变化。

### 🎯 面试总结

**必答三点：**

1. **Claude Code = 模型推理 + Harness 执行**。模型负责"知道怎么做"，Harness 负责提供"眼睛和手"并确保安全。缺一个就是聊天机器人，合起来才是工程师。
2. **Edit 是 Harness 的精华**：old_string 精确匹配提供了自验证的安全编辑——匹配不到不改、不唯一不改、文件变了不改。这比整文件重写或行号定位安全得多。
3. **六层能力体系**：搜索定位（Glob/Grep）→ 精确阅读（Read）→ 安全编辑（Edit/Write）→ 受控终端（Bash）→ 任务管理（TodoWrite）→ 知识注入（Skill/CLAUDE.md）。每一层都有对应的 Harness 安全策略。

**加分项：**
- 能解释为什么 Edit 不用行号而用 old_string 匹配
- 能说出 Bash 工具的权限限制策略（工作目录、命令审核、超时、输出裁剪）
- 能对比 Claude Code Harness 和通用 Harness 的异同

**一句话**：Claude Code 把 Harness 做到了"让模型像工程师一样干活但不让模型像黑客一样搞破坏"的精妙平衡——有眼睛、有手、有安全绳、有进度条。

## 🔍 排查实践

**当 Claude Code 改错代码时，按这个顺序排查：**

1. **搜了吗？** 检查模型是否做了充分的搜索。如果它没搜就直接猜文件位置，大概率找错文件。
2. **读了吗？** 检查模型是否先读了文件再修改。如果没读就改，它改的是"记忆中的版本"。
3. **Edit 匹配对了吗？** old_string 是否唯一？是否把不该改的地方也匹配了？
4. **测了吗？** 修改后是否真的运行了目标测试？命令参数对不对？测试环境是否就绪？
5. **看报错了吗？** 测试失败后，模型是否正确读取了错误栈并针对性修复？

**使用 Claude Code 的最佳实践：**

- 保持修改粒度小：一次 Edit 改一处逻辑，而不是一次改五个文件
- 让它先写 TodoWrite：复杂任务先规划再执行
- 写好 CLAUDE.md：把项目约定、测试命令、代码风格写清楚
- 评审每次修改：用 `git diff` 检查 Agent 的改动
- 不要关闭测试：让 Agent 总是运行测试来验证修改
