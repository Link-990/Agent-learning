# KV 缓存

大模型服务上线后，经常出现一个现象：短问题回答很快，长对话越聊越慢；并发一高，显存突然被吃满；首 token 等很久，但开始输出后速度还可以。面试官问 KV Cache，不是只想听"缓存 K 和 V，加速推理"，而是要看你能不能解释 prefill、decode、显存成本和多轮对话为什么会越聊越贵——以及你打算怎么管。

![KV 缓存概念总览](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/02-llm-framework/05-kv-cache/01-概念总览.png)

---

## 🎭 面试官这样问

**面试官**："你说你优化过推理服务。我问你，KV Cache 缓存的到底是什么？"

**候选人**："缓存的 Key 和 Value，这样 decode 的时候不用重复算历史 token。"

**面试官**："好。那为什么不缓存 Query？历史上为什么不需要以前的 Q？"

**候选人**："呃……因为 Q 是当前 token 去查别人的，历史的用不上？"

**面试官**："对。下一个问题——一个用户发了 8000 token 的合同过来，总结风险点，然后追问了 5 轮。你告诉我，第 5 轮的时候服务端发生了什么？显存为什么涨了？"

**候选人**："因为每一轮都带着完整历史，上下文越来越长，KV Cache 也越来越大……"

**面试官**："那如果你把历史做了摘要，删了 3000 token 的原文但保留了旧的 KV Cache，会出什么问题？"

**候选人**："……"

**面试官**："还有，你说你用 PagedAttention 优化了推理。那你告诉我，MQA、GQA 和 PagedAttention 分别解决什么问题？一个是减少缓存大小，一个是管理缓存碎片——这俩的区别在哪儿？"

---

## 💡 简要回答

KV Cache 是 Transformer 自回归推理中的缓存机制。它把每一层注意力计算中**历史 token 的 Key 向量和 Value 向量**保存下来，让后续生成新 token 时只需计算当前 token 的 Q/K/V，然后用当前 Q 去和历史 K/V 做注意力运算，避免每生成一个 token 就重新计算全部历史上下文的 K/V。

推理因此被分成两个阶段：**prefill**（一次性处理 prompt，建立 KV Cache）和 **decode**（逐 token 生成，复用缓存）。代价是显存随层数、上下文长度、batch size、KV head 数线性增长。长上下文、高并发、多轮对话场景下 KV Cache 往往是 OOM 的真凶。

---

## 📝 详细解析

### 核心矛盾：用显存换重复计算

自回归生成有一个结构性的特点：**第 t 个 token 必须先生成出来，才能生成第 t+1 个**。每一步计算注意力时，新 token 需要"关注"前面所有历史 token。如果没有缓存，每一步都要把全部历史重新做一遍 K/V 投影和注意力计算——第 100 个 token 要算 100 个位置的注意力，第 500 个要算 500 个，计算量随生成长度**平方增长**。

KV Cache 的思路非常朴素：**已经算过的东西不要重算**。历史 token 的 Key 和 Value 在 Transformer 的每一层已经产生，只要这些历史 token 不改变（它们的位置不变、内容不变），它们在每一层的 K/V 就不会变。把它们存起来，新 token 只需要：

1. 计算自己的 Q/K/V
2. 用新 Q 去和**所有历史 K**（从缓存读）做点积得到注意力分数
3. softmax 后加权**所有历史 V**（从缓存读）
4. 把新 token 的 K/V 追加进缓存

**一次计算，终身复用**。第 500 个 token 的计算量和第 10 个差不多——都是新 token 做一次投影 + 和缓存做注意力。

但你马上能看出代价：**每多一个 token，缓存就多存一层的 K 和 V**。上下文越来越长 → 缓存越来越大 → 显存迟早撑不住。这就是用显存换计算的核心 trade-off。

---

### 为什么不缓存 Query？

这个问题面试经常问，答好了很加分。

Query 代表的是"当前 token 想从上下文里查什么信息"。第 t 个 token 的 Query 是"当前位置此时此刻要查询什么"，它和当前 token 的语义、位置、生成状态强相关。第 t+1 个 token 生成时，它所处的位置不同、上下文多了 t+1 的内容，它想问的问题完全不同，所以必须用**新的 Query**。

而 Key 代表的是"我这个 token 能提供什么信息"，Value 代表的是"我这个 token 实际含有什么内容"。历史 token 一旦生成，它"能提供什么"和"含有什么"就固定了——第 10 个 token 的 Key 不会因为后面生成了 100 个 token 而变化。所以 K/V 可以缓存，Q 必须现算。**K/V 是"我是谁"，Q 是"我要找谁"——"我是谁"不变，"我要找谁"每步都变。**

---

### prefill 和 decode 的分工

![KV 缓存运行机制](https://raw.githubusercontent.com/Link-990/Agent-learning/main/images/02-llm-framework/05-kv-cache/02-运行机制.png)

#### prefill 阶段：一次性建缓存

模型把系统提示词、历史对话、用户问题、RAG 片段等**所有输入 token** 一次性送入，并行计算每一层的 K/V，存入 KV Cache，输出最后位置的 logits。

这个阶段的特点是：
- **计算密集但高度并行**：所有 prompt token 的注意力矩阵可以一块算。
- **内存分配大**：一次性为所有 prompt token 分配 N_layers × seq_len × 2 × d_kv 的缓存空间。
- **首 token 延迟的决定因素**：prompt 越长，prefill 越慢，用户越等得久。

#### decode 阶段：复用缓存逐 token 生成

```text
# prefill
prompt_ids = tokenize(prompt)
logits, kv_cache = model.forward(prompt_ids, cache=None)
next_token = decode(logits[-1])

# decode
while not stop:
    logits, kv_cache = model.forward(next_token, cache=kv_cache)
    next_token = decode(logits[-1])
```

decode 的每一步只输入**一个**新 token，计算它每一层的 Q/K/V，把新 K/V 追加到缓存，用新 Q 查询缓存里的历史 K/V，输出 logits。

关键认知：**prefill 决定了"多久开始说话"，decode 决定了"说话有多快"**。长 prompt 的首 token 慢是 prefill 的问题，后续生成慢是 decode 吞吐的问题，两者排查路径完全不同。

---

### 显存代价：一张不可忽视的账单

KV Cache 的显存公式，面试时能写出来说明你真算过：

```text
KV_cache_bytes ≈ batch_size × seq_len × num_layers × 2 × num_kv_heads × head_dim × bytes_per_element
```

拆开来看每个因子：
- **batch_size**：并发请求数。每个请求都有自己的缓存，10 个并发就是 10 份。
- **seq_len**：上下文总长度 = prompt token + 已生成 token。多轮对话每轮都在涨。
- **num_layers**：模型层数。Llama-7B 有 32 层，70B 有 80 层，DeepSeek-V2 更多。
- **2**：K 和 V 各一份。
- **num_kv_heads × head_dim**：KV 的维度。MHA 里等于 q_head 数 × head_dim，GQA 里更小。
- **bytes_per_element**：FP16 是 2 字节，INT8 是 1 字节，FP32 是 4 字节。

**一个直观例子**：假设一个模型 32 层、KV 每层 128KB（中等大小），上下文 4K token，batch=1，KV Cache ≈ 32 × 4000 × 128KB ≈ 16MB——还行。但上下文扩展到 32K → 128MB；batch 到 16 并发 → 2GB；如果模型是 80 层 70B 级别 → 轻松 10GB+。**OOM 往往不是模型权重装不下，而是 KV Cache 在并发下悄悄吃光显存。**

---

### MHA、MQA、GQA：KV Cache 的三代进化

- **MHA（Multi-Head Attention）**：每个 query head 都有独立的 K/V head。KV Cache 大小 = num_heads × head_dim × 2。最"贵"。
- **MQA（Multi-Query Attention）**：所有 query head 共享一组 K/V。KV Cache 缩小为原来的 1/num_heads。但注意力表达能力受限。
- **GQA（Grouped-Query Attention）**：折中方案。把 query head 分若干组，每组共享一组 K/V。比如 32 个 query head 分 8 组，每组 4 个 query head 共享一组 K/V，KV Cache 是 MHA 的 1/4。

**GQA 是现代大模型的主流选择**——Llama 2 70B、Llama 3、Mistral 都在用。它的核心 trade-off 是：用少量注意力质量换取大量显存和带宽节省。在长上下文推理中，这个 trade-off 几乎总是划算的。

---

### 工程例子：多轮对话为什么越聊越慢

用户看到的是"接着聊"，服务端看到的是一场灾难。

**第 1 轮**：用户发来"帮我审查这份合同"（2000 token），模型回复（500 token）。总上下文 2500 token，KV Cache 轻松。

**第 2 轮**：用户追问"第三条风险点详细说一下"（20 token）。服务端把第 1 轮的完整历史拼上——prompt + 第 1 轮回答 + 第 2 轮问题 = 2520 token，加上第 2 轮回答又是 500 到 3000 token。

**第 5 轮**：历史累计 6000-8000 token，每轮都带着完整历史重新 prefill，KV Cache 随 seq_len 线性膨胀。同时，decode 阶段的每一步都需要在 8000 长度的缓存上做注意力，显存和带宽双重吃紧。

**怎么缓解？**

- **滑动窗口**：只保留最近 N 个 token 的 KV Cache，老的丢掉。简单粗暴，但可能丢失关键信息。
- **对话摘要**：每隔几轮对历史做自动摘要，用摘要替代原文。**但要小心**：一旦替换了文本，对应的旧 KV Cache 必须重建，不能用旧的——因为 K/V 和 token 是一一对应的。
- **Prefix Cache**：系统提示词和公共知识前缀在整个对话中不变，只 prefill 一次，后续请求复用同一段缓存。
- **PagedAttention（vLLM）**：把 KV Cache 按"页"管理，不是连续一大块——解决显存碎片和浪费问题。和 MQA/GQA 的维度压缩是**两个正交的优化方向**。

---

### 边界和风险

**第一，KV Cache ≠ 训练时的激活缓存。** 训练需要保存所有中间激活用于反向传播梯度计算，那个量级远超推理时的 KV Cache。面试时别混为一谈。

**第二，KV Cache 不能减少 prefill 成本。** 不管你有多少缓存，长 prompt 的第一个前向传播仍然要把所有输入 token 读一遍。KV Cache 省的是 decode 的重复计算，不省 prefill。

**第三，KV Cache 不是跨用户共享的长期记忆。** 不同用户、不同上下文、不同模型版本之间的 KV Cache 绝对不能复用。你无法"把 A 用户的缓存拿来给 B 用户加速"——这会导致输出串上下文、张冠李戴。

**第四，位置编码必须连续。** 使用 RoPE 等位置编码时，decode 阶段的 position id 必须和历史长度严格对齐。截断、padding、拼接、batch 合并、缓存复用的任何一步位置错了，模型可能突然开始胡言乱语——而且你很难排查，因为输出看起来只是"质量变差"，不是直接报错。

**第五，Prefix Cache 要求前缀完全一致。** 模型版本、tokenizer、位置编码、系统提示词甚至 padding 策略有任何一个变了，缓存就不匹配。只要中间插入、删除或改写历史，对应的后续缓存必须失效重建。

---

## 🎯 面试总结

### 必答三点

1. **KV Cache 缓存的是一份"历史说明书"**：每一层注意力中历史 token 的 Key 和 Value。新 token 生成时，用自己的 Q 去查历史 K，加权历史 V，避免重算。不缓存 Q，因为每个新位置的查询需求不同。

2. **prefill 建缓存，decode 用缓存，两个阶段两个瓶颈**：prefill 一次性处理 prompt 建立全量 KV Cache，决定首 token 延迟；decode 逐 token 追加复用缓存，决定生成吞吐量。长 prompt 首 token 慢是 prefill 的锅，后续生成慢是 decode 的锅。

3. **显存成本随六个因子线性增长**：batch_size × seq_len × num_layers × KV_head_dim × 2 × 精度。长上下文、高并发、多轮对话是 KV Cache 爆炸的三驾马车。MQA/GQA 压缩维度，PagedAttention 管理碎片，Prefix Cache 复用固定前缀——三个手段各解决不同层面的问题。

### 加分项

- 能写出 KV Cache 的显存公式并逐项分析
- 能解释 MHA/MQA/GQA 的演进逻辑和 trade-off
- 能区分"对话摘要后 KV Cache 必须重建"这个易错点
- 能说明位置编码错误如何导致"静默故障"（模型不报错但输出质量下降）

### 一句话

**KV Cache 是大模型推理中的"以空间换时间"——把算过的 Key/Value 存起来省去重复计算，但代价是显存随上下文长度和并发数线性膨胀。管好 KV Cache，就是管好推理服务的命门。**

## 🔍 排查实践

**首 token 延迟高**：先查 prompt token 数 → prefill 时间 → prefix cache 命中率（系统提示词是否每次重新 prefill？）→ 排队时间 → 冷启动。

**后续生成慢**：查 decode tokens/s → batch 调度策略（是否长请求堵住短请求？）→ KV Cache 的读写效率 → 显存带宽 → attention kernel 实现。

**OOM（Out of Memory）**：不要只看模型权重占了多少显存。同时查：① 当前活跃请求数（batch size）；② 每个请求的最大上下文长度（seq_len）；③ 最大输出长度；④ KV 数据类型（FP16 vs INT8 vs FP8）；⑤ 是否用了 GQA/MQA；⑥ 是否有 PagedAttention 管理碎片。**90% 的推理 OOM 不是模型太大，是 KV Cache 没管好。**

**实践中可以做这些**：
- 限制单请求最大上下文和最大输出长度，设硬上限。
- 对多轮历史做摘要或检索式记忆（只保留关键轮次），而不是无脑全部拼接。
- 对系统提示词和公共知识前缀启用 prefix caching。
- 为长请求和短请求分池调度，避免相互阻塞。
- 监控每个请求的 prompt_tokens、completion_tokens、KV_cache_size 和 cache_hit_rate。
- 考虑 KV Cache 量化（INT8/FP8）、滑动窗口注意力或分层缓存策略。

面试回答到这一层，体现的是你理解推理服务的**成本结构**和**故障模式**，而不只是背了一个概念。
