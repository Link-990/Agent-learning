# 记忆管理：长期记忆、短期记忆、上下文工程、记忆更新和读写

## 🎭 面试官这样问

> 面试官：你们 Agent 的记忆是怎么设计的？
> 你：我们用了向量数据库存历史对话，每次用户提问时检索最相关的几条，拼到 prompt 里。
> 面试官：那如果一个 Agent 第一天记住了"用户不喜欢周一早会"，第二天用户开玩笑说了一句"以后都别给我安排会议"，你的系统会把这句话写进长期记忆吗？
> 你：这个……把所有对话都存进去的话，可能会。
> 面试官：那你就是用一个玩笑把用户的所有会议全取消了。我再问你——如果一个工具的某次调用返回了错误数据，这个错误被写进了长期记忆，以后每次任务都参考这个错误信息，你怎么办？
> 你：需要做记忆校验和过期机制……
> 面试官：对。记忆不是聊天记录的仓库，是经过筛选、验证、可追溯、可修正的知识系统。什么该记、什么时候记、怎么检索、怎么更新、怎么删除——这五个问题你答得上来，才能说你会做 Agent 记忆。

面试问记忆管理，面试官不是在考你向量数据库选型，而是在看你是否理解**记忆的质量比数量重要，错误记忆的长期污染比没有记忆更可怕**。

## 💡 简要回答（面试话术）

我理解 Agent 的记忆系统要回答五个核心问题：什么值得记、何时写入、如何检索、何时更新、何时删除或遗忘。它不能简单等同于"把所有聊天记录存进向量库"，而是要分三层来设计。

**短期记忆**服务当前任务。它保存的是"这一轮"的上下文：用户目标、当前计划进度、已调用的工具和返回结果、待确认的动作、中间推导的结论。短期记忆应该结构化——比如订单处理 Agent 不能只记"用户要退款"，要记 `{user_id, order_id, verify_status, policy_result, pending_action}`。任务结束后，短期记忆可以归档或丢弃，避免上下文膨胀。

**长期记忆**服务跨任务连续性。它保存经过验证的稳定信息：用户偏好（"回复语言用中文"）、项目约定（"提交前必须跑 lint"）、经验证的事实（"这个仓库使用 pnpm 而非 npm"）。长期记忆的写入门槛必须很高——不能写入临时情绪、玩笑话、错误工具结果和未经确认的推测。每条记忆要有来源标注、时间戳、置信度评分和过期策略。

**上下文工程**是记忆系统的调度层。每次模型调用前，它根据当前任务的需求，从短期和长期记忆中筛选最相关的信息注入 prompt。它要控制注入数量（token 预算有限）、做相关性排序、防止旧记忆污染当前推理。

总结来说，**短期记忆是白板，长期记忆是档案室，上下文工程是秘书**——秘书知道这次会议该把哪些文件摆在桌上，而不是把整个档案室搬过来。

## 📝 详细解析

### 核心矛盾：连续性有价值，错误记忆会长期污染

咱们先搞清楚记忆为什么重要、为什么危险。没有记忆的 Agent 每轮对话都像第一次见你——代码助手不知道你刚才打开了哪个文件，客服助手不知道你已经在上一轮完成了身份校验。这种"失忆"会让用户反复提供相同信息，体验极差。**记忆的价值在于让 Agent 的行为具有连续性**。

但记忆太随意写入会带来更隐蔽的伤害——**错误记忆的污染是持续放大的**。举个例子：Agent 在某次任务中调用了天气 API，因为网络抖动返回了空数据。如果系统把"天气 API 不可用"写进长期记忆，以后每次用户问天气，Agent 都会绕过 API、靠训练语料编答案。一个瞬间的网络问题，变成了永久的系统缺陷。更危险的是用户情绪化表达——用户某天烦躁说了一句"你的回答总是太啰嗦"，系统写成"用户需要极简回答"——之后所有回答都是两三个字，用户真正需要详细解释时反而得不到。

这就是记忆系统的根本矛盾：**你要用过去的信息让现在更聪明，但过去的信息里混杂了噪声、错误和情绪，不加筛选地复用只会让系统越来越蠢**。所以记忆管理的核心不是"怎么存更多"，而是"怎么存得更准、用得更精"。

### 底层机制：三层记忆架构拆开看

我把三层架构用日常办公来类比，帮你建立直觉。

**短期记忆——你桌上的便利贴**。你今天要处理三个任务，便利贴上写着：任务 A 做到第三步了，已经拉了日志、查了索引，下一步要对比历史数据；任务 B 还在等用户确认；任务 C 工具调用超时了，要重试。便利贴的特点是：只记今天的事，事情做完就扔，写错了随时划掉重写。短期记忆在工程上可以用消息列表、scratchpad 对象、或任务 trace 来实现——关键是**结构化**，不能是一大段自然语言。结构化之后，Agent 的每个决策步骤都能精确引用短期记忆里的字段，而不是在长文本里模糊匹配。

**长期记忆——公司的知识库**。里面存的是经过验证、长期有效的知识：客户档案、产品手册、技术规范、项目约定。知识库有三个特点：入库有审核（不是谁都能往里写）、内容有版本号（改了之后旧版本可追溯）、过时了要归档（不是永久保留）。对应到 Agent 的长期记忆：写入前要做抽取和验证——"用户喜欢在周一早上查周报"是一条好记忆，因为出现了三次且每次都一致；"用户今天心情不好"不该写入，因为这是瞬时状态。记忆还要标注来源——是用户明确说的、是从行为推断的、还是从工具返回提取的——不同来源可信度不同。还要有过期策略——"用户当前项目使用 React 18"明年可能就过时了。

**上下文工程——开会前秘书准备的资料包**。秘书知道几个原则：桌面放不下太多文件（token 预算有限），只放和今天议题相关的（相关性排序），越近的文件越重要（时间衰减），手写的便签和盖章的文件可信度不一样（来源权重）。上下文工程做的是同一件事：从短期记忆拿当前任务状态，从长期记忆检索相关偏好和事实，按优先级排列，控制总长度，放进 prompt——但它不改变记忆本身，只是选择性地暴露。面试时能区分"记忆存储"和"上下文选择"是两层，就说明你理解得够深。

### 工程例子：代码助手如何使用记忆

代码助手是讲记忆管理最好的场景——因为它天然需要短期和长期记忆协作。

**短期记忆**记录当前任务的状态。用户说"修复登录页的表单验证 bug"，短期记忆要记录：当前打开的文件是 `LoginForm.tsx`，用户刚补充了限制"不要改样式，只修验证逻辑"，已经读取了 `validation.ts` 和 `api.ts`，工具 `search_code` 返回了三个相关函数，已修改的代码片段和测试结果。这些信息只在当前任务里有效——任务结束或用户说"开始新任务"后，短期记忆清空。

**长期记忆**记录跨任务的稳定规则。用户的偏好——"使用 TypeScript 严格模式""提交前先跑 lint"；项目约定——"使用 pnpm 管理依赖""组件放在 components/ 目录""API 错误统一用 ErrorBoundary 处理"。当用户说"按我们以前的写法改这个接口"，Agent 不是凭"感觉"写代码，而是从长期记忆中检索"以前的写法"——上次类似接口的签名、错误处理方式、测试覆盖模式——然后把这些约束注入当前上下文。

**关键的设计细节**：长期记忆和当前代码冲突时，以当前代码和用户显式指令为准。记忆里写着"项目用 Jest"，但仓库里已经有了 `vitest.config.ts`，Agent 必须相信当前代码，同时可以选择性地提醒用户"我注意到记忆里存的是 Jest，但当前项目已迁移到 Vitest，是否需要更新记忆？"这个追问动作就是记忆更新策略的体现。

### 边界和风险：记忆不是越多越好

记忆管理最容易踩的坑，我一个一个说。

**写入风险：把推测当事实**。用户说"这次回答还行"，不能写成"用户永远满意"。工具某次返回空数据，不能写成"接口不可用"。写入前必须经过抽取（从对话中提取有效信息片段）、去重（和已有记忆比，是不是重复信息）、来源记录（明确写出是从哪里来的）、置信度判断（多次一致的记录置信度高，单次偶发的置信度低）。面试金句：**宁可少记一条，也不错记十条**。

**安全风险：记忆可能是敏感数据的后门**。聊天记录里可能包含手机号、身份证号、API 密钥、客户数据和内部商业策略。多租户系统必须做严格隔离——用户 A 的记忆不能被用户 B 检索到。用户需要有查看、修改和删除记忆的能力（这是合规要求，不只是产品功能）。记忆内容在落盘前要做敏感信息脱敏。

**上下文污染**：检索出来的旧记忆如果和当前任务无关，却被塞进了 prompt，会误导模型。比如用户当前在问性能优化，但检索系统根据语义相似度召回了一条"用户上次问过 UI 颜色偏好"的记忆——模型可能在回答性能问题时莫名其妙地提一嘴颜色。工程上要做相关性重排序、限制注入条数（比如最多 5 条）、给记忆打标签（话题、时间、来源）来辅助筛选。

**记忆膨胀**：长期记忆越积越多，检索越来越慢，噪声比例越来越高。必须有遗忘机制——超过 N 个月未访问的记忆自动归档、置信度低于阈值的记忆标记为待审核、被新事实覆盖的旧记忆标记为过时。

## 🎯 面试总结

- **必答3点**：① 记忆系统分三层——短期记忆（当前任务状态）、长期记忆（跨任务稳定知识）、上下文工程（按需选择和注入）；② 长期记忆的写入必须经过抽取、去重、来源标注、置信度评分和敏感信息过滤，不能把对话记录原封不动存进去；③ 记忆和 RAG 的区别——RAG 检索外部知识库，记忆管理的是 Agent 自身积累的经验和用户状态，二者可互补但不能混淆。

- **加分项**：① 能讲出"错误记忆的长期污染"——一个单次错误写入如何影响后续所有任务，以及如何通过置信度衰减和过期策略解决；② 能讲出记忆更新的冲突处理——当当前事实和旧记忆矛盾时以谁为准，并给出自动提醒用户更新的机制；③ 能把"记忆存储"和"上下文选择"分成两层讲清楚，展示对 token 预算和相关性排序的理解。

- **一句话总结**：**记忆的价值是让 Agent 有连续性，但记忆的风险是让错误的过去绑架正确的现在——所以记忆管理的精髓不是存储，是筛选和遗忘**。

## 🔍 排查和实践建议

排查记忆问题时看三段日志：**写入时**——这条记忆的原始证据是什么，经过了什么抽取和验证逻辑，置信度是多少；**检索时**——根据什么 query 召回了哪些记忆，每条的相关性得分和时效性权重是多少；**注入前**——最终塞进 prompt 的是哪几条，上下文总长度有没有超预算。如果 Agent 的回答被旧信息带偏，八成是检索的相似度阈值设得太松、时间衰减权重不够、或注入上限太高导致无关信息挤占了关键上下文。

设计记忆系统时坚持四个原则：**少写**（不是所有对话都值得记，宁缺毋滥）、**可解释**（每条记忆能追溯原始来源）、**可删除**（用户和系统都能主动清理）、**可回放**（出问题时能根据日志还原记忆如何影响决策）。记忆不是越多越好——少而精的记忆让 Agent 更聪明，多而杂的记忆让 Agent 更糊涂。
