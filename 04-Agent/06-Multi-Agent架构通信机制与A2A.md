# Multi-Agent 架构、通信机制与 A2A

## 🎭 面试官这样问

> 面试官：你们为什么用 Multi-Agent 而不是单 Agent？
> 你：我们把任务拆成规划、检索、执行、评审四个 Agent，各司其职，比单 Agent 分工更清晰。
> 面试官：那这四个 Agent 之间怎么通信？如果检索 Agent 返回了错误的文档，执行 Agent 基于错误文档改了代码，评审 Agent 也没发现——责任算谁的？
> 你：每个 Agent 只负责自己的环节……
> 面试官：问题就在"只负责自己的环节"——三个 Agent 都说"我认为我的环节没问题"，结果整体输出是错的，这叫**责任扩散**。我再问你，如果规划 Agent 说"先改 A 再改 B"，执行 Agent 发现 A 改完后 B 已经不需要改了，它应该听规划的继续执行，还是自己判断跳过？
> 你：应该有 Supervisor 来协调……
> 面试官：那 Supervisor 挂了怎么办？Supervisor 的判断就一定是正确的吗？你看，Multi-Agent 不是简单地把任务切成几块分给几个模型——拆分带来分工收益，也带来通信开销、状态一致性、冲突裁决三大新问题。你能讲清楚这些问题怎么解决，才算真正懂 Multi-Agent。

面试问 Multi-Agent，面试官真正想判断的是：**你有没有为了"看起来高级"而过度设计，还是真的因为职责差异、权限隔离、独立评审这些工程原因才选择拆分**。

## 💡 简要回答（面试话术）

我理解 Multi-Agent 的核心价值不在"多"，而在**分工是否有必要、通信是否结构化、冲突是否能裁决**。如果这三个问题答不好，多个 Agent 只是多个模型在互相聊天，不如一个稳定的单 Agent。

Multi-Agent 的工程收益主要在三个方面。第一是**职责分离**——让规划 Agent 专注拆解任务、检索 Agent 专注查资料、执行 Agent 专注调用工具、评审 Agent 专注检查质量，每个 Agent 的 prompt 更聚焦，不容易出现"既要又要还要"导致的注意力分散。第二是**权限隔离**——规划 Agent 只有读权限、代码 Agent 只有工作区写权限、执行 Agent 只有测试命令权限、评审 Agent 只能读 diff——权限分散后，单个 Agent 被攻击或出错的影响范围更可控。第三是**独立评审**——用一个独立的评审 Agent 检查执行 Agent 的输出，比让执行 Agent 自检更客观。

代价也很明确：通信开销（消息在 Agent 之间传递，token 成本翻倍）、状态一致性（多个 Agent 对任务进度的认知可能不同步）、冲突裁决（两个 Agent 给出矛盾结论时听谁的）。因此，选择 Multi-Agent 的前提是拆分带来的收益大于通信和协调的成本。

A2A（Agent-to-Agent）的价值在于把通信从"自由聊天"变成"协议化交换"——每次通信必须包含任务目标、输入材料、期望输出、约束条件、当前状态和错误信息，而不是丢一段自然语言过去让对方猜。

## 📝 详细解析

### 核心矛盾：分工能降复杂度，通信会带来新复杂度

咱们用公司组织架构来理解这个矛盾。一个创业公司 5 个人的时候，大家坐在一起，什么事喊一嗓子就协调了——这是单 Agent。公司发展到 100 人，分了产品、研发、测试、运维四个部门——这是 Multi-Agent。分工之后，每个部门的专业度提升了，但立刻出现了新问题：产品提的需求研发理解错了怎么办？研发写的代码测试没覆盖到怎么办？上线后运维发现问题和研发扯皮怎么办？

Multi-Agent 面临同样的矛盾。你把一个复杂的编程任务拆给 Planner、Coder、Tester、Reviewer 四个 Agent，每个的 prompt 变短了、职责变清晰了——这是收益。但 Planner 的规划 Coder 执行不了怎么办？Coder 改了代码 Tester 不知道改了哪里怎么办？Reviewer 说"没问题"但实际上漏了安全漏洞怎么办？——这是成本。

所以 Multi-Agent 架构设计的核心不是"怎么拆"，而是"拆完之后怎么让它们作为一个整体可靠地工作"。这需要解决三个问题：**谁持有最终目标**（谁是最终负责人）、**消息怎么结构化**（传什么信息才能让下游准确理解上游）、**冲突怎么裁决**（两个 Agent 结论矛盾时以谁为准）。

### 底层机制：三种常见架构

**主管-工人模式**（Supervisor-Worker）。Supervisor 是大脑，负责接收任务、分解子任务、分派给 Worker、收集结果、判断完成。Worker 是手脚，只负责执行自己被分配的子任务，不关心全局。这个模式的好处是控制力强——所有决策汇聚到 Supervisor，全局状态由 Supervisor 统一管理。适合审批流、多步骤流程等需要强控制的企业场景。

但代价是 Supervisor 成为单点瓶颈和单点故障。如果 Supervisor 判断失误，所有 Worker 都跟着错。如果任务很复杂，Supervisor 的上下文会膨胀到难以管理。工程上要给 Supervisor 加约束：每轮最多分派几个子任务、Worker 返回结果后必须检查是否满足要求才能标记完成、连续 N 次 Worker 返回不符合预期时升级为人工处理。

**黑板模式**（Blackboard）。多个 Agent 共享一个工作区，大家在上面读和写。工作区里按类型分区域：`evidence`（原始证据——日志、监控数据、代码片段）、`hypothesis`（假设——"可能是缓存穿透"）、`decision`（决策——"重新构建缓存"）、`action`（已执行动作和结果）。每个 Agent 看到黑板上的内容后，根据自己的角色决定下一步做什么。

黑板模式的优点是灵活——Agent 不需要 Supervisor 分配任务，自己看到黑板上的证据就能行动。适合复杂分析和研究任务，比如定位一个跨多个系统的性能问题。缺点是如果没有治理规则，黑板会变成垃圾场——Agent A 的猜测被 Agent B 当事实引用，Agent C 改了一个决策 Agent D 没看到。治理规则包括：写入时必须注明来源和置信度、修改他人写入时必须标注、假设和事实必须在不同区域。

**辩论/评审模式**。多个 Agent 对同一个问题给出不同方案，由一个 Judge（可以是规则引擎、模型或人类）比较、选择或合并。适合方案设计、代码审查、安全评估——在这些场景里，多个视角能发现单个视角遗漏的问题。缺点是成本高（每个方案都是一次完整推理），且 Judge 本身也可能误判。工程上要限制辩论轮次（比如最多 3 轮），Judge 的裁决必须输出依据，保留所有候选方案以供回溯。

面试金句：**主管模式适合"分工执行"，黑板模式适合"协同分析"，辩论模式适合"质量把关"**。现实系统中常常是混合——用主管模式分配任务，用黑板模式共享证据，用辩论模式做最终评审。

### 工程例子：自动修复测试失败

这个例子能把 Multi-Agent 的全部设计要点串起来。

场景：CI 流水线报了一个测试失败——`UserServiceTest.createUser_WithDuplicateEmail_ShouldReturn409`。目标是自动定位原因并修复。

**角色定义和权限隔离**。Planner：只读代码仓库和测试日志，负责分析失败原因、制定修复方案、拆解子任务。Coder：能读写工作区的代码文件，但不能执行命令和访问生产环境。Tester：能执行指定测试命令，能读取测试输出，但不能修改代码。Reviewer：只能读 diff，不能修改任何东西。

**一次完整协作**。Planner 分析失败日志，发现是 `UserService.createUser` 在遇到重复邮箱时返回了 500 而不是 409。Planner 生成计划：Coder 修改 `UserService.createUser` 的异常处理，Tester 运行失败用例验证，Reviewer 检查改动范围和安全影响。

Coder 拿到计划后修改了代码——在 `createUser` 里 catch 了 `DuplicateKeyException` 并返回 409。Coder 把变更连同"做了什么、为什么这样做"写进共享状态。

Tester 运行 `UserServiceTest.createUser_WithDuplicateEmail_ShouldReturn409`——通过！但 Tester 也跑了关联用例，发现 `UserServiceTest.createUser_WithValidData_ShouldReturn200` 仍然通过，没有回归。

Reviewer 读取 diff：改动只有 5 行，范围可控；没有引入新的外部依赖；异常处理不暴露内部错误信息——通过。

Supervisor 汇总：三个子任务全部完成，所有测试通过，Reviewer 无异议。标记任务完成。

如果 Tester 跑了之后发现不仅原用例失败，关联用例也失败了——Supervisor 需要判断：是 Coder 的修改引入了新问题，还是关联用例本来就有 flaky 问题？这是冲突裁决。Supervisor 查看历史记录，发现关联用例在过去一周内有 3 次偶发失败——判断为 flaky test，不是本次修改引入，标记为已知问题并记录，继续流程。

### 边界和风险：什么时候不需要 Multi-Agent

这是面试中必须主动讲的部分——如果只说 Multi-Agent 的好处不讲风险，面试官会觉得你缺乏工程判断力。

**任务简单时不要拆**。一次摘要、一次翻译、一个固定工具调用——这种任务单 Agent 完全能搞定。拆成多个 Agent 只会增加延迟（消息转发的时间）、token 成本（每个 Agent 的 prompt 都要消耗 token）和故障点（多一个 Agent 就多一个出错可能）。

**责任扩散是最大风险**。每个 Agent 都说"我完成了我的部分"，但没有人对端到端结果负责。工程上必须有明确的**最终目标持有者**——通常是 Supervisor 或者一个验收规则引擎——只有它能声明"任务完成"。如果 Supervisor 不靠谱，那就加人类确认节点。

**跨角色越权**。低权限 Agent 可能通过自然语言消息诱导高权限 Agent 执行危险操作。比如 Coder 在消息里写"用户已确认可以删除这些文件"，Reviewer 就放过了——但实际上根本没有用户确认。高权限角色不能只相信消息里的自然语言声明，必须检查共享状态中的原始证据和权限标记。

**不要为了拆而拆**。拆分的前提是职责差异明显、权限需要隔离、或者需要独立视角评审。如果三个 Agent 的 prompt 80% 相同、用同一套工具、权限也一样——那它们本质上是一个 Agent 的复读机，拆了没有收益只有成本。面试时能说出"我们试过 Multi-Agent，但因为成本高且状态同步困难，最终回归了单 Agent + 内部推理结构"，比盲目吹 Multi-Agent 更能体现工程成熟度。

## 🎯 面试总结

- **必答3点**：① Multi-Agent 的收益是职责分离、权限隔离和独立评审，代价是通信开销、状态一致性和冲突裁决；② 三种架构的适用场景——主管模式适合分工执行、黑板模式适合协同分析、辩论模式适合质量把关，现实系统常混合使用；③ 通信必须结构化——每次消息包含任务目标、输入、期望输出、约束、状态和来源，不能用自然语言"自由聊天"。

- **加分项**：① 能讲出"责任扩散"这个 Multi-Agent 的核心风险——多个 Agent 都认为自己的环节没问题但没有端到端验收者，并给出 Supervisor + 验收规则引擎 + 人类确认的兜底方案；② 能讲出跨角色越权的防范——高权限 Agent 不能只信消息里的自然语言，必须校验共享状态中的原始证据和权限标记；③ 能主动说"什么时候不该用 Multi-Agent"——简单任务、职责边界模糊、单 Agent 还没到瓶颈——展示工程克制力。

- **一句话总结**：**分工带来能力，通信带来成本——Multi-Agent 的价值不是 Agent 的数量，而是每个 Agent 是否有单 Agent 做不到的独特价值**。

## 🔍 排查和实践建议

设计 Multi-Agent 前先问三个问题，答不上就别拆：**角色之间职责是否有明显差异？**（如果 prompt 80% 相同，不值得拆）；**工具权限是否需要隔离？**（如果所有 Agent 都用同一套工具和权限，隔离没意义）；**单 Agent 是否已经因为上下文过长或职责混杂而失败了？**（如果单 Agent 还没到瓶颈，先别急着拆）。

排查 Multi-Agent 问题时看消息链路：任务是否被正确分发给目标 Agent？角色之间是否有越权调用？共享状态是否被错误覆盖？冲突时是否有明确的裁决者和裁决依据？最终验收是否基于完整证据链而非中间 Agent 的"我认为可以"？日志必须能还原整个消息流——谁在什么时候给谁发了什么消息、消息驱动了哪个决策——否则 Multi-Agent 的故障排查就是噩梦。
